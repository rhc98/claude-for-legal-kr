# /use-case-triage

1. `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` 읽기. 개인정보 실무
   구성 확인 — 없으면 멈추고 셋업으로 안내.
2. 아래 워크플로우 실행. 활동이 모호하면 명확화.
3. 하우스 트리거 점검 → PIPA §33 의무평가 점검 → 처리방침 충돌 점검.
4. 출력: 분류 (진행 가능 / PIA 권장 / 의무 PIA / 중단), 근거, 의무 시 조건 표, 플러그인 간 핸드오프.
5. 평가가 필요하면 같은 대화에서 PIA 생성 계속 제안.

```
/privacy-legal:use-case-triage "행동 데이터를 콘텐츠 추천 개인화에 사용하는 신규 기능"
```

# 개인정보 처리 활동 트리아지

## 매터 컨텍스트

실무 수준 CLAUDE.md의 `## 매터 워크스페이스` 확인. `활성화`가 `✗`(사내 사용자의 디폴트)이면
이 단락 나머지 skip — 스킬은 실무 수준 컨텍스트 사용, 매터 머시너리는 비활성. 활성이고
활성 매터 없으면: "어느 매터입니까? `/privacy-legal:matter-workspace switch <slug>` 실행하거나
`practice-level` 말씀." 활성 매터의 `matter.md` 로드. 산출물은
`~/.claude/plugins/config/claude-for-legal/privacy-legal/matters/<매터-slug>/`. `매터 간
컨텍스트`가 `on`이 아니면 다른 매터 파일 절대 안 읽음.

## 목적

누군가 PIA를 돌리기 전에 나오는 질문에 답: "이게 PIA가 필요한가?" 필요하면 어떤 종류,
무엇이 길을 막는가.

개인정보 트리아지는 PIA 생성보다 빠르지만 그 위쪽에 있다. 평가를 작성하지 않음 — 필요한지,
어떤 조건에서 필요한지 판정. PIA 생성 스킬이 깊은 작업을 함.

출력은 네 분류 중 하나:
- **진행 가능** — PIA 불필요. 표준 안전조치 적용.
- **PIA 권장** — 배포 전 또는 동시에 평가 필요. (PIPA 의무 트리거 미해당이지만 권장)
- **의무 PIA** — PIPA §33 + 시행령 §35의 의무평가 트리거 해당 (공공기관의 50만 명 이상
  정보주체 / 5만 명 이상 민감정보 / 100만 명 이상 정보주체 시스템 등). CPO·GC 관여 가능성.
- **중단** — 처리 활동이 개인정보처리방침과 충돌하거나 기재된 대로의 적법 처리 근거가 없음.
  진행 전 재설계 필요.

## 관할 가정

이 트리아지는 구성에 명시된 관할(디폴트: 한국 PIPA)을 가정. 처리 활동·처리자·정보주체가
다른 관할에 속하면 이 분류가 그대로 적용되지 않을 수 있음.

**외국 데이터 트리거:** 사용자가 처리 활동에 다음을 언급하면 외국법 overlay 확인 필요:
- EU 정보주체 → GDPR (Art. 35 DPIA 의무 트리거 별도 확인 필요)
- 미국 캘리포니아 거주자 → CCPA/CPRA risk assessment 트리거
- 중국 → PIPL 영향평가
- 일본 → APPI 등

외국 overlay가 적용되면 "이 트리아지는 한국 PIPA 프레임워크. [관할] 정보주체가 포함되어
있어 [관할법] 추가 점검 필요" 명시.

## config 먼저 읽기

트리아지 전 항상 `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` 읽기.
거기의 PIA 트리거 기준·규제 풋프린트·처리방침 약속이 authoritative. 일반 개인정보법
reasoning은 이 회사가 실제로 commit한 것의 대체가 아니다.

파일이 없거나 `[PLACEHOLDER]`를 포함하면 surface:

> 실무 프로파일을 아직 구성하지 않으셨네요 — 그게 PIA 트리거 기준, 규제 풋프린트, 처리방침
> 약속을 당신 실무에 맞게 tailor하는 방법.
>
> **두 선택:**
> - `/privacy-legal:cold-start-interview`(10-15분) 실행해 프로파일 구성, 그 후 당신 실무에
>   맞춤 트리아지.
> - **"잠정"** 말씀하시면 일반 디폴트(한국 PIPA, 중간 위험 성향, 변호사 역할, 플레이북
>   없음) 대조 트리아지하고 모든 출력에 `[잠정 — 맞춤 출력을 위해 프로파일 구성]` 태그.

### 잠정 모드

사용자가 "잠정" 말씀하면 일반 디폴트로 트리아지: 중간 위험 성향, 변호사 역할, 한국 PIPA
관할, 플레이북 없음(구성된 commit과 매칭이 아닌 일반 PIPA 원칙으로 분류). 검토자 메모와
모든 발견 블록에 `[잠정]` 태그. 출력 끝에 추가:

> "기본 가정에 대한 일반 실행이었습니다. `/privacy-legal:cold-start-interview` 실행하면
> 당신 실무에 calibrate된 출력 — 규제 풋프린트, 처리방침 약속, 위험 성향. 10-15분."

## 트리아지 절차

### Step 1: 활동 이해

설명이 모호하면 분류 전 묻기. 구체화:

- 어떤 데이터를 수집·처리? 어떤 항목?
- 정보주체는 누구 — 고객, 직원, 제3자?
- 목적? 해결하는 문제?
- 신규 수집인가, 이미 가진 데이터의 재이용인가?
- 제3자 수탁자 관여? 신규 수탁자, 기존?
- 자동화된 의사결정? 출력이 누군가에게 영향?
- 배포 컨텍스트 — 내부만, 고객 대상, 공개?

"새 기능"과 "데이터 처리 활동"은 정확한 트리아지에 충분하지 않음.

---

### Step 2: 하우스 트리거 점검

`~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` → `## PIA(개인정보
영향평가) 하우스 스타일` → 트리거 기준 읽기. 적용.

하우스 트리거 충족 → 최소 **PIA 권장**.

하우스 트리거 미충족 시 Step 3 진행 후 진행 가능 결론. 일부 활동은 내부 정책과 무관하게
PIA가 필요.

---

### Step 3: PIPA 의무평가 점검

**PIPA §33 + 시행령 §35의 의무평가 트리거 — 법망에서 현행 텍스트 가져와 점검:**

```
mcp__beopmang__law get "개인정보보호법 제33조"
mcp__beopmang__law get "개인정보보호법 시행령 제35조"
```

(법망 미연결 시 `[PIPA §33 — 모델 지식 — 검증 필요]` 태그)

2026년 5월 기준 의무평가 대상(시행령 §35 — 현행 확인 필수):

1. **공공기관**이 다음 개인정보파일을 구축·운용·변경하려는 경우:
   - 5만 명 이상의 정보주체에 관한 **민감정보(§23)** 또는 **고유식별정보(§24)** 처리
   - 50만 명 이상의 정보주체에 관한 개인정보파일을 **다른 개인정보파일과 연계** 처리
   - 100만 명 이상의 정보주체에 관한 개인정보파일

2. 그 외 정보주체 권리 침해 우려가 큰 경우(법령으로 정함)

**민간기업은 직접 의무는 없지만**, 다음의 경우 권장 또는 사실상 필수:
- 신용정보법 적용 금융기관의 신용정보 처리 영향평가
- 의료법 적용 의료기관의 환자정보 시스템 신규 구축
- 정보통신서비스 제공자가 §39-3에 따른 분실·도난·유출 사고 발생 시 사후 평가

해당하면 → **의무 PIA** (또는 그에 준하는 의무 평가).

**강한 지표(의무는 아니지만 PIA 하는 게 맞음):**
- 신규 기술 또는 기존 기술의 novel 사용
- 만 14세 미만 아동 데이터 (정보통신망법 §31-2)
- 별개로 수집된 데이터셋의 결합
- 차별을 가능하게 할 수 있는 데이터
- 사용자가 예상하지 못할 처리
- 자동화된 의사결정 (PIPA §37-2, 2024.3.15. 시행 — 거부권 및 설명요구권)
- 위치정보 (위치정보법)
- 바이오메트릭 인식정보 (PIPA §23 민감정보)

하나 이상의 강한 지표가 있고 의무 트리거가 없으면 → **PIA 권장**으로 escalate (의무는
아니지만 출력에 플래그).

### Step 4: 처리방침 충돌 점검

`~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` → `## 개인정보처리방침
약속` 읽기. 제안된 활동을 모든 기재된 약속과 대조 점검.

**자주 잡히는 충돌:**
- 처리방침 수집 항목에 신규 수집 항목 누락 → 처리방침 갱신 필요 또는 수집 중단
- 처리방침이 "제3자에 제공하지 않음" 명시인데 활동이 벤더에 데이터 제공 → PIPA §17 제3자
  제공 동의 별도 필요 여부 점검
- 처리방침이 보유 기간을 명시했는데 활동이 더 오래 보유
- 처리방침이 "[목적]만 사용" 명시인데 활동이 새 목적으로 사용 → 별도 동의 또는 PIPA §15②
  목적 변경 통지 필요
- 처리방침이 정보주체 권리를 명시했는데 활동이 새 데이터 항목 생성 — 권리행사 절차가
  이 데이터 항목을 cover하지 않음
- **처리위탁 또는 국외이전이 있는데 처리방침에 게재되지 않음** → PIPA §26②·§28-8② 위반,
  처리방침 갱신 없이는 처리 자체가 위법

직접 충돌 → **중단**. "주의하며 진행" 아님 — 처리방침 충돌은 진행 전 해결(처리방침 갱신
또는 활동 재설계) 필요.

### Step 6: 플러그인 간 핸드오프

**AI 거버넌스 핸드오프:** 활동이 개인에 대한 결정을 하거나 영향을 주는 AI 시스템을 포함하면:

> "이 활동은 AI 의사결정을 포함합니다. PIA에 더해 AI 영향평가가 필요할 가능성. AI 기본법

## 엣지 케이스와 실패 모드

**"익명화됐다"는 자동 진행 가능 아님.**
어떻게 익명화됐는지, 재식별이 realistically 가능한지 물어보기. **PIPA는 익명정보(재식별
불가)와 가명정보(추가정보 결합 시 식별 가능, PIPA §28-2 이하)를 구별** — 가명정보는
개인정보 처리에 해당하며, 결합·제3자 제공 등에 별도 규정 적용. 사용자가 "익명화"라
말해도 가명처리인지 진짜 익명화인지 확인.

**"이미 비슷한 거 하고 있다"는 트리아지 아님.**
평가받지 않은 기존 처리가 신규 처리를 grandfather하지 않음. 신규 활동이 규모·목적·데이터
항목에서 material하게 다르면 새로 트리아지.

**"파일럿일 뿐"은 트리아지 skip 아님.**
실제 사용자·직원 데이터를 건드리는 파일럿은 같은 트리거 적용. 같은 분류 적용. PIA가
필요하면 파일럿이 PIA를 가져야 함.

**"벤더가 모든 프라이버시 처리한다."**
벤더는 인프라 처리. 당신은 여전히 목적을 정하는 위탁자(처리자). 개인정보가 벤더로 흐르면
위·수탁계약 필요(PIPA §26), 트리아지는 여전히 목적에 적용.

**추론된 데이터·파생 속성도 카운트.**
활동이 개인에 대한 추론 데이터 생성(행동 점수, 예측 선호도)이면, 추론된 속성을 트리아지
목적상 개인정보로 취급. "그냥 점수 계산"이 그 점수가 무엇을 나타내는지 모호하게 만들지
않게.

**"카카오·네이버 로그인 SSO니까 우리는 개인정보 처리 안 함" — 틀림.**
SSO 토큰을 받아 사용자 식별·세션 유지하면 개인정보 처리. 토큰 + 추가 처리 데이터는
개인정보. SSO 제공자가 받은 항목 외 우리가 직접 받는 항목(이메일·이름) 별도 동의 필요.
