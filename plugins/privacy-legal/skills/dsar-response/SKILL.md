   (시스템 목록·본인확인 방법·SLA) 로드.
2. 아래 워크플로우 실행.
3. 요구 유형 분류. 에스컬레이션 트리거 점검 — 발사하면 진행 전 routing.

# 정보주체 권리행사 응답 초안

## 매터 컨텍스트

CLAUDE.md `## 매터 워크스페이스` 확인. 비활성이면 skip. 활성이면 묻기.

---

## 목적

권리행사는 기한(적용 regime이 정함), 절차(본인확인·locate·면제 평가·응답), 그리고 잘못될
수 있는 여러 지점이 있다. 이 스킬은 각 단계를 walk through하고 응답을 작성.

**한국 PIPA에서 정보주체 권리:**
- **열람** (§35) — 응답 10일, 연장 10일
- **정정·삭제** (§36) — 응답 10일
- **처리정지** (§37) — 응답 10일
- **동의철회** (§22) — 즉시
- **자동화된 결정 거부·설명요구** (§37-2, **2024.3.15. 시행**) — 응답 30일
- **개인정보 이동** (PIPA 본법 일반 의무 아님; 신용정보법 §33-2 신용정보 이동권,
  공공 마이데이터 등 부분 적용)

응답 기한은 GDPR(1+2개월), CCPA(45+45일)과 다름.

## 관할 가정

이 분석은 한국 PIPA 가정. 정보주체가 다른 관할이면 외국법(GDPR 1개월, CCPA 45일 등)
체크 — 외국 거주자가 한국 회사에 요구한 경우 PIPA 적용 + 외국법 추가 적용 가능성.

## 절차 로드

`~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` → `## 정보주체 권리행사
처리 절차` 읽기. 거기에:
- 시스템 목록 (사용자 데이터가 있는 모든 곳)
- 본인확인 방법
- 응답 SLA
- 일상 처리 vs 에스컬레이션

시스템 목록이 비어있거나 stale이면 플래그 — 어디 봐야 할지 모르면 완전한 권리행사 불가.

## 워크플로우

### Step 1: 요구 분류

정보주체가 어느 권리를 invoke하는지 식별:

- **열람** (§35) — 자기 데이터 사본 + 처리 정보
- **정정·삭제** (§36) — 부정확 데이터 정정 또는 삭제
- **처리정지** (§37) — 처리 일시정지
- **동의철회** (§22) — 받은 동의 철회 (즉시 효력)
- **자동화 결정 거부·설명요구** (§37-2, 2024.3.15. 시행) — AI 등 자동화 결정에 대해
- **이용·제공 내역 통지 요구** (§35-3) — 누구에게 제공·이용 시키는지 통지

**진행 전 현행 규정 조사** — `mcp__beopmang__law get` 호출로 각 조문 현행 텍스트 확인.
조항 범위·예외 카테고리·기한 명확화. 응답 기한과 면제 사유는 정보주체별·권리별로 다름 —
법망 검증 필수.

> **침묵 보완 금지.** 법망 결과 부족 시 보고 후 멈춤. 옵션 제시: 쿼리 확장 / 다른 도구 /
> 웹 검색(`[웹 검색 — 검증 필요]`) / 미검증 플래그 후 멈춤.
>
> **출처 attribution 계층.** 모든 인용 — 응답 기한·면제 사유·관할 — 출처 태그 (`[법망]`,
> `[PIPC 고시]`, `[settled — 마지막 확인 YYYY-MM-DD]`, `[검증 필요]`, `[검증 필요-핀포인트]`).

일부 요구는 조합 — "내 계정 삭제하고 내 데이터 먼저 보내달라"는 열람 + 삭제. 두 연결된
요구로 처리.

### Step 2: 본인확인

CLAUDE.md의 방법대로. 흔한 접근:

- **로그인 본인확인:** 인증 세션 내에서 요구 → 본인확인 완료
- **이메일 매칭:** 등록 이메일에서 요구 → 통상 저위험 요구에 충분
- **추가 본인확인:** 고가치 계정·삭제 요구에 → challenge 질문·휴대폰 본인확인(SMS)·공동인증서
- **한국 특유 옵션:** 휴대폰 본인확인(SMS 인증)·아이핀·공동인증서·간편인증(카카오·네이버·페이스북 인증)

**위험에 calibrate.** 과도 본인확인은 권리행사 절차를 장벽으로 만듦(PIPC 시정조치 사례).
미진한 본인확인은 사기범에게 다른 사람 데이터 전달 위험. **PIPC는 처리 중인 정보로 확인
가능하면 그걸 우선시하라고 명시** — 별도 본인확인 서비스(휴대폰 본인확인 등) 추가 요구는
정당화 필요.

본인확인 불가 시:

```markdown
이 요청이 해당 정보주체로부터 온 것임을 확인할 수 없었습니다. 진행을 위해 [본인확인 절차]를
부탁드립니다. 확인할 수 없는 요구에 응하여 개인정보를 제공할 수 없습니다.
```

이건 (어쩌면) clock pause — 하지만 안고 있지 말 것. 본인확인 필요를 며칠 내 회신, 10일째에
하지 않음.

### Step 3: 데이터 locate

CLAUDE.md의 시스템 목록 walk. 각 시스템에 대해:

| 시스템 | 조회? | 데이터 발견? | 내용 |
|---|---|---|---|
| 운영 DB | | | |
| 분석 (예: Amplitude, GA4) | | | |
| 고객지원 티켓 (예: Zendesk, 채널톡) | | | |
| CRM (예: Salesforce, HubSpot, 자체) | | | |
| 이메일 마케팅 (예: Stibee, 마케토) | | | |
| 로그 | | | |
| 백업 | | | (보통 삭제 면제 — 아래 참조) |
| 제3자 수탁자 | | | (삭제 시 수탁자에게도 통지 필요) |

B2B 수탁자: "정보주체"는 보통 *고객의* 최종 사용자. 이게 우리 권리행사가 아니라 고객의
권리행사인지 확인. 많은 수탁자 위·수탁계약은 "권리행사를 위탁자에 전달" 명시.

### Step 4: 면제 분석

모든 것이 제공·삭제되지 않음. **진행 전 현행 규정 조사.** 각 항목에 대해 적용 가능한 면제
사유 식별 (PIPA §35 단서·§36③·§37②):

**열람 거부 사유 (PIPA §35④):**
1. 법률에 따라 열람이 금지되거나 제한되는 경우
2. 다른 사람의 생명·신체를 해할 우려가 있거나 다른 사람의 재산과 그 밖의 이익을 부당하게 침해할 우려가 있는 경우
3. 공공기관의 일부 직무 (조세 부과 등 시행령 §41 참조)

**정정·삭제 거부 사유 (PIPA §36③):**
1. 다른 법령에서 그 개인정보가 수집 대상으로 명시되어 있는 경우

**처리정지 거부 사유 (PIPA §37②):**
1. 법률 특별 규정·법령상 의무
2. 다른 사람의 생명·신체·재산 보호
3. 공공기관 소관 업무 수행에 영향
4. 정보주체와의 계약 이행을 곤란하게 하는 경우 (계약 해지 의사 명확하지 않은 경우)

법망에서 각 조문 가져옴:
```
mcp__beopmang__law get "개인정보보호법 제35조"
mcp__beopmang__law get "개인정보보호법 제36조"
mcp__beopmang__law get "개인정보보호법 제37조"
```

**주관적 호출로 목록 좁히지 않음.** 스킬은 good-faith basis 있는 면제 사유를 제안하고
불확실한 것은 플래그. 변호사가 응답 송부 전 좁힘. 나중에 적용된다고 밝혀진 면제 사유를
drop한 것은 비용 — 자료 공개되면 면제 사실상 사라짐. 그럴듯한 면제 사유 over-주장은 검토에서
변호사가 correctable. 복구 가능한 오류 선호.

모든 제안 면제 사유에 명시 노트: **"제안 — 주장 전 변호사 검토 필요. PIPC는 일괄 면제
주장을 scrutinize하므로 변호사가 목록 좁힘; 스킬은 좁히지 않음."**

흔한 반복 질문:
- 기록에 *다른 사람*에 대한 데이터가 있어 응답 전 redact 필요? (제3자 정보 — PIPA §35①의
  "타인의 생명·신체·재산 침해" 가능성)
- 삭제 차단하는 구체적 법령 보유 의무 있나? 인용 (PIPA §36③1호 — 다른 법령에서 수집 대상).
  예: 전자상거래법 §6 (5년), 통신비밀보호법 §15-2 (1년), 법령상 보존의무.
- 이 개인 데이터에 적용되는 활성 보존 명령(소송·수사 협력)? 검찰·법원 자료 요구가 있으면
  처리정지 거부 사유 §37②2호 또는 §37②3호 적용 가능
- 백업 rotation·기술 가능성 accommodation이 documented 필요? (일반 면제 사유 아니지만 PIPC도
  이론적으로 인정 — 명문화)

**모든 주장 면제 사유 documented.** PIPC가 왜 안 삭제했냐 물으면 "법령상 보유 의무"는 인용 필요.

### Step 5: 응답 초안 — 두 letter

> **리서치 커넥터 사전 점검.** letter 또는 내부 면제 분석 emit 전, 법망 커넥터가 reachable
> 한지 점검. 검토자 메모에 collect — 검토자 메모는 **내부** 면제 분석·cover 메모에 있지,
> 정보주체에게 가는 외부 권리행사 letter에 없음. Step 1·4·기한 관리 조사에서 커넥터가
> 결과 반환 안 하면 내부 검토자 메모의 **소스:** 라인에 기록 — 예: `법망 미연결 — 인용은
> 훈련 지식; 주장 면제 사유·응답 기한·연장 메커니즘은 특히 fabrication-prone, 정보주체나
> 규제 당국에 면제 주장 전 검증 필요`. 인용별 `[모델 지식 — 검증 필요]` 태그는 인라인 유지.

대부분 regime은 본응답과 별도의 신속 접수확인을 기대(또는 요구). 둘 다 생성 — 10일째에
나가는 한 letter로 collapse하지 않음.

- **Step 5a — 접수확인 letter.** 수령 후 며칠 내 (target: 당일~3-5일, regime의 법정 window
  훨씬 내). 수령 확인, 처리자가 이해한 요구 내용 명시, 응답 clock과 target 날짜, 미해결
  본인확인 자료 요청. 본응답 disclosure는 포함 안 함. 신속 접수확인은 권리행사 절차가
  작동한다는 첫 PIPC-가시적 신호; 중복 요구·조기 민원 위험도 감소.
- **Step 5b — 본응답 letter.** 실제 disclosure, 삭제 확인, 또는 이동 export. 법정 기한 (또는
  내부 SLA가 더 빡빡하면) 내 송부. 본인확인 완료 + Step 3/4 데이터 locate + 면제 분석 후.

**정보주체에게 어느 letter라도 송부 전:** CLAUDE.md `## 누가 이 플러그인을 사용하나`
읽기. 역할이 변호사가 아니면:

> 권리행사 응답 송부는 법적 결과 있음 — 내용·주장 면제 사유·누락이 모두 PIPC 검토 가능,
> 잘못된 진술은 집행 노출. 변호사 검토 거치셨나요? Yes면 진행. No면 brief:
>
> [1쪽 요약: 정보주체, invoke된 권리, 적용 regime, 시스템 목록에서 locate된 것, 어떤 면제로
> 보류, 본인확인 상태, 응답 기한, letter 송부 전 변호사에게 물을 세 가지.]
>
> 자격 변호사: 대한변호사협회·한국개인정보보호법학회.

명시적 yes 없이 통과 안 함.

> **Note:** 권리행사 letter 둘 다 정보주체에게 가는 외부 향 산출물. work-product 헤더 (`##
> 산출물`) 양쪽 letter에 **포함 안 함**. 내부 노트·로그·면제 분석은 attorney work product —
> 별도 유지 + work-product 헤더 prepend.

> **letter 송부 전:** 변호사 검토용 초안, 송부할 응답이 아님. 송부는 처리자를 입장에 binding,
> 면제 사유 포기 가능, PIPC clock 시작 가능. 면허 변호사 검토·편집·승인 후 정보주체에게.
> 미검토 송부 금지.

#### Step 5a — 접수확인 letter 템플릿

```markdown
제목: 귀하의 개인정보 권리행사 요청을 접수하였습니다 — [회사명] — [날짜]

[성명]님께,

귀하께서 [수령일]에 보내주신 [열람 / 삭제 / 정정 / 처리정지 / 자동화 결정 거부 / 동의철회] 요구를
접수하였습니다.

**요청 내용 (저희가 이해한 바):** [한 문장으로 재진술 — 예: "귀하의 계정과 연관된 저희가
보유한 모든 개인정보의 사본 및 제3자 제공 카테고리에 관한 정보를 요청하셨고, 사본을 받은 후
계정 삭제를 요청하셨습니다."]

**다음 일정:**
- 본응답 target 날짜는 [날짜 — regime의 법정 기한보다 늦지 않게; 내부 SLA 더 빡빡하면 그것 사용]
  입니다. [본인확인 미해결 시: "[구체적 본인확인 절차]를 진행해야 진행할 수 있습니다 —
  아래 참조."]
- 요청 처리가 복잡하거나 같은 시기에 다른 요청이 다수 들어와 추가 시간이 필요하면 최초 기한
  전에 알려드리고 이유를 설명드리겠습니다. (PIPA §35③, §36②: 정당한 사유로 10일 범위 연장 가능)
- 이 요청에 수수료는 발생하지 않습니다. (단, 열람 사본 발급에 실비 청구 가능 — PIPA §35⑤)

[본인확인 미해결 시:]
**본인확인을 위해** [구체적 본인확인 절차 — 예: 등록 이메일에서 회신하면서 결제 수단 마지막
4자리]를 부탁드립니다. 이는 저희 기한 진행을 멈추지 않습니다; 병행하여 작업하겠습니다.

문의: [개인정보 담당자 / 이메일 / 전화].

[발신자]
[직책]
[회사명] 개인정보 보호책임자
```

**Clock 시작 규칙.** 응답 clock은 요구 *수령* 시점부터 시작, 본인확인 완료 시점이 아님 —
regime이 별도로 말하지 않는 한. 본인확인에서 침묵으로 clock toll하지 않음. PIPA에서
별도 명시는 없음 — 수령부터 10일이 보수적 기준.

### Step 6: 로그

권리행사는 감사 대상. 기록:
- 수령일
- 본인확인 완료일
- 응답일
- 산출·삭제 내용
- 주장 면제 사유 및 근거
- 처리자
- (해당 시) 분쟁조정·민원 결과

팀에 권리행사 추적 도구 있으면 거기 기록. 없으면 로그 파일.

## 에스컬레이션 트리거

CLAUDE.md `## 에스컬레이션` 표에 따라 다음에 에스컬레이션:

- 요청자가 (또는 가능성) 원고·반대측 변호인·기자
- 요구 범위가 비정상("저에 관한 모든 내부 통신 포함")
- 이 개인 데이터에 보존 명령(소송·수사 협력) — 삭제 요구 + 보존 명령 = 충돌, 변호사 결정
- 요청자가 이전 권리행사 응답에 이의 제기 — 분쟁조정 가능성
- PIPC 또는 KISA가 CC되거나 언급
- 요청자가 단체 소송 가능성 언급 (집단분쟁조정 PIPA §49)

## 기한 관리

**두 letter 규칙.** 모든 권리행사가 접수확인 letter(target: 수령 후 당일~3-5일)와 본응답
letter(법정 기한 내) 모두 산출. PIPA는 명시적으로 접수확인을 요구하지 않으나 PIPC 가이드라인과
모범 사례는 신속 접수확인 권장. 10일째에 보내는 단일 letter는 처리는 맞아도 절차 실패.

**현행 응답 기한 조사.** invoke된 권리와 적용 관할에 대한. 연장 메커니즘 존재 여부, 추가
시간 얼마, 정보주체에게 어떤 통지 필요한지. 1차 소스 인용·핀포인트. 시행일 메모 — 권리행사
응답 기한은 자주 개정.

기본 PIPA 기한:
- 열람·정정·삭제·처리정지: **10일** (연장 10일, PIPA §35③, §36②)
- 자동화 결정 거부·설명요구: **30일** (PIPA §37-2④, 2024.3.15. 시행)
- 동의철회: 즉시 (PIPA §22①)

CLAUDE.md `## 정보주체 권리행사 처리 절차`에 법정 기한보다 빡빡한 내부 SLA 기록되어 있으면
내부 SLA 사용 + 법정 backstop 메모.

연장 필요하면 첫 기한 *훨씬 전*에 "추가 시간 필요" 통지 송부. 당일 연장은 보기 안 좋음.

## 분쟁조정·민원 대응

권리행사 응답에 정보주체가 만족하지 않으면 다음 옵션:
1. 개인정보 분쟁조정위원회(KISA 운영) 분쟁조정 신청 — PIPA §43
2. PIPC 민원·진정 — PIPA §62
3. 법원 소송 (손해배상 §39)

분쟁조정·민원이 접수되면 별도 절차 — 에스컬레이션, 외부 변호사 자문, 사실관계 정리 필요.

## 이 스킬이 안 하는 것

- 시스템을 직접 쿼리하지 않음. 체크리스트 walk through; 사람(또는 연결 도구)이 실제 쿼리.
- 가까운 면제 사유에 결정하지 않음. 변호사에게 플래그.
- 응답 송부하지 않음. 초안·검토·사람 송부.
