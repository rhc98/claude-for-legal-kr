# PIA 작성

## 매터 컨텍스트

실무 수준 CLAUDE.md의 `## 매터 워크스페이스` 확인. `활성화`가 `✗`(사내 사용자 디폴트)이면
skip — 스킬은 실무 수준 컨텍스트 사용. 활성이고 활성 매터 없으면 묻기. 활성 매터의 `matter.md`
로드. 산출물은 매터 폴더로.

---

## 목적지 점검

산출물 생성 전 가는 곳 확인. 사용자가 목적지를 명시(채널·배포 리스트·상대방·"모두")하면
비밀유지 서클 내부인지 묻기. 공개 채널, 회사 전체 리스트, 상대방·반대측 변호인, 벤더,
의뢰인(work product)은 보호를 포기시킴. 외부로 보이면 플래그하고 (a) 법무만 보는 비밀유지
버전, (b) 더 넓은 채널용 sanitize 버전, (c) 둘 다 제안 — 침묵으로 비밀유지 헤더 적용 후
보호 안 되는 곳에 paste 돕지 않음. CLAUDE.md `## 공유 가드레일 → 목적지 점검` 정본 참조.

## 목적

PIA는 제품팀과의 대화를 캡처한 것. 묻는다: 무슨 데이터를 왜, 얼마나 오래, 누가 보는지,
무엇이 잘못될 수 있는지. 이 스킬은 그 대화를 구조화하고 이 팀의 포맷으로 출력 — 콜드스타트
중 표준 PIA에서 학습한 것.

## 관할 가정

이 평가는 구성된 관할 범위(디폴트: 한국 PIPA)를 가정. 외국 정보주체·해외 처리자가 포함되면
프레임워크가 그대로 적용되지 않을 수 있음 — 출력에 명시.

## 이전 컨텍스트 로드

신규 PIA 작성 전 같은 기능·처리 활동·상대방에 대한 이전 작업이 산출물 폴더에 있는지 확인.
`~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` → `## 산출물`에서 경로 읽기. scan:

- **이전 `use-case-triage` 결과** — 트리아지의 위험 등급·의무 조건·언급된 우려가 PIA 진입점
- **같은·중첩 활동의 이전 `pia-generation` 산출물** — 대체 PIA는 화해 필요(변경된 것, carry over된 것)
- **범위 내 벤더의 이전 `dpa-review` 산출물** — DPA 검토 발견이 수탁자·국외이전·보유 위험 분석에 영향

이전 산출물 발견 시 PIA에 인용:

> "이전 트리아지([날짜])가 이를 [위험 등급]으로 평가, [조건] 요구. 이 PIA는 그 발견에 기반 —
> [어떤 조건 충족, 어떤 잔여, 어떤 재범위화]."

이전 PIA 존재 시:
> "이 PIA는 [날짜] PIA를 대체 — 이유: [범위 변경·신규 데이터 항목·벤더 변경·규제 변경].
> 이월 결론: [X]. 수정 결론: [Y, 이유 Z]."

**업스트림 심각도를 바닥으로 carry** — CLAUDE.md `## 공유 가드레일`의 스킬 간 심각도 바닥
규칙. 트리아지가 고위험으로 평가한 활동이 PIA가 저위험으로 결론낼 수 없다 — 이유와 변경
사항 명시 없이는.

이전 산출물 없으면 명시: "산출물 폴더에 이 활동의 이전 트리아지·PIA 없음; 콜드스타트."

## 하우스 스타일 로드

`~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` → `## PIA(개인정보 영향평가)
하우스 스타일` 읽기. 거기에 있음:
- PIA를 트리거하는 것(규제 의무 트리거와 다를 수 있음 — 일부 팀은 모든 것에 PIA, 일부는
  고위험만)
- 표준 PIA에서 추출한 구조 템플릿
- 일반 깊이
- 사인오프 누구

표준 PIA 구조가 config CLAUDE.md에 있으면 **사용**. 요점은 이 PIA가 이 팀이 만드는 다른
PIA처럼 보이게, 일반 PIA처럼이 아니라.

## Step 0: PIA가 필요한가?

`~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` 트리거 기준 점검. 팀의
하우스 답.

또한 **현행 의무평가 트리거 조사** — PIPA §33 + 시행령 §35 (50만 명 이상 정보주체 / 5만 명
이상 민감정보 / 100만 명 이상 정보주체 시스템 / 다른 파일과 연계 등). 법망에서 현행 텍스트
가져옴:

```
mcp__beopmang__law get "개인정보보호법 제33조"
mcp__beopmang__law get "개인정보보호법 시행령 제35조"
```

법망 미연결 시 `[PIPA §33 의무 트리거 — 모델 지식 — 검증 필요]` 태그 강제.

> **침묵 보완 금지.** 법망 검색이 빈 결과면 보고 후 멈춤. 웹 검색이나 모델 지식으로
> 갭을 채우지 않음, 묻지 않고는. 말한다: "법망에서 [N]건 반환. [관할/주제]에 대한 coverage
> 얇음. 옵션: (1) 검색 쿼리 확장, (2) 다른 리서치 도구, (3) 웹 검색(결과는 `[웹 검색 — 검증 필요]`
> 태그, 의존 전 1차 소스 대조), (4) 미검증 플래그 후 멈춤. 어느 것?" 변호사가 결정.

법정 의무 외에도 다음은 **PIA를 하는 것이 의무 아니더라도 가치 있는 강한 지표**(각각이
개별적으로 의무 트리거하는지 별도 조사):

- 신규 기술 또는 기존 기술의 novel 사용
- 만 14세 미만 아동 데이터 (정보통신망법 §31-2, PIPA §22-2)
- 별개로 수집된 데이터셋의 결합 / 가명정보 결합 (PIPA §28-3 — 결합 전문기관 거쳐야 함)
- 차별을 가능하게 할 수 있는 데이터
- 사용자가 예상하지 못할 처리
- 자동화 의사결정 (PIPA §37-2)
- 위치정보 (위치정보법)
- 바이오메트릭 (PIPA §23)

법정 트리거가 없고 하우스 트리거도 미충족 → "이 활동에 PIA가 필요하지 않은 것으로 보임.
파일용 한 단락 메모, 누군가 묻는 경우 대비."

## 인테이크

본격 작성 전 제품팀에서 다음을 받음. 대화형 OK — 보낼 form이 아님.

### 무엇·왜

- 기능·제품·변경은?
- 사용자에게 해결하는 문제?
- 어떤 개인정보 항목 건드림? 구체적으로 — "사용자 데이터"는 답이 아님. 어떤 필드?
- 신규 수집인가, 이미 가진 데이터인가?
- 처리는 — 저장, 분석, 공유, 자동화 결정?

### 적법 처리 근거

PIPA §15(개인정보 수집·이용)·§17(제3자 제공)·§18(목적 외 이용·제공)·§22(동의 요건) —
각 목적별 적법 근거 식별. **법망에서 현행 텍스트 가져와 인용**:

- 동의 (정보주체) — PIPA §15①1호 / 별도 동의 §22
- 법률 특별 규정 — 어느 법률·조항
- 법령상 의무 이행
- 공공기관 소관 업무 수행 — 공공기관에 한정
- 정보주체 또는 제3자 정당한 이익 — 균형 검토 필요
- 계약 이행 — 정보주체와의 계약 또는 계약 체결 절차
- 명백히 정당한 이익 (개인정보처리자)

제3자 제공 또는 목적 외 이용은 별도 동의 또는 §17·§18의 다른 근거 필요. 외국법 풋프린트
포함 시 (GDPR Art. 6, CCPA "sale"·"share" 등) 별도 조사·인용.

### 누구·어디

- 회사 내부 누가 이 데이터 봄? 엔지니어? 지원팀? 분석가?
- 제3자? 수탁자(PIPA §26), 제공받는 자(§17), 분석·광고?
- 저장 위치 어디? 국내·국외? 신규 인프라·기존?
- 보유 기간? 삭제 일정 있나, 영구 보유?

### 무엇이 잘못될 수 있나

- 이 데이터 유출 시 본인에 대한 피해?
- 의도치 않게도 차별에 이용될 수 있나?
- 사용자가 놀랄 처리인가? ("이상함 테스트" — 법적 기준 아니지만 유용)
- 옵트아웃 있나? 있어야 하나?

## 위험 품질 기준

PIA의 위험은 **구체적이고 설계와 연결**되어야 함, 일반론 금지. 일반 위험은 문서를 padding
하고 독자를 skim하도록 훈련시킴.

| 나쁜 위험 | 왜 나쁨 | 더 나음 |
|---|---|---|
| "데이터 침해" | 모든 것에 적용; 아무 말도 안 함 | "위치 이력이 감사 로그 없이 관리자 패널에서 지원 담당자가 접근 가능 — 악의적 내부자가 사용자를 추적하고도 발견되지 않음" |
| "PIPA 위반" | 순환 — PIA가 컴플라이언스를 *평가*하는 것 | 구체적 조항과 갭 명시 |
| "사용자가 안 좋아할 수도" | 모호 | "마케팅을 옵트아웃한 사용자도 이 동의가 별도 받지 않아 광고 받음" |

목표: 15개 padded가 아닌 2-5개 진짜 위험.

## 처리방침 diff

모든 PIA는 `~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md`의 처리방침
약속과 대조 점검. 흔한 drift:

- 처리방침 "X, Y, Z 수집" — 신규 기능이 W 수집. 처리방침 갱신 필요, 또는 W 수집 중단.
- 처리방침 "데이터 판매·공유 안 함" — 신규 기능이 광고 파트너와 공유. 별도 동의 필요 (PIPA §17)
  또는 처리방침 수정.
- 처리방침 보유 기간 "계정 유지 중" — 신규 기능이 계정 삭제 후 보유.
- **처리위탁 게재 누락** — 새 수탁자가 처리방침의 처리위탁 표에 없음 (PIPA §26②·③ 위반)
- **국외이전 게재 누락** — 국외 수탁자·이전받는 자, 항목, 목적, 보유기간, 이전방법 게재
  필요 (PIPA §28-8②)

모든 mismatch 플래그. 출시 전 하나가 바뀌어야 함.

## 게이트: PIPC에 영향평가 제출

내부 PIA 작성은 리서치·문서화. **PIPC에 PIA 제출 — 또는 PIPC 조사에 자발적 공개 — 은
결과적 행위.**

PIPA §33⑤: 공공기관은 영향평가 결과를 PIPC에 제출 의무. 민간은 제출 의무 없으나 PIPC 자료
요구·조사 응대 시 공개.

**PIPC·KISA·관할 행정기관에 영향평가(또는 그에 준하는 평가)를 제출하기 전:**
config CLAUDE.md의 `## 누가 이 플러그인을 사용하나` 읽기. 역할이 변호사가 아니면:

> 규제 당국 제출은 법적 결과 있음 — 문서가 감독 기록의 일부가 되고 중요 누락·오류가
> 집행 노출이 됨. 변호사 검토 거치셨나요? Yes면 진행. No면 변호사에게 가져갈 brief:
>
> [1쪽 요약 생성: regime과 규제 당국, 제출 이유(의무 트리거 또는 자발), 식별된 위험,
> 완화 후 잔여 위험, 플래그된 불확실성, 제출 전 변호사에게 물을 세 가지.]
>
> 자격 있는 변호사를 찾아야 하면: 대한변호사협회(`https://www.koreanbar.or.kr`),
> 한국개인정보보호법학회 회원변호사 검색.

명시적 yes 없이 이 게이트 통과 안 함.
