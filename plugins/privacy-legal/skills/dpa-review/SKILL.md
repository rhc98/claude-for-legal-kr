# 위·수탁계약 검토

## 매터 컨텍스트

실무 수준 CLAUDE.md `## 매터 워크스페이스` 확인. 비활성이면 skip. 활성이면 묻기.

---

## 목적

DPA(위·수탁계약)는 두 flavor가 있고 검토는 거의 반대. 고객이 자기 DPA를 보내면(우리가 수탁자)
우리는 운영 유연성을 방어. 우리가 벤더에 DPA를 보내면(우리가 위탁자) 우리(그리고 고객) 데이터를
보호. 두 검토 모두 같은 `~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md`
플레이북에서 읽지만 반대 행에서.

**한국 PIPA의 위·수탁계약**: §26와 시행령 §28이 정함. 미국·EU의 controller/processor
구조와 비슷하나 차이 있음:
- 위탁자는 수탁자의 PIPA 위반에 대해 **처리자 책임** 부담 (§26⑦)
- 처리위탁 사실은 **개인정보처리방침에 게재 의무** (§26②·③)
- 재위탁은 **위탁자 사전동의 필수** (§26⑥)
- 시행령 §28이 위·수탁계약의 **7개 필수 기재사항** 정함

## 먼저: 어느 방향?

다른 어떤 것보다 먼저 확립:

- **우리가 수탁자** → 고객이 자기 DPA 보냄 → `~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` →
  "우리가 수탁자(고객이 위탁자)인 경우" 표 읽기
- **우리가 위탁자** → 우리가 벤더에 DPA 보냄(또는 그들 것 검토) → "우리가 위탁자" 표 읽기

모호하면 묻기. 잘못 잡으면 모든 권고가 뒤집힘.

## 관할 가정

이 검토는 구성된 관할(디폴트: 한국 PIPA). 위탁자·수탁자·정보주체가 다른 관할이면 적용 안
될 수 있음. **외국 데이터 트리거**:
- EU 정보주체 → GDPR Art. 28 DPA 요건도 점검 (SCC, 국외이전 SCC, supplementary measures)
- 미국 → BAA(HIPAA), Reg P(GLBA) 등 sectoral 점검 필요
- 외국법 적용은 외국 변호사 자문 필요 — 플래그.

## 이전 컨텍스트 로드

검토 전 같은 상대방·처리 활동에 대한 이전 작업이 있는지 산출물 폴더 확인:
- 이전 `use-case-triage` 결과 — 위험 등급·조건 honor 또는 명시적 이탈
- 이전 `pia-generation` 산출물 — 완화 조치가 DPA에 반영되어야 할 가능성
- 이전 `dpa-review` 산출물 — 받아들였던 것·플래그한 것·합의한 것의 기대

이전 산출물 발견 시 인용:
> "이전 트리아지([날짜]) — [위험 등급], [X] 조건. 이 검토는 그 발견과 일치." 또는
> "이전 트리아지([날짜]) — [위험]. 이 검토는 [이유 — 새 사실·다른 범위·계약 변경]으로 이탈."

업스트림 심각도를 바닥으로 carry (CLAUDE.md `## 공유 가드레일`).

## 플레이북 로드

`~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` → `## 위·수탁계약 플레이북`
읽기. `## 개인정보처리방침 약속`도 — DPA는 처리방침 약속과 모순할 수 없음.

## 분야법 overlay (조항별 walk 전 먼저 물음)

조항별 검토 전: **이 DPA로 흐르는 데이터에 분야법 적용 카테고리가 있나?** PIPA는 일반법이며
분야법이 종종 추가 floor를 공급. 일반 DPA 플레이북에 안 나타남.

> **활동 기반 분야법 overlay — 먼저 물음:**
>
> 이 처리가 다음에 닿나:
> - **금융 신용정보 또는 개인신용정보** (신용정보법 §17 신용정보업자 위탁, §40 개인정보 보호의무)?
>   금융 위·수탁계약은 신용정보법의 추가 요건 — 신용정보업감독규정·금감원 행정지도 등.
> - **건강정보·진료정보** (의료법 §21·22, 의료기관 → 클라우드 서비스 위탁 시 의료법 시행규칙
>   §16-2 적용)?
> - **위치정보** (위치정보법 §15 위치정보사업·위치기반서비스사업, §26 위치정보 수집·이용 동의)?
> - **통신 비밀**(통신비밀보호법, 정보통신망법 §44-7 등)?
> - **만 14세 미만 아동 데이터**(PIPA §22-2, 정보통신망법 §31)?
> - **공공기관 위탁** (개별 법령상 위탁 근거 필요한 경우)?
>
> Yes면: 분야법이 통상 controlling substantive 제한 — PIPA 면제가 아닌. 1차 소스 조사·인용.
> 분야법 갭은 PIPA 갭과 함께 deal-breaker 목록에 플래그.

분야법 미적용이면 명시: "분야법 카테고리 미식별; 분야 overlay N/A" — 검토자가 점검이 일어났음을
보게.

## 조항별 검토

### 핵심 조항 (모든 DPA 점검)

모든 DPA를 다음 조항들로 walk, clause by clause. *구체적* 숫자·실질 입장(통지 기간·침해
타임라인·수용/거부 floor)은 `~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` →
`## 위·수탁계약 플레이북`. 법적 floor는 1차법에서 — **각 적용 regime의 현행 규칙을 법망에서
조사·인용**.

법망 호출 예:
```
mcp__beopmang__law get "개인정보보호법 제26조"
mcp__beopmang__law get "개인정보보호법 시행령 제28조"
mcp__beopmang__law get "개인정보보호법 시행령 제30조"  # 안전조치
mcp__beopmang__law get "개인정보보호법 제34조"  # 침해통지
mcp__beopmang__tools verify {"citation": "..."}
```

> **침묵 보완 금지.** 법망에서 floor·요건 정보가 부족하면 보고 후 멈춤. 옵션 제시: 쿼리
> 확장 / 다른 도구 / 웹 검색(`[웹 검색 — 검증 필요]`) / 미검증 플래그 후 멈춤. 변호사가 결정.
>
> **출처 attribution 계층.** 모든 인용 — 법적 floor·SCC 버전·고시·판례 — 출처 태그:
>
> - `[법망]` / `[법제처]` — 도구·1차 소스에서 가져옴
> - `[settled — 마지막 확인 YYYY-MM-DD]` — 안정적인 법조 인용. **PIPA는 2023.9.15. 대개정
>   되었으므로 §28-8 등 신설·개정 조항의 settled 인용은 2024년 이후 확인 날짜 필수.**
> - `[검증 필요]` — 검증해야 할 실재 모델 지식 인용
> - `[검증 필요-핀포인트]` — 핀포인트 인용은 fabrication 위험 최고, 항상 1차 검증

### 핵심 조항 점검 표

| 조항 | 찾는 것 | 플레이북 필드 | 흔한 다툼 |
|---|---|---|---|
| **위탁 목적·범위** (시행령 §28 필수 1) | 명시되고 한정됨 | — | "관련 목적" 등 open-ended 확장 |
| **재위탁** (PIPA §26⑥, 시행령 §28 필수 5) | 위탁자 사전동의 메커니즘 | 재위탁 절차 | 일괄 사전동의 vs 거부권 vs 통지만 |
| **안전조치** (시행령 §30) | annex로 구체적 통제 또는 표준 (개인정보의 안전성 확보조치 기준 PIPC 고시) | 안전조치 | "적정한 기술적·관리적 조치"만 적어두고 annex 없음 = 공허한 약속 |
| **침해 통지** (PIPA §34, 시행령 §40) | 트리거·기한 정의 | 침해 통지 시한 | 72시간 (개인정보위 + 정보주체) 이내 — 1차 floor. 더 빡빡한 SLA 약속받음? |
| **감사·점검** (PIPA §26④) | 방법(보고서 vs 현장), 주기, 통지, 비용 분담 | 감사·점검 | 짧은 통지의 현장 감사 |
| **국외 이전** (PIPA §28-8) | 5트랙 중 하나 명시: 별도동의·법률조약·인증·적정성·안전조치 | 국외이전 | 트랙 명시 없으면 위법 (별도 동의로 디폴트 처리 시 처리방침 게재 의무 §28-8② 충족 여부) |
| **종료 시 파기·반환** (시행령 §28 필수 7) | 일정·certify·백업 carve-out | 종료 시 파기 | "상업적으로 합리적" = ??? |
| **손해배상 책임** | 본계약 cap 내·별도 / 예외 | 손해배상 책임 | 무한 책임 = 존재 위협 |

### 우리가 수탁자일 때: 방어 검토

고객 DPA는 운영 부담을 우리에게 push하려 함. 각 조항에 대해 고객의 ask를 플레이북과 비교.
ask가 플레이북 밖이면 표준 입장으로 push back, 수용 가능 폴백 준비.

| 조항 | 위험 | 조사 / 플레이북 lookup |
|---|---|---|
| 재위탁 거부권 | 매 고객별 승인 없이 인프라 변경 불가 | 플레이북 입장 적용. PIPA §26⑥은 위탁자 사전동의 필수 — 거부권 다른 방식으로 만족? |
| 짧은 통지의 현장 감사 | 규모에서 unworkable | 감사 권리 플레이북 입장 적용. SOC 2 또는 ISMS-P 보고서가 통상적 폴백. |
| 공격적 침해 통지 시한 | 알기도 전에 통지 요구 | PIPA §34는 72시간 (개인정보위 + 정보주체). 더 빡빡하게 commit하면 운영 가능한지 확인. |
| Hard 데이터 소재 (단일 국가/DC) | 아키텍처와 안 맞을 수 있음 | 데이터 소재 플레이북 입장. 실제 commit 가능한 것 확인. |
| 무한 책임 | 회사 존재 걸기 | 손해배상 플레이북 입장. |
| "위탁자가 binding 지시 발급 가능" | open-ended 운영 통제 | "본 계약에 documented 또는 서면 합의된 지시"로 정의. |
| 매우 짧은 파기 일정 | 백업·로그 보유로 불가능 | 종료 시 파기 입장. 백업 rotation carve-out 명문화. |
| **국외이전 동의 책임 위탁자에게** | 우리가 수탁자로서 별도 동의 받을 수 없음 — 위탁자가 받아야 | 위탁자가 별도 동의를 처리방침에 게재하고 받은 상태에서 위탁 가능 |

### 우리가 위탁자일 때: 보호 검토

벤더 DPA는 우리에게 아무것도 주지 않으려 함. 각 조항을 위탁자 측 플레이북과 비교.

| 조항 | 갭 | 조사 / 플레이북 lookup |
|---|---|---|
| 수탁자 목록 없음 | 누가 우리 데이터 만지는지 모름 | 플레이북 — 현행 목록 + 사전 통지 + 처리방침 게재 |
| "업계 표준 보안" | 의미 없음 | 구체적 통제 annex 요구, 또는 표준 reference (ISMS-P, ISO 27001, SOC 2) |
| 침해 통지 시한 없음 | 언제든 알림 | PIPA §34 72시간 floor 적용. 플레이북 입장 요구. |
| 감사 권리 전무 | 검증 불가 | 최소 독립 감사 보고서 요구 |
| 벤더가 "서비스 개선" 위해 데이터 사용 가능 | 우리 데이터로 학습 가능성 | 삭제; 처리는 서비스 제공에만 한정 |
| 국외이전 메커니즘 없음 | PIPA §28-8 5트랙 중 하나도 없음 | **현행 트랙 조사** — 별도동의 / 법률·조약 / 인증(ISMS-P 적정성·BCR 등) / 적정성 인정 / 안전조치. 트랙 미선택은 🔴. |
| 파기 약속 없음 | 데이터 영구 보유 | 플레이북 입장 + 요청 시 certify |

## 처리방침 일관성 점검

서명할 DPA는 처리방침이 cover 안 하는 것을 약속할 수 없고 그 반대도. **PIPA에서는 특히
처리위탁·국외이전 게재가 의무**:

- DPA가 X, Y, Z 목적 처리에만 commit — 처리방침이 그 목적을 나열?
- 처리방침 "처리위탁 없음"인데 DPA 체결 → 처리방침 갱신 필요 (§26②·③)
- 처리방침 "국외이전 없음"인데 DPA가 미국 인프라 → 처리방침 갱신 + 별도 동의 받는 메커니즘
  필요 (§28-8②)
- 처리방침이 특정 수탁자 카테고리 명명 — DPA 수탁자 목록이 매칭?

mismatch 플래그. 보통 처리방침이 stale인 것이지 DPA가 잘못인 것이 아니지만, 누군가 하나를
고쳐야 함.

## 레드라인 입자성

**가능한 가장 작은 입자에서 편집.** 레드라인은 협상 artifact지 rewrite가 아니다. 대규모
조항 교체는 "당신 drafting 다 버림" 신호 — aggressive, 상대방이 전체를 다시 읽게 강제, fine한
부분도 버림. 외과적 레드라인 — 단어 삭제, 구문 삽입, 부조항 재구성 — "우리는 구체적 ask가
있음" 신호, 읽기·이해·수용 빠름.

플레이북 입장 달성하는 가장 작은 편집을 디폴트로:
- **단어** 교체 먼저 ("(12)" → "(24)")
- **구문** 교체 ("매수인 부담" → "매수인 부담 및 의무")
- **부조항** 재구성 (compound 조건 (a)·(b)로 분리)
- **문장** 교체
- **조항 전체** 교체는 상대방 버전이 외과 편집보다 새 초안이 읽기 쉬울 때만 — 그 경우
  transmittal에 명시: "§8.2를 mark up 대신 교체했습니다. 변경이 광범위했기 때문 — delta walk
  through 가능합니다."

의심 시 더 작게. 외과적 레드라인 받은 의뢰인은 당신이 carefully 읽었다고 신뢰. 대규모
교체 받은 의뢰인은 당신이 읽기는 했는지 의심.

## 국외이전 메모

DPA가 국경 간 데이터 이전을 상정하면, **현행 PIPA §28-8 트랙 조사**:
- 별도 동의 (정보주체로부터 받은 별도 동의 + 처리방침 게재 §28-8② 필수 항목)
- 법률·조약의 특별 규정
- 개인정보위 인증
- 적정성 인정국 (현재 EU, EEA, UK 등 — 변동 가능, 현황 확인)
- 안전조치 + 정보주체 권리 보장 (PIPA §28-8①5호)

각 origin/destination 쌍에 대해: 적용 regime, 트랙 선택, 처리방침 게재 사항(이전국·받는
자·항목·목적·보유기간·이전 방법). 1차 소스 인용 + 현행성 확인 — 적정성 인정·트랙 요건은
PIPC 고시 변경 가능.

이전 메커니즘 없는데 국외이전 있음 → 🔴 — 적법 이전 메커니즘 없음.

## 게이트: DPA 서명

DPA 검토는 리서치. *서명* — 또는 우리를 위해 부서명하라고 지시 — 결과적 행위.

**DPA 서명·부서명·자동 실행 동의 전:** 역할이 변호사가 아니면:

> 위·수탁계약 서명은 법적 행위 — 회사를 정보주체·규제 당국에 대한 구체적 의무에 binding.
> 변호사 검토 거치셨나요? Yes면 진행. No면 brief:
>
> [1쪽 요약: 상대방, 방향(우리가 수탁자/위탁자), 플레이북 이탈 조항과 해결, 미해결 폴백
> 결정, 실행 전 변호사에게 물을 세 가지.]
>
> 자격 변호사: 대한변호사협회(`https://www.koreanbar.or.kr`), 한국개인정보보호법학회.
