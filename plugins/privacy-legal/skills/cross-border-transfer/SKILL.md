# /cross-border-transfer

1. `~/.claude/plugins/config/claude-for-legal-kr/privacy-legal/CLAUDE.md` 로드.
2. 국외이전 시나리오 분석: 이전 데이터, 이전받는 자, 목적지 국가, 이전 목적, 이전 방법.
3. PIPA §28-8 5트랙 중 가장 적합한 적법 근거 선택.
4. 처리방침 게재 6개 항목 확인.
5. 안전조치 체크리스트.
6. 처리방침·동의 메커니즘 갱신 권고 출력.

```
/privacy-legal:cross-border-transfer "AWS us-east-1에 한국 회원 개인정보 저장"
```

```
/privacy-legal:cross-border-transfer "글로벌 모기업과 직원 인사 데이터 공유"
```

# 국외이전 적법성 평가

## 매터 컨텍스트

실무 수준 CLAUDE.md `## 매터 워크스페이스` 확인. 비활성이면 skip. 활성이면 묻기.

## 목적

한국 PIPA §28-8 (구 §17① 단서·시행령 §17 등에서 2023.9.15. 대개정으로 §28-8 신설)은
개인정보의 국외이전을 5개 트랙으로 규정. 트랙 미선택 또는 처리방침 게재 누락은 위법.
이 스킬은 시나리오별 트랙 선택 + 게재 항목 + 안전조치 확인.

**왜 별도 스킬인가:** 한국 SaaS·플랫폼이 AWS·GCP·Azure 등 해외 클라우드를 쓰는 것이 일상화
되어 있고, 글로벌 본사·자회사 간 데이터 공유도 빈번. PIPA §28-8은 자주 위반되며 PIPC 시정조치·
과징금 사례가 누적되어 있어 별도 분기 분석 가치 있음.

법망에서 현행 텍스트 가져오기:
```

```

## 5개 트랙 개요

### 트랙 1: 별도 동의 (§28-8①1호)

정보주체로부터 다른 동의와 **명백히 구분된 별도 동의**를 받음.

**요구 조건:**
- 다음 6개 항목 모두 고지: 이전받는 자, 이전받는 자 연락처, 이전 항목, 이전 목적, 이전받는 자의 보유·이용 기간, 이전 방법
- 거부 권리와 거부 시 결과 안내
- 서비스 약관에 묻어두면 안 됨

**언제 사용:** 가장 일반적·간단. B2C 서비스에서 회원가입 시 받기 좋음.

**단점:** 동의 거부 시 서비스 제공 불가하면 동의의 자발성 issue (필수 동의 vs 선택 동의).

### 트랙 2: 법률 또는 조약의 특별 규정 (§28-8①2호)

다른 법률·조약이 국외이전을 구체적으로 요구·허용.

**예:**
- 금융실명거래법: 해외 거래 신고
- 자유무역협정의 데이터 흐름 조항
- 조세 협정상 정보 교환

**언제 사용:** 매우 제한적. 특정 산업·거래에서만.

### 트랙 3: 개인정보보호위원회 인증 (§28-8①3호)

이전받는 자가 PIPC가 고시한 인증을 받음. 시행령 §29-7.

**현재 인증 (2026.5 기준 — 변경 가능, 확인 필수):**
- ISMS-P (정보보호 및 개인정보보호 관리체계)는 국내 인증 — 해외 이전받는 자에 직접 적용 어려움
- 해외 인증 중 PIPC가 적정성 인정 절차로 인정한 것 (현재 거의 없음)

**언제 사용:** 향후 PIPC가 외국 인증을 적정성 인정하면 활용 가능. 현재는 트랙으로 거의 미사용.

### 트랙 4: 적정성 인정국 (§28-8①4호)

PIPC가 보호 수준이 한국 PIPA와 실질적으로 동등하다고 인정한 국가·지역. 시행령 §29-8.

**현재 적정성 인정 (2026.5 기준 — 변경 가능):**
- **EU·EEA** (2021.12.18. 적정성 결정 — EU 측에서 한국을 적정 국가로 인정 + 한국 측 인정)
- **영국** (2022 적정성 결정)
- 기타 PIPC 고시 확인 필수

**언제 사용:** 이전받는 자가 적정성 인정국에 있으면. AWS Frankfurt(독일·EU)·Azure West Europe·
GCP europe-west는 EU GDPR 하에 있어 트랙 4 적용 가능 (단, 실제 데이터 처리 위치 확인 필요).

**중요 함정:** AWS us-east-1·GCP us-central1 등 **미국 리전은 적정성 인정 안 됨** (2026.5 기준).
미국 인프라 사용은 트랙 1·5 사용 필요.

### 트랙 5: 안전조치 + 정보주체 권리 보장 (§28-8①5호)

이전받는 자와 다음 조치를 약속하는 계약·기업 내 규칙을 체결:
- 시행령 §29-9의 안전조치
- 정보주체 권리 행사 절차

**시행령 §29-9의 안전조치:**
- 암호화 등 기술적 안전조치
- 접근통제·접속기록
- 침해사고 통지 절차
- 정보주체 권리행사 응답 메커니즘
- PIPC 자료요구 응답 메커니즘

**언제 사용:** 트랙 4 적용 불가(미국·일본·싱가포르 등) + 트랙 1 동의 받기 어려운 B2B·내부
이전. 글로벌 회사가 본사·자회사 간 이전에 자주 사용 (Binding Corporate Rules 비슷).

**계약 내용:**
- 표준 계약 조항 (PIPC가 모범 양식 제시 가능 — 확인 필요)
- 또는 자체 협의 (다국적 본사·자회사 간 internal data transfer agreement)
- 또는 위·수탁계약에 포함

## 워크플로우

### Step 1: 시나리오 분석

사용자에게 묻기 (또는 paste된 시나리오에서 추출):

- **이전 데이터:** 어떤 개인정보? 항목? 민감정보·고유식별정보 포함?
- **이전받는 자:** 누구? 우리 해외 자회사·본사 / 해외 수탁자(예: AWS·Salesforce) /
  완전히 별개의 제3자?
- **목적지 국가:** 어느 나라·리전? 데이터가 실제 저장·처리되는 위치
  (CDN edge 다수 vs 주 저장소)
- **이전 목적:** 백엔드 인프라 / 글로벌 서비스 운영 / 분석 / 마케팅 / 내부 인사관리 /
  법무·재무
- **이전 방법:** 직접 API 호출 / 클라우드 동기화 / 정기 batch / SFTP 등
- **보유 기간:** 이전받는 자가 얼마나 보유

### Step 2: 트랙 선택

다음 의사결정 흐름:

```
1. 법률·조약 특별 규정 있나? → Yes → 트랙 2
                          → No  → 2

2. 목적지가 PIPC 적정성 인정국? → Yes → 트랙 4 (단, 실제 처리 위치 확인)
                              → No  → 3

3. 이전받는 자가 PIPC 인증 받음? → Yes → 트랙 3
                              → No  → 4

4. 정보주체로부터 별도 동의 가능? → Yes (B2C 회원가입 시 등) → 트랙 1
                              → No (B2B 인프라 사용 등) → 트랙 5

5. 트랙 5 안전조치 + 정보주체 권리 보장 계약 체결 가능? → Yes → 트랙 5
                                                  → No  → 이전 불가
```

**다중 트랙 가능:** 한 이전 활동에 여러 트랙 결합 가능. 예: AWS Frankfurt(EU) 사용 + EU 적정성
인정(트랙 4) + 추가 안전조치 계약(트랙 5).

### Step 3: 처리방침 게재 6개 항목 (§28-8② + 시행령 §29-10)

트랙 1·5 선택 시 처리방침에 다음 6개 항목 게재 의무 (§28-8②):

| # | 항목 | 본 시나리오 |
|---|---|---|
| 1 | 이전받는 자의 성명·명칭 (단체·법인의 경우 그 성명·명칭과 연락처) | [기재] |
| 2 | 이전 항목 | [기재] |
| 3 | 이전받는 국가 | [기재] |
| 4 | 이전 목적 | [기재] |
| 5 | 이전받는 자의 보유·이용 기간 | [기재] |
| 6 | 이전 방법 | [기재] |

**트랙 4 (적정성 인정국)도 처리방침 게재가 권장**됨 — 의무는 아니나 transparency.
**트랙 2 (법률·조약)는 법령 인용을 처리방침에 게재**.

### Step 4: 안전조치 체크리스트 (§29-9)

모든 트랙에서 적용 (트랙 5는 명시 의무, 다른 트랙도 사실상 필요):

| 안전조치 | 확인 |
|---|---|
| 전송 암호화 (TLS 등) | ☐ |
| 저장 암호화 (이전받는 자 측에서도) | ☐ |
| 접근통제 (IAM, 로그) | ☐ |
| 접속기록 보관 (1년 이상, 민감정보·고유식별정보는 3년) | ☐ |
| 침해사고 통지 절차 (이전받는 자 → 위탁자, PIPA §34 기한 만족) | ☐ |
| 정보주체 권리행사 응답 메커니즘 (이전받는 자 → 위탁자 통한 응답) | ☐ |
| PIPC 자료요구 응답 메커니즘 | ☐ |
| 종료 시 파기·반환 메커니즘 | ☐ |

### Step 5: 추가 고려사항

#### 민감정보·고유식별정보의 국외이전

§23·§24·§24-2의 별도 동의가 필요한 항목을 국외이전 시 **이중 동의** 필요:
1. 민감정보·고유식별정보 처리에 대한 별도 동의 (§23·§24)
2. 국외이전에 대한 별도 동의 (§28-8①1호)

각각 명백히 구분되어야.

#### 처리위탁과 국외이전 동시 발생

해외 수탁자에 위탁하면 두 가지 모두 적용:
- PIPA §26: 처리위탁 (위·수탁계약 + 처리방침 게재)
- PIPA §28-8: 국외이전 (5트랙 + 처리방침 게재)

위·수탁계약에 §29-9 안전조치 포함하면 트랙 5 + §26 의무 동시 만족.

#### 클라우드 사용 시 흔한 함정

- **"AWS 사용해도 데이터는 한국 리전이라 국외이전 아님" — 위험.** 한국 리전 데이터도 AWS
  US 본사의 운영진이 접근 가능 (인프라 관리·법령 협력). 약관·DPA 확인 필요. 보수적으로
  국외이전으로 처리하는 것이 안전.
- **"CDN edge에서 일시 통과만 — 이전 아님" — 모호.** PIPC 명확 가이드라인 없음. 보수적
  처리 권장.
- **"SaaS 벤더의 미국 본사에 데이터가 backup될 수도" — 확인 필수.** DPA에서 데이터 location
  명시 확인.
- **"개인정보 아닌 정보만 이전" — 진짜인가 확인.** IP 주소·디바이스 ID·쿠키 ID는 개인정보
  해당 가능.

```markdown
- [ ] 위·수탁계약 갱신 (§26 동시 적용)
- [ ] 변호사 검토 후 시행
```

## 흔한 실수

- **"미국 인프라 쓰는데 트랙 4 (EU 적정성)" — 틀림.** EU 적정성은 EU·EEA·영국만. 미국은 별도.
  AWS·GCP·Azure US 리전은 트랙 1·5 필요.
- **"적정성 인정국이면 처리방침 게재 안 해도 돼" — 의무는 트랙 1·5만, 트랙 4는 강력 권장.**
  실무상 게재하지 않으면 PIPC 점검에서 transparency issue.
- **"위·수탁계약 체결했으니 트랙 5 자동 충족" — 시행령 §29-9의 안전조치 + 정보주체 권리
  보장 명시 포함 확인.**
- **"이전받는 자가 한국 자회사 — 국외이전 아님" — 자회사도 별도 법인이면 §17 제3자 제공
  + 위치가 한국이면 §28-8 비적용. 단 모회사·해외 자회사는 §28-8 적용.**
