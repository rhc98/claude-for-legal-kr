# 개인정보처리방침 모니터

## 목적

개인정보처리방침은 한 방향으로 실무와 drift: 실무는 전진, 정책은 뒤에. PIA가 새 데이터
카테고리 승인. DPA가 어디에도 안 적힌 수탁자와 체결. 트리아지 결과가 새 use case를 정책이
아직 안 한 공개 요건과 함께 조건부로 표시. 몇 달 뒤, 누가 정책 읽고 실제 일어나는 것과 안 맞음.

이 스킬은 drift가 문제되기 전 잡음 — 산출물 폴더 주간 crawl, 또는 직접 질문 응답: "X 시작
하려는데 정책에 어떤 의미?"

출력은 항상 같음: 여기 갭, 여기 제안 문구.

**한국 PIPA 맥락에서 특히 중요한 것:**
- **처리방침 게재 의무는 법정 의무** (PIPA §30) — 미게재·허위 게재는 §74 과징금 대상
- **처리위탁** (PIPA §26② 게재)·**국외이전** (PIPA §28-8② 6개 항목 게재)는 빈번한 drift 영역
- **개인정보 보호책임자(CPO) 정보** 갱신은 §30②4호 의무
- **이용·제공 내역 통지** (§35-3, 시행령 §44-2) — 일정 규모 이상 사업자는 연 1회 정보주체 통지

약속은 요약; 실제 문서가 편집 제안의 authoritative.

### 개인정보 약속은 여러 surface에 존재 — 모두 sweep

웹사이트 처리방침은 한 surface. 현대 개인정보 프로그램은 규제 당국이 적극 scrutinize하는
최소 4개 다른 곳에 binding 약속:

1. **쿠키 동의 배너 / CMP** (예: OneTrust, Cookiebot, 자체). PIPC와 EU DPA 모두 CMP
   misconfiguration에 집행 사례 있음. 한국에서 정보통신망법 §22 쿠키 등 자동 수집 정보
   고지·동의가 별도 issue.
2. **앱스토어 개인정보 라벨.** Apple App Privacy("영양 표시")와 Google Data Safety는
   자기 신고이고 PIPC·FTC·각국 데이터 보호 당국이 enforceable. 처리방침 갱신했는데
   앱스토어 라벨 갱신 안 하면 material한 규제 당국-가시적 불일치.
3. **인앱 동의 화면.** 사용자가 데이터 사용 선택하는 실제 화면(온보딩 동의·설정 토글·
   "정책 변경" 다이얼로그). 정책이 무엇을 한다고 말함; 동의 화면이 사용자가 동의한 것을
   말함. 일치해야.
4. **분야별 고지.** 금융(신용정보 활용체제), 의료(개인정보 처리방침 외 의료법 고지),
   교육(초·중등 교육법령), 통신(전기통신사업법 약관). 각각 자체 갱신 의무와 일관성 요건.

## Sweep 모드 워크플로우

### Step 1: 산출물 폴더 scan

`## 산출물` → 산출물 폴더 경로 읽기. 최근 sweep 이후 (또는 지난 30일) 파일 enumerate:

- 승인된 PIA: 새 데이터 카테고리·새 목적·새 수탁자·새 제3자 제공·새 보유 기간 추출
- 서명된 DPA 검토: 새 수탁자, 새 국외이전, 새 데이터 흐름
- 트리아지 결과: 진행됐는데 정책 침묵인 활동, 조건부 진행됐는데 조건이 정책 변경 포함

### Step 2: 약속 대조 각 산출물 diff

각 산출물에 대해, 정책의 무엇과 비교:

| Drift 카테고리 | 정책이 말하는 것 | 산출물이 보여주는 것 | 갭 |
|---|---|---|---|
| 데이터 항목 | "수집: A, B, C" | PIA가 D 추가 | 필수 |
| 처리 목적 | "[목적 1, 2]" | DPA가 새 목적 enable | 필수 |
| 처리위탁 | "수탁자: X, Y" | DPA가 Z 추가 — 정책 미게재 | 필수 — PIPA §26② 위반 |
| 국외이전 | 미게재 | DPA가 미국 인프라 — 별도동의 없이 | 차단 — PIPA §28-8 위반 |
| 보유 기간 | "[기간]" | PIA가 더 긴 기간 | 필수 |
| 정보주체 권리 | "열람·정정·삭제" | 새 데이터 항목에 권리 절차 미적용 | 권장 |
| 자동화 결정 | 미게재 | AI 기능 출시 | 필수 — PIPA §37-2 시행 |

### Step 3: 갭 분류

- **필수** — 정책이 현재 실무를 misrepresent. PIPA §30·§26·§28-8 등 게재 의무 위반 가능.
  즉시 갱신.
- **권장** — 정책 침묵. 위반은 아니나 transparency 측면에서 보완 권장.

### Step 4: 각 갭에 대한 문구 초안

각 필수 갭에 대해:

```markdown
### 갭 #N: [짧은 이름]

**산출물:** [PIA 또는 DPA 인용]

**정책 현재 문구:** "[인용]"

**갭:** [구체적]

**제안 갱신 문구:**

"[새 문구 — 정책 톤에 맞춰]"

**근거:** [PIPA 조항 + 산출물 인용]

**시점:** [즉시 / 다음 정책 갱신 사이클 / 활동 시작 전]
```

### Step 5: CLAUDE.md 갱신

`## 산출물` → 최근 정책 sweep 날짜를 오늘로 갱신. 발견된 갭은 시정 추적 목록에 추가.

## 직접 질의 모드 워크플로우

사용자가 제안 신규 실무 설명:

> "마케팅이 행동 데이터를 온보딩 이메일 개인화에 사용하려 함"

워크플로우:
1. 제안 실무 parse: 어떤 데이터, 무슨 목적, 누구에게, 새 수집인가
2. config의 정책 약속 대조 diff
3. 실제 정책 문서도 읽기 (config 요약 위에)
4. 출력: cover됨 / 빠짐 / 충돌

```markdown
[WORK-PRODUCT 헤더]

# 정책 점검: [제안 실무 한 줄]

## 핵심 결론

[현재 정책이 cover / 갱신 필요 / 충돌 — 한 문장]

## Diff

| 측면 | 정책 현재 | 제안 실무 | 상태 |
|---|---|---|---|
| 데이터 항목 | [현재] | [제안] | 🟢/🟡/🔴 |
| 처리 목적 | [현재] | [제안] | |
| 동의 요건 | [현재] | [제안] | |
| 보유 기간 | [현재] | [제안] | |
| 제3자 제공 | [현재] | [제안] | |

## 갭 또는 충돌

[갭 있으면 위 Step 4의 같은 포맷 사용]

## 시점

- 행동 가능: [지금 / 정책 갱신 후 / 별도 동의 받은 후]
- 정책 갱신 효력: [고지 후 즉시 / 사용자 통지 후 / 시점별 다름]

## 다음 단계

[활동 진행, 정책 갱신, 또는 둘 다]
```

## 외부 surface도 sweep (앱스토어 라벨·CMP·인앱 동의)

산출물 폴더만 보는 것은 부분적. CLAUDE.md `## 산출물`의 다음을 확인:
- **쿠키 동의 배너 / CMP** — 최근 재구성 날짜, 정책의 쿠키 카테고리와 일치?
- **앱스토어 개인정보 라벨** — 최근 갱신 날짜, 정책의 데이터 카테고리·목적과 매칭?
- **인앱 동의 화면** — owner, 최근 검토 날짜, 정책 약속에 매칭?
- **분야별 고지** — 적용 regime별, 정책과 일관?

이런 surface가 stale이면 별도 갭 항목.

## PIPA-특화 sweep 점검

특별히 확인할 항목:

- **처리위탁 게재** (PIPA §26②·③) — 모든 활성 수탁자가 처리방침의 처리위탁 표에?
- **국외이전 게재** (PIPA §28-8② 6개 항목) — 이전국·받는 자·항목·목적·보유기간·이전방법 모두?
- **CPO 정보** (PIPA §30②4호) — 성명·소속·연락처 현행?
- **자동화 결정 게재** (PIPA §37-2 시행 — 2024.3.15.) — AI 시스템 도입했는데 처리방침에 미게재?
- **이용·제공 내역 통지 의무** (PIPA §35-3) — 일정 규모 이상은 연 1회 통지 의무
- **만 14세 미만 아동 처리** (PIPA §22-2) — 별도 게재 + 법정대리인 동의

## 검토자 메모 + 다음 단계 트리

`## 산출물` 패턴대로.
