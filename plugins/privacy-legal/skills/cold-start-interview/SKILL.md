# /cold-start-interview

1. `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` 확인 — populate되어
   있고 `--redo`가 아니면 덮어쓰기 전 확인.
2. 아래 인터뷰 워크플로우 실행.
3. 시드 문서: 개인정보처리방침(URL 또는 파일), 표준 위·수탁계약, 표준 PIA 하나. 셋 다 읽기.
4. 추출: 처리방침 약속, 위·수탁계약 입장(stated와의 delta 메모), PIA 구조.
5. 마이그레이션: populate된 CLAUDE.md(placeholder 없음)가
   `~/.claude/plugins/cache/claude-for-legal/privacy-legal/*/CLAUDE.md`에는 있는데
   config 경로에 없다면, config 경로로 복사하고 무엇이 마이그레이션되었는지 사용자에게 보여줌.
6. `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` 작성(필요시 상위
   디렉토리 생성). 요약 보여주기. 첫 작업 제안.

## `--check-integrations`

통합 가용성 점검(문서 저장소·Slack·스케줄드 태스크·**법망 MCP**)을 재실행하고
`~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md`의
`## 연결 가능한 통합` 섹션을 갱신. 재인터뷰는 하지 않음. MCP를 연결·해제한 뒤
플러그인이 인식하길 원하지만 전체 셋업을 다시 돌리고 싶지 않을 때 사용.

점검 시: MCP 도구 호출이 실제로 성공한 경우에만 ✓ 보고. 구성만 되어 있고 미테스트
커넥터는 ⚪로 표시하고 확인 방법 한 줄 첨부. `.mcp.json` 선언만으로 ✓ 보고하지 않음 —
사용자에게 "연결됨"으로 잘못 알려 trust를 깬다.

**법망(beopmang) MCP는 이 플러그인에서 1순위 통합이다.** 미연결이면:
- 모든 조문·판례 인용이 `[모델 지식 — 검증 필요]`로 떨어짐
- 환각방지 검증이 동작하지 않음
- 위임조문 체인 조회 불가
사용자에게 다음 안내: "법망 MCP를 등록하세요: `claude mcp add beopmang https://api.beopmang.org/mcp --transport http`.
인증 불필요, 무료. 자세한 내용은 `docs/BEOPMANG_INTEGRATION.md` 참조."

```
/privacy-legal:cold-start-interview
```

# 콜드스타트 인터뷰: 개인정보 실무

## 목적

*이* 개인정보팀이 어떻게 일하는지 학습 — 어떤 규제가 실제로 적용되는지, 위·수탁계약에서
무엇은 받아들이고 무엇은 거부하는지, 좋은 PIA가 여기서는 어떻게 생겼는지(어느 회사의
PIA와 어떻게 다른지). `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md`에
써서 다른 모든 스킬이 같은 이해에서 읽도록.

한국 개인정보 실무는 회사에 따라 크게 다르다. B2B SaaS 수탁자는 B2C 컨슈머 앱
위탁자와 공통점이 거의 없다. 인터뷰는 무엇이 무엇인지를 다른 어떤 것보다 먼저 알아낸다.

## 콜드스타트 점검

`~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` 읽기:
- **존재하지 않음** → 인터뷰 시작.
- **`<!-- SETUP PAUSED AT: -->` 포함** → 사용자를 맞이하고 해당 섹션부터 재개 제안.
- **`[PLACEHOLDER]` 마커가 있는데 pause comment는 없음** → 템플릿이 완료되지
  않았음; 새로 시작할지 placeholder부터 재개할지 제안.
- **populate됨 (placeholder도 pause comment도 없음)** → 이미 구성됨; `--redo`가
  아니면 skip.

템플릿 구조는 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`에 있다 — 섹션 골격으로 사용.
완성된 실무 프로파일을 config 경로에 쓰고 필요시 상위 디렉토리 생성.

옛 캐시 경로 `~/.claude/plugins/cache/claude-for-legal/privacy-legal/*/CLAUDE.md`에
CLAUDE.md가 있는데 config 경로에는 없다면, 앞으로 복사.

## 공유 회사 프로필 확인

`~/.claude/plugins/config/claude-for-legal/company-profile.md` 확인.

- **있으면:** 읽기. 한 줄 확인 보여주기: "당신은 [성명], [실무 환경], [회사], [업종],
  [관할]에서 운영. 맞나요? (또는 '업데이트'를 말해 공유 프로필 변경.)" 확인되면
  회사 질문은 skip — 바로 플러그인 특화 질문으로.
- **없으면:** 당신이 이 사용자가 셋업한 첫 플러그인. 오리엔테이션과 fork 후, 회사
  질문을 묻고 공유 프로필에 쓴다(플러그인 루트의 `references/company-profile-template.md`
  템플릿 따름). 그 후 플러그인 특화 질문 계속. 사용자에게: "회사 프로필 저장 완료 —
  다른 법무 플러그인들이 이걸 읽고 이 질문들을 skip합니다."

공유 프로필에 속하는 회사 질문(있으면 재질문 금지): 실무 환경, 회사명, 업종,
무엇을 파는지, 규모, 관할, 규제 당국, 위험 성향, 에스컬레이션 담당자. 플러그인
특화 질문(플레이북 입장, 검토 프레임워크, 하우스 스타일, 감독 모델 등)은 플러그인별로 유지.

## 설치 스코프 점검

오리엔테이션 전, 작업 디렉토리가 프로젝트 내부(사용자 홈이 아닌)임을 알아채면 한 번
플래그:

> **참고 — 이 플러그인이 프로젝트 스코프로 설치된 것 같습니다. 그러면 저는 [현재
> 디렉토리] 안의 파일만 읽을 수 있습니다. 다른 곳(다운로드·문서·드롭박스)의 문서를
> 읽게 하려면 user 스코프로 설치하세요 — QUICKSTART 참조. 프로젝트 스코프로 계속해도
> 되지만, 파일을 이 폴더로 이동해야 합니다.**

진행 전 사용자 확인 요청: 프로젝트 스코프로 계속, 또는 잠시 멈추고 user 스코프로
재설치. 작업 디렉토리가 사용자 홈이면 이 점검을 침묵으로 skip.

## 인터뷰 시작 전

다른 어떤 것을 묻기 전, fork-first 프리앰블 — 3-4줄, 더 길게 하지 않음:

> **`privacy-legal`은 개인정보 프로그램을 운영하는 사람들을 위한 플러그인입니다:
> PIA 작성, 위·수탁계약 검토, 정보주체 권리행사 응답, 정책 갭 분석.** 본인 영역이
> 아니신가요? `/legal-builder-hub:related-skills-surfacer`.
>
> **2분**이면 역할, 위·수탁계약에서 어느 쪽인지(위탁자/수탁자/양쪽), 주요 관할이
> 설정되고 다른 모든 것은 sensible 디폴트로 채워집니다. **15분**은 위·수탁계약
> 플레이북 입장(위탁자·수탁자 양면), 표준 PIA에서 추출한 PIA 템플릿 구조, 전체 규제
> 풋프린트, 처리 활동 시드를 추가합니다.
>
> 빠른 셋업 or 풀? (언제든 `/cold-start-interview --full`로 업그레이드.)

사용자 선택을 기다린 뒤 다른 어떤 것도 보여주지 않음.

## 사용자가 빠른/풀을 선택한 후

선택되었으면, 첫 인터뷰 질문 전 오리엔테이션:

> "이 플러그인은 실무 프로파일(위·수탁계약 플레이북, PIA 하우스 스타일, 규제 풋프린트),
> 처리활동 등록부, 활동별 PIA·위·수탁계약 검토를 유지합니다. 이 셋업 인터뷰는 실제
> 일하는 방식 — 실무, 위·수탁계약 입장, PIA 하우스 스타일 — 을 학습하고 플러그인이
> 매번 참조할 평문 파일에 씁니다. 답한 모든 것은 나중에 바꿀 수 있습니다. 끝나면
> 플러그인 커맨드는 일반 템플릿이 아닌 당신이 일하는 방식대로 동작합니다."
>
> 그 뒤: "셋업은 당신 답변에서 fresh한 전문가 프로파일을 만듭니다. 개인 Claude 기록,
> 다른 대화, 홈 디렉토리 CLAUDE.md를 읽지 않습니다. 대화 컨텍스트에서 관련 정보를
> 발견하면 — 예: 앞에서 회사를 언급하신 경우 — 사용 전 묻겠습니다. 타이핑하거나
> 승인하지 않은 개인 정보는 실무 구성에 들어가지 않습니다."
>
> 그 뒤: "준비됐나요? 빠른 질문 몇 개 먼저, 그 뒤 더 깊이."

**왜 중요한가.** 이 플러그인의 모든 커맨드는 이 인터뷰가 쓴 구성을 읽습니다. 일반
구성은 일반 출력을 줍니다 — 디폴트 위·수탁계약 입장, 디폴트 PIA 포맷, 디폴트 권리행사
워크플로우, 그리고 당신의 B2B 수탁자 협의를 컨슈머 위탁자처럼 다루는 검토. 플러그인에게
실제 규제 풋프린트, 실제 위·수탁계약 입장, 실제 PIA 하우스 스타일을 말하는 것이 "개인정보
AI 도구"와 "당신 프로그램이 일하는 방식대로 동작하는 도구"의 차이를 만듭니다. 답이
구체적일수록 출력이 당신 것처럼 느껴집니다.

실무 프로파일은 사용자의 타이핑 답변과 세 시드 문서에서만 채운다. `~/CLAUDE.md`를
읽거나 ambient 컨텍스트에서 실무 사실을 끌어오지 않는다. 대화에서 이미 보이는
관련 정보가 있으면 사용 전 묻는다.

**빠른 셋업 경로:** Part 0(역할·실무 환경·통합)과 규제 풋프린트만 묻고, 나머지는
`[DEFAULT]` 마커로 config 작성. 마무리: "완료. 이제 커맨드를 쓸 수 있습니다. 위·수탁계약
입장, 권리행사 응답 timing, PIA 임계점에 sensible 디폴트를 썼습니다. 스킬 출력이
부자연스러우면 보통 튜닝해야 할 디폴트 — 어느 것인지 알려드릴 겁니다.
`/privacy-legal:cold-start-interview --full`을 언제든 실행해 전체 인터뷰, 또는
`/privacy-legal:cold-start-interview --redo <섹션>`으로 한 부분만 재실행."

**풀 셋업 경로:** 아래의 기존 인터뷰 흐름.

## 인터뷰 페이싱

- **답이 어딘가 적혀 있을 것이라고 가정.** 회사 설명·플레이북·에스컬레이션 매트릭스·스타일
  가이드·핸드북·관할 목록·매터 포트폴리오처럼 어딘가 적혀 있을 가능성이 높은 정보를
  묻는 질문은, 메모리에서 타이핑하라 하기 전에 link 또는 paste를 요청한다. "링크
  또는 doc를 paste 하거나 짧은 버전을 주세요"가 한 문장 이상 답이 필요한 모든 것의
  디폴트 ask. 사람들이 이미 적은 것을 재타이핑하게 하는 인터뷰어는 인터뷰어의 첫
  업무에 실패한 것이다.
- **batch 크기 — 부분 질문 카운트.** "한 turn에 2-3 질문 이상 묻지 않음"은 답할 수
  있는 prompt 2-3개를 의미하며, 부분 질문도 카운트. 부분 질문 5개 있는 질문 하나는
  5개. 테스트: 사용자가 스크롤 없이 답할 수 있나? 질문이 한 화면에 안 들어가면 너무
  많다. 가능하면 구조화된 tap-through 질문 선호 — 스크롤·타이핑 불필요.

**진짜 답에 멈춤.** 어떤 질문은 빠른 tap-through 답(위탁자 vs 수탁자, 규제 풋프린트).
어떤 질문은 사용자가 타이핑·설명·문서 업로드 필요(처리방침·표준 위·수탁계약·표준 PIA·
위·수탁계약 협상 입장·권리행사 시스템 목록). 빠른 tap보다 더 필요한 질문:

- **묻고 기다림.** 명시적으로 말함: "이건 타이핑 답이 필요합니다 — 기다리겠습니다."
  사용자 답 전에 다음 질문으로 안 감.
- **시드 문서 업로드:** "내용을 paste하거나, 파일 경로/URL 공유, 또는 '나중에 skip' 말씀.
  skip하면 실무 프로파일에 갭 플래그가 붙어 나중에 채울 수 있습니다." 그 뒤 실제로
  기다림.
- **실무 프로파일 쓰기 전:** 인터뷰 검토. skip되거나 placeholder로 답한 질문 모두
  목록화(특히 세 시드 문서와 위·수탁계약 입장). 말함: "실무 프로파일을 쓰기 전, 아직
  열린 것: [목록]. 지금 채우시겠습니까, 아니면 placeholder로 두시겠습니까?" 그 뒤 답
  기다림.
- **침묵의 갭이 있는 실무 프로파일을 절대 쓰지 않음.** 모든 `[PLACEHOLDER]`는
  사용자가 의도적으로 skip한 결정이지, 지나간 질문이 아님. 위·수탁계약 템플릿이나
  표준 PIA가 skip되면 `[POSITIONS UNTESTED]`로 표시해 하류 스킬이 인지하게 함.
- **일시정지·재개.** 사용자에게 미리 말함: "멈춰야 하면 '잠시 멈춤'(또는 '나중에',
  '잠깐') 말씀해주세요. 진행 저장합니다. `/privacy-legal:cold-start-interview`를
  나중에 다시 실행하면 마지막 지점에서 재개합니다." 일시정지되면, 부분 구성을
  `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md`에 쓰고 상단에
  `<!-- SETUP PAUSED AT: [섹션 이름] — /privacy-legal:cold-start-interview를 실행해
  재개 -->` 코멘트와 답 없는 필드에 `[PENDING]` 마커(placeholder와 구분). 셋업이 재실행
  되어 일시정지 config를 발견하면, 사용자 환영: "다시 오셨네요. [섹션]에서 멈추셨습니다.
  이전 답이 저장되어 있습니다. 거기서 재개할까요, 처음부터 다시 할까요?" 이미 답한
  질문은 재질문하지 않음.

**셋업 중 사용자가 주장한 법적 사실을 검증.** 사용자가 인터뷰 질문에 특정 규정·법조
번호·판례·기한·임계점·관할·등록번호로 답할 때 — 그리고 sanity-check 가능한 것 —
구성에 쓰기 전 점검. 그들이 말한 것이 당신의 이해와 충돌하거나 그들이 paste한 것과
충돌하면 surface: "임계점을 X라 하셨는데, 제 이해는 Y입니다 — 프로파일에 어느 것이
들어갈지 확인해주실 수 있나요? `[전제 플래그 — 검증 필요]`" CLAUDE.md에 쓰인 잘못된
사실은 모든 미래 출력으로 전파되고, 여기서 잡는 것이 제품에서 가장 leverage가 높은
순간 중 하나.

**법망 MCP가 연결되어 있으면 적극 활용해 검증.** 예: 사용자가 "PIPA §28-8 국외이전은
적정성 인정국이면 자유롭게 가능"이라 답하면, `mcp__beopmang__law get` 호출로 §28-8
현행 텍스트를 가져와 5개 트랙(별도동의·법률조약·인증·적정성·안전조치) 모두 확인 후
"§28-8 5개 트랙 중 하나로 가능하며 적정성 인정국 트랙도 그 중 하나입니다. 자유롭게는
아니고 처리방침 게재 의무는 여전히 있습니다 — 프로파일에 어떻게 쓸까요?" 식으로
보완. 검증된 답은 검증 로그에 자동 기록.

#### 누가 쓰고 있나?

> 이 플러그인을 일상에서 누가 쓸 건가요? (이건 모든 스킬의 work-product 헤더와 출력

> framing과 법적 결과 단계 전 변호사 검토 체크포인트.)

> 1. **변호사 또는 법무 전문가** — 변호사, 변호사 자격 없는 사내 법무 실무자(변호사

>    프로그램 매니저, 사내·외부 변호사를 부를 수 있는 창업자.
> 3. **변호사 자격 없는 실무자 (정기적 변호사 상담 어려움)** — 본인이 직접 처리.

답이 2 또는 3이면, 한 번만(매 출력마다 반복하지 않음):

> 1. **출력을 결재가 아닌 변호사 검토용 리서치로 framing합니다.** "서명 OK" 대신

설명에서 프로파일을 만들되, 채워진/적응된/빈 템플릿 필드를 플래그. 강제로 맞춘

#### 무엇이 연결되어 있나?

> 이 플러그인은 다음과 작업할 수 있습니다: 문서 저장소(Google Drive, SharePoint),
> Slack, 스케줄드 태스크, 그리고 **법망 MCP**(한국 법령·판례). 어느 커넥터가
> 구성되어 있는지 점검하겠습니다 — 필요한 기능은 동작하고, 없는 기능은 침묵으로
> 실패하지 않고 수동으로 graceful하게 fallback합니다.

**구성된 것이 아닌 실제로 연결된 것을 점검.** `.mcp.json`에 나열된 커넥터는 *가용*.
실제 응답하는 커넥터는 *연결됨*. 이 둘은 다르며 혼동하면 trust가 깨진다. 이 플러그인이
쓰는 커넥터 각각에 대해:

- 연결을 테스트할 수 있으면(간단한 MCP 도구 호출 — list 또는 search), 성공 응답 시에만
  ✓ 보고.
- 테스트 불가(여기서 probe할 방법 없음)이면, ⚪ "구성됨이나 검증 안 됨 — MCP 설정에서
  확인하세요" + 한 줄 how-to.
- 구성만으로 절대 ✓ 보고하지 않음.

**법망 점검 구체:**
```
mcp__beopmang__law search "개인정보보호법"
```
응답 객체에 `laws` 또는 유사 필드와 1건 이상 결과가 있으면 ✓. 연결 오류·timeout·빈
배열이면 ✗.

연결되지 않은 커넥터에 대해 사용자에게 어떻게 연결하는지 알림. 예: "법망이 연결되지
않았습니다. Claude Code: `claude mcp add beopmang https://api.beopmang.org/mcp --transport http`.
이 플러그인은 법망 없이도 동작하지만 — 인용 검증이 모두 `[모델 지식 — 검증 필요]`로
떨어집니다 — 연결하면 환각방지 검증이 활성됩니다."

그 뒤 다음 형식으로 발견 보고:

> - ✓ [통합] — 연결됨 (테스트됨)
> - ⚪ [통합] — 구성됨이나 검증 안 됨. MCP 설정에서 확인.
> - ✗ [통합] — 발견 안 됨. [기능]은 [수동 대안]으로 fallback. [연결 방법]. 나중에
>   셋업하면 `/privacy-legal:cold-start-interview --check-integrations` 재실행.
>
> 전부 필요하진 않습니다. 핵심 기능은 파일 접근만으로 동작. **단, 법망 MCP가 없으면
> 인용 검증이 동작하지 않으니 의존 전 외부 검증이 필요합니다.**

### Part 2: 위·수탁계약 협상 입장 (3-4분)

*(이 입장은 `/dpa-review`에 영향 — 모든 incoming 위·수탁계약이 당신의 표준·폴백·
'절대 안 됨'으로 레드라인됨. 여기 잘못된 입장 = 매번 잘못된 레드라인.)*

구조화된 질문 전: "기존 위·수탁계약 템플릿, 협상 플레이북, 폴백 입장 메모가 있으면
paste 또는 파일 경로 공유 — 재타이핑 대신 입장을 추출합니다. 없으면 '없음' 말씀,
질문을 한 번에 하나씩 묻겠습니다."

사용자가 업로드하면: 읽기, 입장 추출, 발견한 것 확인, 해당하는 상세 질문 skip.

**위·수탁계약 플레이북을 업로드하지 않은 경우:** 이 섹션 끝에서 제안: "공유·유지할
독립 위·수탁계약 플레이북으로 write up할까요? 방금 실무 프로파일에 캡처한 같은 내용,
배포하거나 새 개인정보 직원에게 줄 수 있는 팀용 문서 포맷."


여기가 스킬의 진가가 발휘되는 곳 — 대부분 개인정보팀은 위·수탁계약 입장은 있지만
적어두지는 거의 않음.

**우리가 수탁자일 때 (고객이 자기 위·수탁계약을 보냄):**
- 표준 위·수탁계약을 우리가 push하나, 아니면 고객 종이(paper)를 받나?
- 안전조치(시행령 §30): SOC 2 보고서가 제안인가, 현장 감사 받는가?
- 침해통지: PIPA §34의 72시간(개인정보위 + 정보주체)을 그대로 commit하나, 아니면
  더 빠른 window를 약속받았나?
- 재위탁: PIPA §26⑥은 위탁자 사전동의 필수 — 이걸 우리가 어떻게 충족? 일괄 사전동의
  + 변경 통지, 아니면 매번 동의?
- 데이터 소재 약속: 리전 commit 가능, 아니면 "AWS가 두는 곳"?
- 종료 시 파기: 며칠, certify 발급?

**우리가 위탁자일 때 (우리가 벤더에 위·수탁계약을 보냄):**
- 같은 질문, 반대 polarity. 벤더에게 무엇을 *요구*하나?

**위·수탁계약에서 자동 reject가 되는 한 가지:**
- 자동 reject 조항은?

### Part 3: 하우스 스타일 (1-2분)

**PIA:** *(이건 `/pia-generation`에 영향 — 스킬은 트리거·포맷·깊이·사인오프를 스킬이
초안 잡는 모든 PIA의 디폴트 템플릿으로 사용.)*
- 회사에서 PIA를 트리거하는 것? 모든 신규 기능? 특정 카테고리만?
- 좋은 PIA의 길이 — 2장인가 20장인가?
- 사인오프 — 본인만, 검토위원회 있나?

**권리행사 (DSAR):** *(이건 `/dsar-response`에 영향 — 시스템 목록이 locate 단계를
구동, 핸들러가 runbook을 받음, SLA가 기한 계산을 구동.)*
- 월간 — 한 달에 한 건인가 백 건인가?
- 처리 — 본인, 아니면 runbook이 있는 지원팀?
- 권리행사가 닿는 시스템 — 사용자 데이터가 몇 군데 있나?
- 본인확인 방법 — 휴대폰 본인확인, 아이핀, 공동인증서, 이메일 챌린지, 또는 처리
  중인 정보로 확인? (PIPC는 과도한 본인확인을 시정조치한 사례가 있음 — 처리 중
  정보로 확인 가능하면 그게 우선.)

### Part 4: 시드 문서 (3-4분)

> 세 가지를 보고 싶습니다. 그것들이 실제 일하는 방식을 알려줍니다.
>
> 1. **현재 개인정보처리방침.** 공개된 것. 무엇을 commit했는지 이해하기 위해 읽음 —
>    모든 PIA와 위·수탁계약은 이것과 일치해야 함.
>
> 2. **표준 위·수탁계약 템플릿.** 고객(또는 벤더)에 push하는 것. 이게 stated 플레이북 —
>    말씀하신 것과 대조.
>
> 3. **만족하는 PIA 하나.** 완벽한 것이 아니라 *대표적인* 것. 구조·tone·깊이·skip하는
>    것을 학습.

**시드 문서 읽는 법:**

**처리방침:** 모든 약속 추출. 수집 항목, 처리 목적, 보유 기간, 제3자 제공, 처리위탁,
국외이전, 정보주체 권리. 이건 PIA 스킬이 대조해야 할 약속.

**표준 위·수탁계약:** 모든 조항을 인터뷰 답에 매핑. delta가 흥미 — "72시간 침해통지라
하셨는데 템플릿은 '지체 없이' — 실제 입장은?"

**PIA:** 구조를 템플릿으로 추출. 섹션 헤딩, 분석 깊이, 위험 진술 포맷. 이게 pia-generation
스킬의 디폴트 출력 포맷.

### Part 5: 산출물과 정책 문서 위치 (1분)

> "두 가지 마지막 — 정책을 현행으로 유지하기 위해 어디를 봐야 하는지 알아야 합니다."

- **완료된 PIA·위·수탁계약 검토·트리아지 결과를 어디 저장하나요?** 폴더 경로
  또는 공유 드라이브 위치. 여기가 정책-모니터 스킬이 실무가 처리방침보다 앞서갔는지
  감지하기 위해 crawl할 곳. (이건 `/policy-monitor`에 영향 — 이 경로 없으면 drift sweep은
  직접 질의 모드만 동작.)
- **공개된 처리방침 문서는 어디 있나요?** 고객·이용자에게 공유되는 그것. drift가
  발견되면 편집 제안을 위해 읽어야 함.
- **출력 파일에 명명 규칙이 있나요?** (예: `PIA_기능명_YYYY-MM-DD`) 또는 ad hoc?

산출물이 아직 저장되지 않으면:
> "괜찮습니다 — policy-monitor 스킬은 여전히 직접 질의 모드로 동작합니다
> ('X를 시작하려는데 처리방침이 다루나?'). crawl sweep은 산출물 저장 시작 전에는 scan할
> 것이 없을 뿐."

## 실무 프로파일 작성

플러그인 디렉토리의 `CLAUDE.md` 템플릿 구조를 사용. 모든 답을 적절한 섹션에 채우기.
`[PLACEHOLDER]` 마커는 사용자가 의도적으로 skip한 항목에만 남김(silent 갭이 아니라
명시적 결정).

자동화 가능한 검증:
- 사용자가 인용한 PIPA 조항은 `mcp__beopmang__law get` 호출로 현행 텍스트 가져와 대조
- 사용자가 인용한 판례는 `mcp__beopmang__case search`로 존재 확인
- 사용자가 인용한 PIPC 결정은 *법망 미커버* — 사용자에게 PIPC 공식 사이트 URL 부탁
  (`docs/DATA_GAPS.md` 참조)

작성 후, `verification-log.md`에 다음을 추가:

`[YYYY-MM-DD] 콜드스타트 인터뷰 - [성명]이(가) 다음을 확인: [검증한 항목 목록] - 법망에 대조해 [확인 / 정정]`

## 작성 후

**이 플러그인이 무엇을 도와줄 수 있는지 보여주기.** 닫기 전 제안:

> **제가 무엇을 도와드릴 수 있는지 보시겠습니까?**

yes면 다음 맞춤 목록 보여줌 (일반 템플릿이 아님 — 이 플러그인이 가장 잘하는 구체적인 것들):

> **개인정보 실무에서 제가 잘하는 것:**
>
> - **위·수탁계약을 플레이북으로 검토** — 예: "위탁/수탁 자동 감지; 당신 입장에서의
>   deviation 플래그." 시도: `/privacy-legal:dpa-review`
> - **처리 활동 트리아지** — 예: "PIA 의무·임의·진행 가능 — 처리방침 충돌 surface 포함."
>   시도: `/privacy-legal:use-case-triage`
> - **하우스 포맷으로 PIA 작성** — 예: "구조화된 인테이크, 위험 분석, 규제 분류, 권고."
>   시도: `/privacy-legal:pia-generation`
> - **정보주체 권리행사 walk-through** — 예: "본인확인, locate, 면제 평가, 응답 letter 초안."
>   시도: `/privacy-legal:dsar-response`
> - **새 규제를 정책 대조 diff** — 예: "갭 목록 출력 + owner·기한 있는 시정 계획."
>   시도: `/privacy-legal:reg-gap-analysis`
> - **정책 drift sweep** — 예: "저장된 PIA·위·수탁계약 검토·트리아지를 가로질러 정책이
>   더 이상 실무와 안 맞는 곳 찾기." 시도: `/privacy-legal:policy-monitor`
> - **민감정보·고유식별정보·주민번호 처리 적법성** (KR 전용) — 예: "PIPA §23·§24·§24-2
>   요건 자동 점검." 시도: `/privacy-legal:pipa-spi-handling`
> - **국외이전 5트랙 평가** (KR 전용) — 예: "별도동의/법률조약/인증/적정성/안전조치 트랙
>   분기 + 처리방침 게재 의무 체크리스트." 시도: `/privacy-legal:cross-border-transfer`
>
> **첫 번째 추천:** 하나의 실제 처리 활동에 `/use-case-triage`를 실행 — 플레이북이 맞는
> 분기를 잡는지 가장 빠르게 보는 방법. 또는 당신 plate에 무엇이 있는지 말씀, 제가 고르겠습니다.

이게 콜드스타트 문제(감독자가 첫 일을 모름)와 value-prop 문제(플러그인이 무엇을
하는지 모름)를 한 제안으로 해결. 목록을 구체적으로. 감독자가 인터뷰 중 구체적 첫
작업을 이미 명명했으면 이 단계 skip.

   > 프로파일을 줍니다.

## 실패 모드

- **PIPA 외 외국법을 가정하지 마라.** 일부 국내 회사는 "GDPR 신경 써야 한다고 들었다"고
  하지만 — 실제로 EU 정보주체가 있는지 물어보라. 한국 회사가 미국 본사·SaaS를 통해
  데이터를 처리해도 PIPA가 1차로 적용된다.
- **위탁자/수탁자 질문을 skip하게 두지 마라.** 확실치 않으면 walk through: "고객의
  사용자 데이터가 당신 시스템에 들어올 때, 어느 처리방침이 governing인가 — 당신
  것이나 고객 것?"
- **일반 입장에서 위·수탁계약 플레이북을 쓰지 마라.** 위·수탁계약을 많이 협상한 적이
  없으면 config CLAUDE.md에 명시: `[POSITIONS UNTESTED — 이 팀은 아직 많은 위·수탁계약
  을 협상하지 않음. 출발점이지 settled 입장이 아니라고 취급.]`
- **PIPC 결정례를 인용하면 법망에 없다고 명시.** 사용자가 "PIPC가 이렇게 결정했다"고
  말할 때, 법망에서 직접 verify 불가 — `[PIPC 결정례 — 직접 검증 필요]` 태그.
