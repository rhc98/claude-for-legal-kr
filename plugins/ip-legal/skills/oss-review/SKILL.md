# /oss-review

오픈소스 라이선스 컴플라이언스 점검을 `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md`
실무 프로파일 대조로 돌린다. 의존성을 라이선스 family로 분류하고, 의무를 배포 형태에 매핑하고,
라이선스-불명·OSS인 척하는 비-OSI 패키지를 플래그하고, 조치를 권고한다 — 준수, 교체, 제거, 변호사
검토, 상용 라이선스 확보.

가드레일·헤더·검토자 메모·이중 심각도·태그 어휘·결정 자세·산출물 포맷·대시보드 제안은 모두 이
플러그인의 `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md`가 정본이다 — 여기서
재진술하지 않고 가리킨다. 충돌 시 그 CLAUDE.md가 우선한다.

## 지침

1. **`~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md` 로드.** `[PLACEHOLDER]`가
   남아 있으면 멈추고 안내: "`/ip-legal:cold-start-interview`를 먼저 실행하세요 — 검토 전 실무
   프로파일(그리고 OSS 정책이 있으면 그것)을 알아야 합니다." 실무 프로파일이 업로드된 OSS 정책을
   가리키면 그것도 읽는다 — 이 팀에서 허용·검토·금지 라이선스의 정본이다.

2. **스코프 확정:** 의존성 목록(package.json, requirements.txt, go.mod, Gemfile, Cargo.toml,
   pom.xml, SBOM), 단일 라이브러리, 또는 팀이 오픈소스로 공개하려는 배포 예정 코드. 사용자가 경로를
   줬으면 파일에서 추론, 아니면 묻는다.

3. **배포 형태를 먼저 확정**한 뒤 의무를 분류한다 — SaaS, 배포 바이너리, 사내 전용, 임베디드. 같은
   의존성 목록도 이것에 따라 다른 의무를 트리거한다.

4. **아래 워크플로우를 따른다.** 특히:
   - 메타데이터가 아니라 **실제 라이선스 텍스트**를 읽는다 — LICENSE 파일이 틀릴 수 있고 메타데이터는
     stale일 수 있다.
   - 각 패키지를 허용형 / 약한 copyleft / 강한 copyleft / 퍼블릭도메인 / 비-OSI / 불명으로 분류.
   - 라이선스-불명은 디폴트 허용형이 아니라 "검토 필요"로 플래그.
   - OSS인 척하는 비-OSI source-available 라이선스(SSPL, BUSL, Commons Clause, Elastic License,
     fair-source)를 플래그 — 이들은 오픈소스가 아니다.
   - 배포 예정 코드는 선택한 outbound 라이선스가 임베디드된 모든 의존성과 양립하는지 점검.

5. **메모를 산출**한다 — 아래 템플릿대로: 문서 헤더, 핵심 결론, 메모 상단 플래그, 심각도별 패키지
   블록, 관할 노트, outbound 점검(해당 시), 승인 라우팅.

6. **결정 자세를 존중한다.** copyleft 트리거 분석이 다툼 있는 문제(AGPL의 "네트워크 상호작용",
   GPL-3.0의 "conveying", LGPL 링킹 범위)에 달려 있으면 `[검토]`로 플래그하고 양쪽으로 작용하는
   요소를 surface한다. 강한 copyleft 또는 라이선스-불명으로 분류된 것은 배포·공개 전 변호사에게 간다.

## 예시

```
/ip-legal:oss-review ~/code/my-project/package.json
/ip-legal:oss-review ~/code/my-project/requirements.txt
/ip-legal:oss-review redis
/ip-legal:oss-review ~/code/my-project  # 레포 루트 — 모든 매니페스트 스캔
```

## 더 잘 작동하는 연결

OSS 클리어런스 요청은 보통 티켓 시스템으로 들어온다. Jira·Linear·Asana에 연결되면 이 스킬은:
들어오는 OSS 요청 모니터링, 티켓에 직접 가이드 응답(불완전 정보 플래그, 레포 링크 요청, 라이선스
family 분류 반환), 요청 간 클리어런스 상태 추적을 할 수 있다. 커넥터 없으면 티켓을 paste하거나 요청을
설명하면 한 건씩 처리한다.

## 매터 컨텍스트

실무 수준 CLAUDE.md `## 매터 워크스페이스` 확인. `활성화`가 `✗`(사내 사용자의 디폴트)이면 이 단락
skip — 스킬은 실무 수준 컨텍스트 사용, 매터 머시너리 비활성. 활성이고 활성 매터 없으면: "어느
매터입니까? `/ip-legal:matter-workspace switch <slug>` 실행하거나 `practice-level` 말씀." 활성 매터의
`matter.md` 로드. 산출물은 `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/matters/<매터-slug>/`.
`매터 간 컨텍스트`가 `on`이 아니면 다른 매터 파일 절대 안 읽음.

## 목적

사용자에게 의존성 트리에 어떤 라이선스가 있는지, 코드가 어떻게 배포되는지에 따라 그 라이선스가 어떤
의무를 트리거하는지, 각각에 무엇을 해야 하는지 알려준다. 산출물은 변호사(또는 변호사 접근 가능한
엔지니어)가 행동할 수 있는 메모다 — 준수, 교체, 제거, 변호사 검토, 상용 라이선스 확보.

**이것은 1차 분류다.** copyleft 분석은 배포 형태·링킹 정도·관할, 그리고 때로 법원에서 검증되지 않은
법적 질문(특히 AGPL의 "네트워크 상호작용", GPL-3.0의 특허 조항)에 달려 있다. 강한 copyleft 또는
라이선스-불명으로 분류되는 모든 것은 배포·공개 전 변호사가 평가한다. 스킬은 발견한 것을 보고하고,
변호사가 무엇을 할지 결정한다.

> **한국법 컨텍스트.** 한국에서 오픈소스 라이선스는 **저작권법상 이용허락(라이선스) 계약**으로
> 이해된다 — 저작권자가 일정 조건 하에 복제·배포·개작을 허락하는 것이고, 조건 위반은 (a) 라이선스
> 범위 일탈에 따른 **저작권 침해**, (b) 라이선스 **계약 위반**의 두 측면을 가진다 `[모델 지식 — 검증
> 필요]`. GPL 등 copyleft 의무의 한국 법원 집행례는 제한적이며 `references/ip-currency-watch.md`
> 영역이다. 인용해야 하면 `[검증 필요]`. 외국 준거법·외국 배포는 `[외국법 — 외국 변호사·대리인 검증
> 필요]`. 조문 인용 시 법망 대조:
>
> ```
> mcp__beopmang__law get "저작권법 제46조"     # 저작물의 이용허락
> ```

## 전제: 실무 프로파일 로드

**의존성 스캔 전 `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md`를 읽는다.** 없거나
`[PLACEHOLDER]`가 남아 있으면 멈추고 `/ip-legal:cold-start-interview`. 실무 프로파일이 알려주는 것:

- 이 팀에서 누가 OSS 검토를 소유하나(흔히 엔지니어링 + 법무 sign-off)
- copyleft 의무의 에스컬레이션 라우팅
- prepend할 문서 헤더

업로드된 OSS 정책이 있으면 그것도 읽는다 — 어떤 라이선스를 팀이 허용·검토·금지하는지의 정본이다.

## 워크플로우

### Step 1: 스코프가 무엇인가

물어보거나 사용자가 준 것에서 추론한다:

> 무엇을 검토하나요?
>
> 1. **의존성 목록** — `package.json`, `requirements.txt`, `go.mod`, `Gemfile`, `Cargo.toml`,
>    `pom.xml`, SBOM(SPDX / CycloneDX), 락파일
> 2. **단일 라이브러리** — 추가를 고려 중인 특정 패키지 하나
> 3. **우리 코드** — 오픈소스로 공개하려는데 무엇이 임베디드되어 있는지 점검 필요

분석 경로가 다르다:

- 의존성 목록 → 모든 항목 분류, 의무 roll-up
- 단일 라이브러리 → 한 패키지 분류 + 가능하면 transitive 의존성 추적
- 배포 예정 코드 → 임베디드된 것(직접·transitive) 점검, 선택한 outbound 라이선스가 모든 임베디드
  라이선스와 양립하는지 점검, LICENSE/NOTICE 파일이 정확한지 점검

### Step 2: 배포 형태가 무엇인가

라이선스 목록 다음으로 가장 중요한 입력 — 같은 라이브러리도 소프트웨어가 어떻게 전달되느냐에 따라
다른 의무를 진다. 묻는다:

> 어떻게 배포되나요?
>
> 1. **SaaS / 호스팅 서비스** — 사용자가 네트워크로 접근; 사용자에게 아무것도 배포 안 함
> 2. **배포 바이너리** — 컴파일된 코드를 사용자에게 배포(데스크톱·모바일 앱, 온프레미스 서버, CLI)
> 3. **사내 전용** — 회사 내부에서만 사용, 외부 배포 없음
> 4. **임베디드 / 펌웨어** — 하드웨어에 탑재 또는 closed-system 펌웨어로 배포

| 배포 형태 | 실질적으로 중요한 라이선스 |
|---|---|
| SaaS | AGPL(네트워크 트리거), UI상 허용형 attribution, 경쟁 서비스로 재활용 시 SSPL/BUSL/Elastic |
| 배포 바이너리 | GPL, LGPL, MPL, EPL(모두 배포에 트리거), 허용형 attribution |
| 사내 전용 | 대부분 copyleft 미트리거 — 배포 없음. 허용형 attribution은 여전히 위생상 권장. AGPL은 회사 외부 사용자가 네트워크로 상호작용하면 여전히 트리거. |
| 임베디드 / 펌웨어 | GPL 준수가 특히 어렵다(소스 공개 + 재현 가능 빌드 + 경우에 따라 설치 정보). 배포 후가 아니라 배포 전 계획. |

배포 형태를 출력 메모에 플래그한다 — 같은 의존성 목록도 "SaaS" vs "배포 바이너리" 대조면 다른 의무를
낳는다.

### Step 3: 각 의존성 분류

모든 패키지에 대해 라이선스를 확정한다. 메타데이터가 아니라 **실제 라이선스 텍스트**를 읽는다 —
LICENSE 파일이 틀릴 수 있고(파일은 MIT인데 헤더는 GPL; README는 Apache라는데 라이선스 파일이 없음),
패키지 매니저 메타데이터는 stale일 수 있다.

분류:

| 버킷 | 예 | 핵심 의무 |
|---|---|---|
| **허용형(Permissive)** | MIT, BSD-2/3-Clause, Apache-2.0, ISC, Zlib, Unlicense | attribution, 라이선스 텍스트 보존, Apache-2.0은 특허 grant + NOTICE 요건 추가 |
| **약한 copyleft** | LGPL-2.1, LGPL-3.0, MPL-2.0, EPL-1.0/2.0, CDDL | 파일·라이브러리 단위 소스 공개; 링킹 규칙 다름 |
| **강한 copyleft** | GPL-2.0, GPL-3.0, AGPL-3.0, OSL, EUPL(버전에 따라) | 광범위 소스 공개; AGPL은 네트워크 사용까지 확장 |
| **퍼블릭도메인 / 헌정** | CC0, Unlicense, WTFPL | 보통 의무 없음, 단 퍼블릭도메인 헌정을 인정하지 않는 관할(한국 포함 — 저작권 포기·헌정 법리가 명확치 않음)에서는 다툼 `[검토]` |
| **비-OSI source-available** | SSPL, BUSL, Commons Clause, Elastic License, Confluent Community, fair-source family | 오픈소스 아님 — 상용·경쟁서비스 사용 제한. 개별 라이선스를 읽는다. |
| **기타 / 커스텀 / 불명** | 벤더 전용, 독점, 라이선스 파일 없음, 파일과 헤더의 라이선스 충돌 | 멈춤 — 디폴트로 허용형 취급하지 않음 |

플래그:

- **듀얼 라이선스 패키지** — 우리는 어느 라이선스로 쓰나? 선택이 의무를 바꿀 수 있다(예: Qt, MySQL —
  GPL vs 상용).
- **deprecated 패키지** — 더 이상 유지보수 안 됨; 지원되는 대체재가 있나?
- **자기 트리에 copyleft 의존성을 가진 패키지** — 최상위 라이선스는 허용형인데 transitive 의존성이
  copyleft.
- **최근 라이선스를 바꾼 패키지** — Redis, MongoDB, Elastic, HashiCorp — pin된 버전이 당신이
  생각하는 라이선스 하에 있는지 확인.

### Step 4: 의무를 배포 형태에 매핑

분류된 각 의존성에 대해 배포 형태가 무엇을 트리거하는지 명시:

```markdown
### [package@version] — [라이선스]

**분류:** [허용형 / 약한 copyleft / 강한 copyleft / 퍼블릭도메인 / 비-OSI / 불명]

**우리 배포([SaaS / 바이너리 / 사내 / 임베디드])에서의 의무:**

- [ ] [구체적 의무 — 예: "앱과 함께 배포되는 NOTICES 파일에 attribution 포함"]
- [ ] [예: "개변·배포 시 우리 개변분의 소스 공개"]
- [ ] [예: "AGPL 네트워크 트리거 — 사용자가 우리 개변 버전을 네트워크로 접근하면 그들에게 소스 제공"]

**법적 위험:** 🔴 차단 | 🟠 높음 | 🟡 중간 | 🟢 낮음
**사업·상용화 마찰:** 🔴 출시 중단 | 🟠 출시 지연·재설계 | 🟡 협상·비용 부담 | 🟢 비가시

**권고:** [의무 준수 | [대체재]로 교체 | 제거 | 배포 전 변호사 검토 | [벤더]에서 상용 라이선스 확보]
```

> **copyleft 의존성이 어떻게 소비되나?** 링킹 관계가 copyleft 실제 트리거 여부를 결정한다. 묻거나 확정:
> - **정적 링킹 / 함께 컴파일:** 작품이 하나의 바이너리로 결합. copyleft 트리거의 강한 신호(LGPL
>   "라이브러리에 기반한 작품", GPL 2차적저작물).
> - **동적 링킹 / 공유 라이브러리:** 런타임에 분리 가능. LGPL은 이를 명시적으로 허용("라이브러리를
>   사용하는 작품"). GPL의 입장은 다툼(FSF는 2차적저작물이라 보고 일부는 반대) `[검토]`.
> - **헤더 inclusion / inline 함수:** 포함량에 따라 2차적저작물을 만들 수 있다.
> - **서브프로세스 / IPC:** 잘 정의된 인터페이스로 통신하는 별도 프로세스. 일반적으로 2차적저작물 아님.
> - **네트워크 API 호출:** 대부분 라이선스에선 아님. **AGPL**은 네트워크-상호작용 조항 때문에 네트워크로
>   소프트웨어를 제공하는 것이 배포에 해당. 마이크로서비스 아키텍처에서 API 뒤의 AGPL 컴포넌트도 트리거.
> - **파일 단위 copyleft(MPL):** 개변된 파일만 copyleft, 전체가 아님. copyleft 파일이 개변됐는지 점검.
>
> **심각도 등급이 이것에 달려 있다.** 링킹 분석 없는 "LGPL — 약한 copyleft, 링킹 규칙 다름"은 엔지니어를
> 소송에 빠뜨리는 답이다. 독점 제품에 정적 링킹된 LGPL은 🔴 차단. 동적 링킹 LGPL은 🟢 낮음. 같은
> 라이선스, 반대 등급.

**심각도 calibration(이중 축 — CLAUDE.md `## 공유 가드레일` 정본).**

법적 위험:

| 등급 | 의미 |
|---|---|
| 🔴 차단 | 트리거하는 배포에 강한 copyleft(배포 바이너리의 GPL, SaaS의 AGPL). 사업 모델이 실제 충돌하는 비-OSI(managed service 구축 중 SSPL). 라이선스 확정 불가 + load-bearing 패키지. |
| 🟠 높음 | 팀이 준비 안 된 의무가 있는 약한 copyleft(파일 단위 공개, NOTICE 요건). 선택 라이선스가 모호한 듀얼. 파일과 헤더의 라이선스 불일치. |
| 🟡 중간 | 빌드에 안 엮인 attribution 요건이 있는 허용형(NOTICES 누락, 배포물에 LICENSE 누락). 소비 방식에 따라 트리거할 수도 안 할 수도 있는 transitive copyleft. |
| 🟢 낮음 | 의무가 이미 충족된 허용형. 트리거하지 않는 배포 형태의 copyleft(재배포 없이 사내 전용 GPL). |

사업·상용화 마찰: 🔴 출시 중단·딜 브레이크 / 🟠 출시 지연·재설계 / 🟡 협상·비용 부담 / 🟢 비가시.
**두 축을 모두 매긴다** — copyleft 소스 공개 의무는 법적으로 준수 가능해도 영업비밀·독점 코드 공개를
강제해 사업 마찰 🔴일 수 있다.

### Step 5: 실패 모드 플래그

다음 중 하나라도 있으면 메모 상단 섹션에 call out:

- **라이선스 불명** — 디폴트 허용형이 아니라 "검토 필요"로 분류. 미분류 의존성은 배포 결정을 멈춰야
  하지 통과시키면 안 된다.
- **라이선스 파일이 파일 헤더와 충돌** — 둘 다 읽고 충돌을 보고.
- **양립 불가 조합** — GPL-2.0 only + Apache-2.0은 역사적으로 알려진 비양립; MPL/EPL/GPL 조합을 주의 깊게 점검.
- **OSS인 척하는 비-OSI 라이선스** — SSPL, BUSL, Commons Clause, Elastic License, Confluent Community.
  라이선스를 읽는다; GitHub의 "open source" 배지에 의존하지 않는다.
- **라이선스 변경** — 이전 버전이 허용형이고 현재 버전이 source-available이면 pin이 중요하다.

### Step 6: Outbound 점검 (우리 코드를 오픈소스로 공개하기 전)

사용자가 코드를 오픈소스로 공개하려는 경우:

- 선택한 outbound 라이선스가 모든 임베디드 의존성 라이선스와 양립하는지 확인(예: GPL 코드를 임베디드
  했으면 MIT로 공개 불가 — 결합 작품은 GPL이어야 함)
- LICENSE 파일이 존재하고 정확한지 확인
- NOTICE 파일이 존재하고 필요한 attribution을 나열하는지 확인(Apache-2.0 등)
- 필요한 곳에 제3자 라이선스 텍스트가 번들되었는지 확인
- 레포 히스토리에 독점·비밀 코드, 고객 데이터, 임베디드 credential이 없는지 확인 **(영업비밀
  비밀관리성 — 부정경쟁방지법 §2. 공개는 일방통행 문이다)**
- 프로젝트명의 상표·브랜드 정책 확인(저작권 라이선스와 별개 — 상표법)

## 결정 자세

라이선스를 확신 있게 분류할 수 없으면 **"검토 필요"**로 플래그한다 — 허용형이라 부르지 않는다.
라이선스 위험을 과소-분류하는 것은 일방통행 문이다: 디폴트-허용형 가정으로 내린 배포 결정이 몇 달 뒤
소스 공개 의무·금지청구가 된다. 과다-플래그는 양방통행 문 — 변호사가 검토에서 목록을 좁힌다.

마찬가지로 copyleft 트리거 분석이 다툼 있는 문제(AGPL의 "네트워크 상호작용", GPL-3.0의 "conveying",
LGPL 링킹 범위)에 달려 있으면 `[검토]`로 플래그하고 양쪽으로 작용하는 요소를 surface한다.
