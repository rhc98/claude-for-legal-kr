# /portfolio

무엇이 언제 갱신·연차료 기한이 도래하는지, 그리고 이미 추가기간에 들어갔거나 소멸 위험인
권리가 무엇인지를 surface한다.

가드레일·헤더·검토자 메모·태그 어휘·결정 자세·산출물 포맷·대시보드 제안은 모두 이 플러그인의
`~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md`가 정본이다 — 여기서 재진술하지
않고 가리킨다. 충돌 시 그 CLAUDE.md가 우선한다. 이 스킬은 거의 전부 데이터·캘린더 작업이라
인용은 최소다 — 법 인용이 실제로 들어가는 곳(존속기간 조문, 갱신·연차료 기한)에만 출처·검증
태그를 붙인다.

## 무엇을 하나

1. **config 읽기.** `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md` 읽기 —
   알림 리드타임(`## IP 실무 프로파일` → 포트폴리오 등록부 → 갱신 알림 리드타임), 에스컬레이션,
   외부 대리인(출원·갱신 변리사)을 확보. 없거나 `[PLACEHOLDER]`가 남아 있으면 substantive 작업
   전 멈추고 `/ip-legal:cold-start-interview`로 안내. (단순 등록부 조회는 셋업 없이도 동작하나,
   알림 리드타임·목적지·에스컬레이션은 config에서 온다.)

2. **등록부 읽기.** 라이브 등록부는
   `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/portfolio.yaml`
   (config 디렉토리 — 플러그인 업데이트에도 살아남음). 없으면 첫 추가 시 빈 `assets:` 목록으로
   생성. 이 스킬의 `references/portfolio.yaml`은 TEMPLATE/예시일 뿐 라이브 데이터가 아니다.

3. **플래그에 따라 모드 라우팅:**
   - 플래그 없음 또는 `--report`: Mode 2 — 임박 기한·추가기간·소멸 항목 surface (디폴트)
   - `--init`: Mode 1 — IP 관리 시스템·스프레드시트·수동에서 등록부 초기화
   - `--add`: Mode 3 — 신규 권리 대화식 추가 (관할별 갱신 규칙 포함)
   - `--update`: Mode 4 — 납부·갱신 완료 기록 (자격 없는 실무자 게이트)
   - `--audit`: Mode 5 — 등록부 위생·등록현황 갭·소유관계·사용실태 헬스 감사

## 예시

```
/ip-legal:portfolio
```

```
/ip-legal:portfolio --report --days 180
```

```
/ip-legal:portfolio --audit
```

## 목적

아무도 권리 하나를 두 번 docketing하지 않는다. 존속기간 만료일·갱신/연차료 기한은 등록 시점에
한 번 산정되고, 그 뒤 어딘가에 산다 — 이상적으로는 기한 6개월·3개월·1개월 *전*에 당신에게
소리치는 곳에. *후*가 아니라. 한 번 놓친 연차료·갱신은 권리 자체를 잃는다 — 추가기간(할증)으로
구제되는 범위는 권리 유형마다 다르고, 그 창마저 닫히면 회복은 제한적이다.

이 스킬은 포트폴리오 등록부를 유지하고 다가오는 것을 surface한다.

> **docketing된 잘못된 기한은 docketing 안 된 기한보다 나쁘다 — 잘못된 안심을 주기 때문.**
> 이 스킬이 계산하는 기한은 모두 참조용이다. **존속기간·갱신기한·연차료(특허료)·요율은 특허청
> 고시로 변동되고, 권리 유형·기산점에 따라 적용이 다르다.** 의존 전 KIPRIS·특허로(patent.go.kr)·
> 특허청 고시로 대조한다. 등록부가 "납부 완료"로 표시해도 특허청이 수리했다는 보장이 아니다.

## 중요: 존속기간·기한 참조 주의

> 이 스킬은 존속기간(특허 출원일+20년·상표 등록일+10년·디자인 등록일+20년 등), 갱신등록신청 기간
> (상표는 **존속기간 만료 전 1년 이내** + 만료 후 추가기간 등), 연차등록료(특허료) 납부기한·
> 추가납부·회복 요건을 **기억으로 하드코딩하지 않는다.** 존속기간·기한·요율은 법령·특허청 고시
> 개정으로 바뀌고, 기산점(출원일 vs 등록일)·권리 유형에 따라 달라진다.
> **의존 전 현행 조문(특허법·상표법·디자인보호법)을 법망에서 대조하고, 기한·수수료는 특허청
> 고시·KIPRIS로 대조하라.** 출원·갱신을 변리사·특허법인에 위탁한 경우, 그 위탁처의 docketing
> 시스템이 해당 권리에 대해 권위 있는 1차 자료다 — 이 등록부는 그 데이터를 조직·surface하는
> 도구이지 대체물이 아니다. 모든 기한 인용은 `[특허청 고시 — 직접 검증 필요]` / `[KIPRIS — 직접
> 검증 필요]`로, 존속기간 조문은 `[검증 필요]`로 플래그한다.

법망 검증(연결 시 — 법망 미연결이면 인용은 `[모델 지식 — 검증 필요]`):

```
mcp__beopmang__law get "특허법 제88조"
mcp__beopmang__law get "상표법 제84조"
mcp__beopmang__tools verify {"citation": "디자인보호법 제91조"}
```

## 권리 유형·관할 가정

> 등록부는 각 권리의 **권리 유형**(특허/실용신안/상표/디자인/저작권)과 **관할**(KR 특허청 / 마드리드
> 지정국 / PCT 진입국 등)을 기준으로 기한을 계산한다. 존속기간·기산점·갱신 메커니즘이 권리 유형마다
> 크게 다르다 — 한 유형의 기한을 다른 유형에 복사하지 않는다.
>
> **유형별로 갈리는 지점(현행 조문·특허청 고시 대조 필요 — 단정 금지):**
>
> - **특허·실용신안:** 존속기간은 **출원일로부터** 기산(특허 20년 §88), 등록 후 **연차등록료(특허료)**를
>   매년·기간별로 납부하지 않으면 권리 소멸. 납부기한 도과 시 **추가납부기간(할증)**, 그마저 지나면
>   회복 요건이 제한적. 존속기간 연장(의약·농약 등) 별도. `[특허청 고시 — 직접 검증 필요]`
> - **상표:** 존속기간은 **설정등록일로부터 10년**, **갱신등록**으로 갱신(상표법 §83·§84). **갱신등록신청은
>   존속기간 만료 전 1년 이내**에 가능하고, 만료 후에도 일정 추가기간(할증) 내 신청 가능 — 기간·요율은
>   특허청 고시. 갱신은 횟수 제한 없이 반복. 불사용취소(§119) 리스크는 별도 — 등록만이 아니라 지정상품
>   사용실태도 본다. `[검증 필요]` / `[특허청 고시 — 직접 검증 필요]`
> - **디자인:** 존속기간은 **설정등록일로부터 20년**(디자인보호법 §91), 연차등록료 납부 구조. 디자인일부심사
>   등록(물품류 구분)·관련디자인·비밀디자인 제도가 있다. `[특허청 고시 — 직접 검증 필요]`
> - **저작권:** **등록이 효력요건이 아니다**(원칙 무방식주의 — 저작권법 §10). 한국저작권위원회 등록은
>   추정·대항요건일 뿐이고 보호기간(원칙 사후 70년·업무상저작물 공표 후 70년)은 등록과 무관하게 진행 —
>   "갱신 기한"이라는 개념이 특허·상표와 다르다. 등록부에 넣더라도 갱신 docketing이 아니라 등록 상태·
>   보호기간 만료의 기록으로 다룬다. `[검토]` `[저작권위 등록 — 직접 검증 필요]`
> - **영업비밀:** **존속기간·갱신 기한이라는 개념 자체가 없다** — 비밀관리성이 유지되는 한 보호되고,
>   관리가 끊기면 권리가 소멸한다. 이 등록부의 기한 모델에 맞지 않는다. 영업비밀은
>   `/ip-legal:trade-secret-protection`으로 관리하고, 여기에는 (넣더라도) KIPI 원본증명 갱신 등
>   별도 일정만 메모로 둔다.
>
> **외국 권리**(마드리드 지정국·PCT 진입국·외국 직접출원)는 관할별 존속기간·갱신·연차료 규칙이 한국과
> 다르다. 현지 대리인·외국 변리사가 관리하면 `agent_managed: true`로 표시하고, 등록부는 독립적으로
> 기한을 계산하기보다 그 대리인에게 후속 확인할 시점을 플래그한다.
> **외국 권리는 `[외국법 — 외국 변호사·대리인 검증 필요]`.** 한국 존속기간 doctrine을 외국 권리에
> 침묵으로 적용하지 않는다.

## 등록부 파일

라이브 등록부는 `~/.claude/plugins/config/claude-for-legal-kr/ip-legal/portfolio.yaml`
(config 디렉토리 — 플러그인 업데이트에도 살아남음). 키는 machine-stable 영문, 사람이 보는 값·
코멘트는 한국어. 각 레코드:

```yaml
- asset_id: "TM-001"                          # 내부 식별자
  right_type: "상표"                           # 특허 / 실용신안 / 상표 / 디자인 / 저작권
  title: "[브랜드/표장명]"                       # 상표·상호 / 발명의 명칭 / 디자인 물품명
  jurisdiction: "KR"                           # KR(특허청) / 마드리드 지정국 / PCT 진입국 — 외국은 현지 대리인
  application_no: "40-2023-0001234"
  registration_no: "40-2001234"               # null이면 출원 계류
  application_date: 2023-03-15                 # 특허·실용신안 존속기간 기산점
  registration_date: 2024-09-01               # 상표·디자인 존속기간 기산점
  term_end: 2034-09-01                         # 존속기간 만료일 [특허청 고시 — 직접 검증 필요]
  next_action: "갱신등록신청"                    # 갱신등록신청 / 연차등록료(특허료) 납부 / 분할납부 / 회복
  next_action_due: 2033-09-01                  # 갱신·연차료 기한
  next_action_due_provenance: "[특허청 고시 — 직접 검증 필요]"   # 계산 기한엔 항상 검증 플래그
  grace_end: 2034-03-01                        # 추가납부·추가기간 종료일 (있으면) [검증 필요]
  status: "registered"                         # pending | registered | grace | lapsed | abandoned
  owner: "(주)[회사명]"                          # 권리자
  business_owner: "브랜드팀 / 지현"              # 사업 담당자 (핑 대상)
  agent_firm: "[특허법인]"                       # 출원·갱신 대리 변리사 / 외국은 현지 대리인
  docket_id: "FIRM-TM-555"                      # 대리인 사건번호 / IP 관리 시스템 ID
  agent_managed: false                         # 대리인이 갱신 기한을 관리하면 true
  kipris_verified: 2026-06-01                  # KIPRIS 등록현황 최종 대조일 [KIPRIS — 직접 검증 필요]
  next_review: 2033-03-01                       # 등록부 다음 점검일
  notes: "주력 브랜드 — 갱신 필수. 지정상품 사용실태 점검(불사용취소 리스크)."
```

**계산 기한은 모두 검증 플래그를 동반한다.** `next_action_due`·`term_end`·`grace_end`는 산정값일
뿐이다. 검증 플래그(`next_action_due_provenance`)를 날짜와 함께 붙여, 검증 플래그가 주변 산문이
아니라 날짜를 따라다니게 한다. 검토자가 KIPRIS·특허청 고시로 대조한 뒤에야 의존한다.

**기산점을 권리 유형으로 가른다.** 특허·실용신안은 `application_date`(출원일)에서, 상표·디자인은
`registration_date`(설정등록일)에서 존속기간을 계산한다. 둘을 섞으면 만료일이 통째로 틀린다 —
가장 흔하고 결과가 큰 오류다.

**상태 값:**
- `pending` — 출원 계류 (등록 전; 갱신·연차료 기한 없음, 단 출원 단계 기한은 별도 docketing)
- `registered` — 등록 유지 중
- `grace` — 납부·갱신 기한 도과, 추가납부기간·추가기간(할증) 내 — **소멸 임박, 오늘 움직여야 함**
- `lapsed` — 기한·추가기간 도과로 소멸 (회복 요건 별도 검토)
- `abandoned` — 의도적 포기 (유지 결정 후 비유지)

## 모든 기한에 영업일·기산 점검

**등록부의 기한은 유효하게 납부·신청할 수 있는 마지막 날이어야 한다.** 기한 계산 시:

1. **존속기간 만료일 계산.** 권리 유형의 기산점에서 산정(특허 출원일+20년, 상표 등록일+10년,
   디자인 등록일+20년). `[특허청 고시 — 직접 검증 필요]`.
2. **갱신·연차료 기한 산정.** 상표 갱신등록신청은 존속기간 만료 전 1년 이내 + 만료 후 추가기간;
   특허·디자인 연차료는 기간별 납부기한 + 추가납부기간(할증). 기간·요율은 특허청 고시 — 단정하지
   않고 플래그.
3. **공휴일·말일 점검.** 기한이 토·일·공휴일이면 한국 민법·특허법 시행 규칙상 기간 계산 규칙을
   확인(특허청 절차상 기간 만료일이 공휴일이면 다음 영업일로 연장되는 취급 등 — 계약 갱신과 달리
   행정 절차는 forward 연장이 일반적이나, 규칙·고시 대조 필요). `[특허청 고시 — 직접 검증 필요]`
4. **계산 기한엔 항상 검증 플래그.** `next_action_due_provenance`에 `[특허청 고시 — 직접 검증 필요]`.
5. **알림은 리드타임 기준으로 발사.** config의 갱신 알림 리드타임(디폴트 만료/기한 6개월·3개월·
   1개월 전)에 따라 Mode 2의 긴급도 밴드를 산정한다.

## Mode 1: 초기화

등록부가 없을 때, 또는 `--init`으로 처음부터 재생성할 때.

### Step 1: 소스 확보

`~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md` → `## IP 실무 프로파일` → IP 관리
시스템(Anaqua·CPA Global·PatSnap·특허법인 시스템) 연결 여부를 확인한다.

- **IP 관리 시스템이 연결된 경우:** 시스템에서 활성 권리·기한 필드를 조회해 일괄 적재. 시스템이
  기록의 권위 자료(system of record)다 — 충돌 시 시스템이 이긴다.
- **스프레드시트·docketing 내보내기가 있는 경우:** 사용자에게 업로드를 요청해 ingest.
- **둘 다 없는 경우:** 수동 입력(Mode 3 플로우). 권리를 하나씩 받는다.

### Step 2: 권리별 존속기간·기한 확정

각 권리에 대해 권리 유형·관할·기산점에서 존속기간·갱신/연차료 기한을 산정하되, **캐시된 일정에
의존하지 않는다** — 존속기간·요율·추가기간은 특허청 고시로 변동된다.

1. 사용자에게 **출원·갱신 대리 변리사·특허법인의 docketing 시스템 또는 KIPRIS 등록현황**이 있는지
   묻는다 — 가장 권위 있는 1차 자료.
2. 없으면 사용자가 아는 것을 받는다(권리 유형, 출원일·등록일, 등록번호, 차기 기한, 대리인).
3. 사용자가 모르는 기한은 해당 권리를 `pending`/`unknown`으로 두고 플래그 — 캐시 참조에서 채우지
   않는다. 다음 단계는 KIPRIS·대리인 docketing·특허청 고시 대조.

### Step 3: 등록부 쓰기

`~/.claude/plugins/config/claude-for-legal-kr/ip-legal/portfolio.yaml`을 모든 권리와 함께 생성.
초기 상태:
- 등록 유지·기한 도래 없음이면 `registered`
- 리드타임 창 내 기한 임박이면 `registered`(Mode 2에서 임박으로 surface)
- 기한 도과·추가기간 내면 `grace`
- 추가기간 도과면 `lapsed`
- 출원 계류면 `pending`

생성 후 요약:

```
IP 포트폴리오 등록부 초기화 완료.

권리: [N]건 (특허 [N] · 상표 [N] · 디자인 [N] · 저작권 [N])
관할: KR [N] · 외국 [N]

상태 요약:
  ✅ 등록 유지:   [N]
  ⏰ 기한 임박:   [N] (리드타임 창 내)
  🔴 추가기간/소멸: [N]  (오늘 움직여야 함)
  ❓ 미확인:      [N] (KIPRIS·대리인 docketing·특허청 고시 확인 필요)

/ip-legal:portfolio --report 로 도래분을 확인하세요.
```

## Mode 3: 자산 추가 (`--add`)

신규 권리를 대화식으로 추가. IP 검토·출원 핸드오프 또는 수동 입력. 한 번에 한 필드씩 묻거나 paste
수용. 필수 필드: `right_type`, `title`, `jurisdiction`, `application_date` 또는 `registration_date`,
`status`, `business_owner`. `term_end`·`next_action_due`는 위 영업일·기산 점검을 거쳐 계산하고
검증 플래그를 단다.

**비표준 관할 규칙.** 외국 권리·드문 권리 유형을 추가하면서 그 관할·유형의 존속기간·갱신 규칙이
참조에 없으면, 참조 표에서 채우지 말고 캡처한다:

> [관할/유형]의 존속기간·갱신 규칙이 참조에 없습니다. 추적을 위해 캡처하겠습니다.
> 1. 존속기간은 무엇을 기산점으로 몇 년인가요?
> 2. 갱신·연차료는 어떤 주기·기한인가요? 추가기간(할증)이 있나요?
> 3. 현지에 이 권리를 관리하는 대리인이 있나요? (있으면 `agent_managed: true`)

답을 등록부 레코드의 `notes`·관할 메모에 저장하고, 외국 권리는 `[외국법 — 외국 변호사·대리인
검증 필요]`로 플래그.

같은 권리(같은 등록번호)가 이미 있으면, 교체(갱신된 정보)인지 별개 권리인지 묻는다.

## Mode 4: 업데이트 (`--update`)

납부·갱신 완료, 상태 변경을 기록.

### 게이트: 출원·갱신·연차료 납부 지시

**갱신등록신청·연차료 납부·권리 포기를 지시·확정하기 전:**
`~/.claude/plugins/config/claude-for-legal-kr/ip-legal/CLAUDE.md` → `## 누가 이 플러그인을 사용하나`
읽기. 역할이 **변호사·변리사 아님**이면:

> 갱신등록신청·연차등록료(특허료) 납부·권리 포기는 법적 결과를 낳습니다 — 기한 도과·미납은 권리
> 소멸로 이어지고, 포기는 되돌릴 수 없습니다. 변리사·특허법인과 검토하셨나요? 예라면 기록하겠습니다.
> 아니라면 가져갈 브리프:
>
> - 권리·유형·관할·차기 기한(존속기간 만료일, 갱신/연차료 기한, 추가기간 종료일)
> - 등록부가 보여주는 직전 납부·갱신 이력
> - 미해결 질문(권리자·지정상품·청구범위가 여전히 정확한가, 유지할 가치가 있는 권리인가, 추가기간
>   할증이 적용되는가)
> - 무엇이 잘못될 수 있나(기한 도과 → 권리 소멸, 불사용 상표 갱신의 무용성, 단종 제품 특허 연차료 낭비)
> - 변리사에게 물을 것(올해 실제로 내야 할 연차료·갱신은 무엇인가, 포기해도 되는 권리가 있나, 누가 지시·서명하나)
>
> 변리사·특허법인을 찾아야 하면: 대한변리사회 또는 사내 자문 특허법인에 연결.

이 게이트를 명시적 "예" 없이 넘어 새 납부·갱신 이력을 기록하지 않는다. 등록부 읽기·기한 리포트·
"무엇이 도래" 출력은 게이트 불필요.

납부·갱신·포기를 행동의 산출물로 다루는 것은 권리 귀속·유지 비용·소멸 효과가 걸린 판단이다 —
유지 여부 자체가 사업·법적 결정이므로 단정 말고 `[검토]`.

### 업데이트 기록

변리사·실무자가 무엇을 했는지 말한다:
> "[권리]의 연차등록료를 [날짜]에 납부. 차기 기한 [날짜]."
> "[상표]의 갱신등록을 [날짜]에 완료. 차기 존속기간 만료 [날짜]."
> "[권리]는 유지하지 않기로 — 포기. 차기 연차료 미납 예정."

Claude 갱신:
- `next_action`·`next_action_due` → 차기 기한으로 굴림
- `term_end` → 갱신 시 다음 만료일로 (상표 갱신 = +10년)
- `status` → `registered` / `lapsed` / `abandoned`
- `kipris_verified` → KIPRIS 대조했으면 일자
- 비유지 결정은 `notes`에 사유 기록

## Mode 5: 헬스 감사 (`--audit`)

단순 기한 상태를 넘는 광범위 검토. surface:

**기한 위생:**
- 추가기간(`grace`) 항목 — 소멸 임박
- 미확인·`unknown` 상태 항목 — KIPRIS·대리인 docketing 확인 필요
- KIPRIS 대조 stale (`kipris_verified` 12개월 초과 또는 없음) — 등록현황·소유관계 미확인

**등록·소유관계 갭:**
- 권리자(`owner`)가 현 법인명·합병 후 명칭과 불일치 가능 — **명의변경(이전등록) 누락**은 권리행사·
  대항에 문제. `[검토]` `[KIPRIS — 직접 검증 필요]`
- 출원 계류(`pending`)가 장기 — 거절·취하·심사 지연 여부 KIPRIS 확인.

**사용실태·유지 가치:**
- **상표 불사용 리스크** — 등록 상표 중 지정상품에 실제 사용되지 않는 것은 **불사용취소심판(상표법
  §119) 노출**. 갱신 비용을 들이기 전 사용실태 점검. `[검토]`
- **유지 가치 낮은 권리** — 단종 제품 라인의 특허, 사용 안 하는 상표 — 차기 연차료·갱신 전 유지
  여부 결정. 비유지가 합리적일 수 있다(불필요한 연차료 절감).

**존속기간 지평:**
- 6개월·12개월 내 갱신/연차료 도래 권리 카운트 — 예산·대리인 지시 일정.
- 회복 불가능하게 소멸 임박(추가기간 종료 근접) — 최우선.

출력 포맷:

```
IP 포트폴리오 헬스 감사 — [날짜]

기한 위생
  추가기간(소멸 임박): [N]  (오늘 움직여야 함)
  미확인: [N]
  KIPRIS 대조 stale (12개월 초과): [N]

등록·소유관계
  명의변경 의심(권리자 불일치): [N]  [검토] [KIPRIS — 직접 검증 필요]
  장기 출원 계류: [N]

사용실태·유지 가치
  상표 불사용 리스크(갱신 전 점검): [N]  [검토]
  유지 가치 재검토 후보: [N]

존속기간 지평
  6개월 내 도래: [N]
  12개월 내 도래: [N]
  추가기간 종료 임박: [N]  (최우선)

권장 조치
  1. [최우선 — 통상 추가기간 소멸 임박 또는 불사용 상표 갱신 회피]
  2. [...]
```

트래커가 ~10건을 초과하거나 사용자가 요청하면 대시보드 제안(CLAUDE.md `## 산출물`).

## 이 스킬이 하지 않는 것

- 아무것도 출원·갱신·납부하지 않는다. 산출은 등록부와 할 일 목록이며, 출원·갱신·납부는 변리사·
  특허법인이 한다.
- 라이브 등록현황을 특허청에서 확인하지 않는다 — KIPRIS·대리인 docketing이 1차 자료. 마지막
  대조 시점을 추적할 뿐이다.
- 권리 유지·포기를 결정하지 않는다. 기한·유지 비용·사업 담당자를 surface한다 — 결정은 사업·법적 판단.
- 존속기간·갱신기한·연차료·추가기간·요율을 기억으로 단정하지 않는다. 모두 **현행 조문(특허법·
  상표법·디자인보호법) + 특허청 고시 + KIPRIS** 대조 — `[검증 필요]` / `[특허청 고시 — 직접 검증
  필요]` / `[KIPRIS — 직접 검증 필요]`.
