# /customize

## 실행 조건

사용자가 `/regulatory-legal:customize` 입력. 콜드스타트 인터뷰 전체 재실행 없이, YAML 수동
편집 없이 규제 대응 프로파일의 무엇 — 규제기관 watchlist·재료성 임계값·정책 라이브러리·갭
대응 절차·소스 구성 — 을 변경하고 싶음.

## 할 일

1. **config 읽기.** `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`
   (와 한 단계 위 `~/.claude/plugins/config/claude-for-legal-kr/company-profile.md`) 읽기.
   플러그인 config 없거나 `[PLACEHOLDER]` 값 남아 있으면:

   > 셋업을 아직 안 하셨습니다. 먼저 `/regulatory-legal:cold-start-interview` 실행 —
   > customize는 이미 가진 프로파일을 조정하기 위한 것.

2. **맞춤화 가능 맵 보여주기.** 프로파일에 있는 것을 그룹화, 현재 값의 한 줄 요약과 함께:

   - **회사 / 우리는 누구인가** — 회사명·업종·관할·위험 성향·실무 환경
     *(전 플러그인 공유 — `company-profile.md` 통해 변경 전파)*
   - **규제기관 watchlist** — watch하는 소관 부처·위원회, 각각의 watch 이유, 피드 소스
   - **정책 라이브러리** — 색인된 정책명·파일 위치·최종 갱신·owner
   - **중요도(재료성) 임계값** — "우리 분야" 정의(규제기관 의결의 분야 매칭 기준), tier별
     캘리브레이션 — `/reg-feed-watcher` 필터링에 영향
   - **갭 대응 절차** — 트리아지 담당, 정책 업데이트 owner, 갭 추적 방법, material 갭
     에스컬레이션 대상
   - **소스·피드 구성** — 공포 법령·입법예고/행정예고·계류 법률안·규제기관 직접 피드,
     점검 주기, 다이제스트 출력 경로
   - **의견제출 결정 owner** — 입법예고·행정예고에 의견을 제출할지 누가 결정하는지
     *(`/comments`의 consequential-action 게이트에 영향)*
   - **워크플로우** — 매터 워크스페이스 on/off·경로
   - **통합** — 국가법령정보 MCP·문서 저장소·Slack 상태, 폴백

3. **무엇을 바꿀지 묻기.**

   > 무엇을 조정할까요? 섹션을 고르거나 자기 말로 변경 설명.

4. **변경 수집.** 사용자가 섹션을 고르면 그 섹션의 현재 값을 보여주고 새 값 묻기. 사용자가
   자유 형식으로 설명하면 어느 섹션이 영향인지 식별하고 mapping 확인.

5. **변경 전 검증.** 변경에 법적 인용·조문·기한이 있으면 국가법령정보 대조 검증:
   ```
   mcp__korean-law__get_law_text "[조항]"
   mcp__korean-law__legal_analysis {"mode": "verify_citations", "citation": "[조항]"}
   ```
   인용이 안 맞으면 사용자에게 확인 ("당신은 X라 하셨는데, 국가법령정보의 현행 텍스트는 Y. 어느
   것으로 갈까요?"). 계류 법률안·규제기관 의결례처럼 국가법령정보 커버 밖인 사실 주장은 검증
   불가를 명시하고 `[국회 의안시스템 — 직접 검증 필요]` / `[규제 당국 결정 — 직접 검증 필요]`
   태그를 붙인 채 반영.

6. **변경 적용.** CLAUDE.md의 해당 섹션을 갱신. 변경 전후 diff 보여주기.

7. **확인 + 검증 로그 기록.**
   ```
   [YYYY-MM-DD] customize - [성명]이(가) [섹션]에서 [구체적 변경]
   ```
   `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/verification-log.md`에 추가.

## 흔한 맞춤화 패턴

**규제기관 watchlist에 항목 추가:** "이 부처도 추가" → `/reg-feed-watcher`가 다음 실행부터
이 기관을 sweep. 피드 소스가 `references/source-catalog.md`에 있는지 확인 — 없으면 사용자에게
소관 사이트·RSS 유무를 묻고 `(verify)` 표기로 추가.

**재료성 임계값 튜닝:** "우리 분야" 정의를 좁히거나 넓히기(예: "유사 사업자 범위에 B사도
포함") → 영향: `/reg-feed-watcher`의 규제기관 의결·시정조치 분류(분야 매칭 기준)가 바뀜.
**계류 법률안을 material로 분류해달라는 요청은 거부하고 설명한다** — 아래 가드레일 참조.

**에스컬레이션·갭 owner 변경:** 옛 담당자에서 새 담당자로 — 단순 이름 교체. 갭 대응 절차
표 전체 보여주기.

**의견제출 결정 owner 변경:** "의견제출 owner를 X로" → 영향: `/comments`가 새 신규 입법예고·
행정예고 항목에 이 owner를 배정, Slack 알림 대상도 변경.

**정책 라이브러리 항목 추가·갱신:** "정책 Y를 라이브러리에 추가" → `/policy-diff`가 다음
diff에서 이 정책을 포함. `/policy-redraft`가 이 정책에 대한 레드라인을 산출 가능.

**다이제스트 출력 경로 변경:** 새 저장 경로로 → policy-monitor류 sweep이 없는 이 플러그인에서는
`/reg-feed-watcher`의 파일 출력 경로가 바뀜을 명시.

**국가법령정보 MCP 추가·제거:** `.mcp.json` 직접 편집 안내 또는 자동화.

## 가드레일

- **계류 법률안 = material 조정 절대 불가.** 사용자가 "계류 법안도 즉시 알림으로", "심사 중인
  법안을 material로 분류해달라"고 요청하면 **거부하고 설명한다**: "계류 법률안은 아직 법이
  아니라 컴플라이언스 의무를 만들지 않습니다 — 이 플러그인의 정본(CLAUDE.md `## 중요도(재료성)
  임계값`)이 이걸 절대 material로 분류하지 않도록 고정합니다. 대신 알림 강도를 높이고 싶으면
  '이 법안을 관심 워치리스트 상단에 고정'하거나 `/regulatory-legal:bill-watch`의 심사단계
  알림 빈도를 조정할 수 있습니다 — watch 유지, material 승격은 아닙니다."
- **섹션 삭제 금지.** 사용자가 규제기관·정책을 "빼고 싶다"면, `[Monitor only]`로 표시하고
  monitor는 피드를 아카이브에 유지하되 활성 다이제스트에서 제외함을 설명.
- **내부 불일치 플래그.** 변경이 프로파일을 불일치하게 만들면(예: watchlist에 기관 추가 +
  해당 기관의 피드 소스 없음; "매일 점검" + 재료성 임계값이 분기당 한 건도 안 나오게 됨),
  긴장을 플래그.
- **가드레일 저하 플래그.** 인용된 규정의 `[검증 필요]` 태그, 피드 항목의 출처 attribution,
  갭 트리아지의 `[검토]` 플래그는 load-bearing — 제거하지 않는다. 재료성 임계값은 조정
  가능하나, 다이제스트가 소음이 되는 지점 아래로 낮추는 것이 요점이면 그 방향임을 경고.
- **한 번에 한 변경.** 전체 인터뷰를 재질문하지 않는다.
