   있는 만큼 읽기.
4. 추출: watchlist 근거, 정책 라이브러리 색인, 소스·피드 구성, `references/source-catalog.md`

5. 마이그레이션: populate된 CLAUDE.md(placeholder 없음)가
   `~/.claude/plugins/cache/claude-for-legal-kr/regulatory-legal/*/CLAUDE.md`에는 있는데
   config 경로에 없다면, config 경로로 복사하고 무엇이 마이그레이션되었는지 사용자에게 보여줌.
6. `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md` 작성(필요시 상위
   디렉토리 생성). 요약 보여주기. 첫 작업 제안.

## `--check-integrations`

통합 가용성 점검(**국가법령정보 MCP**·문서 저장소·Slack·스케줄드 태스크)을 재실행하고
`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`의
`## 연결 가능한 통합` 섹션을 갱신. 재인터뷰는 하지 않음. MCP를 연결·해제한 뒤 플러그인이
인식하길 원하지만 전체 셋업을 다시 돌리고 싶지 않을 때 사용.

점검 시: MCP 도구 호출이 실제로 성공한 경우에만 ✓ 보고. 구성만 되어 있고 미테스트
커넥터는 ⚪로 표시하고 확인 방법 한 줄 첨부. `.mcp.json` 선언만으로 ✓ 보고하지 않음 —
사용자에게 "연결됨"으로 잘못 알려 trust를 깬다.

**국가법령정보(korean-law) MCP는 이 플러그인에서 1순위 통합이다.** 미연결이면:
- 공포 법령·판례·행정규칙·의결례 인용이 모두 `[모델 지식 — 검증 필요]`로 떨어짐
- 환각방지 검증(`verify_citations`)이 동작하지 않음
- 공포일/시행일 대조(현행·시행예정 조회)가 안 됨
사용자에게 다음 안내: "국가법령정보 MCP를 등록하세요: `claude mcp add korean-law "https://mcp.gomdori.app/law?oc=<OC키>" --transport http`.
무료 — open.law.go.kr에서 OC 인증키 발급 필요(1분). 자세한 내용은 `docs/LAW_MCP_INTEGRATION.md` 참조."

**단, 국가법령정보가 연결되어도 입법예고·행정예고·계류 법률안·규제기관 의결례 최신·전문은
커넥터 범위 밖이거나 부분 커버다.** 이건 통합 문제가 아니라 구조적 데이터 갭이다 —
`## 데이터 갭`(CLAUDE.md)에 늘 남는다.

```
/regulatory-legal:cold-start-interview
```

```
/regulatory-legal:cold-start-interview --check-integrations
```

# 콜드스타트 인터뷰: 규제 대응 실무

## 목적

*이* 규제 대응팀이 실제로 무엇을 watch하는지, "재료성"이 여기서 무엇을 뜻하는지, 정책이
어디 있는지, 계류 법률안·입법예고를 어떻게 다루는지 학습 —
`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`에 써서 다른 모든
스킬이 같은 이해에서 읽도록.

모든 규제기관은 끊임없이 무언가를 발표한다. 대부분은 당신에게 중요하지 않다. 이 인터뷰는
어느 규제기관을 watch할지, 그리고 결정적으로 "material하다"는 것이 여기서 무엇을 뜻하는지
학습한다 — 그래야 모니터가 소음이 아닌 신호를 낸다. **한국 규제 실무의 핵심 변수는 "우리가
어느 소관 부처·위원회 × 어느 법령 단계(공포 법령/입법예고·행정예고/국회 계류 법률안)를
watch하는가"다.** 미국 원본의 "Federal Register API 한 방"은 한국에 존재하지 않는다 — 공포
법령·예고안·계류 법안·기관 의결이 서로 다른 소스에 흩어져 있고, 발생하는 법적 효과도 다르다.
**가장 중요한 KR 규율: 국회에 계류 중인 법률안은 아직 법이 아니다 — 절대 material로 분류하지
않는다.** 이 인터뷰는 이 규율을 사용자에게도 명확히 전달한다.

## 콜드스타트 점검

`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md` 읽기:
- **존재하지 않음** → 인터뷰 시작.
- **`<!-- SETUP PAUSED AT: -->` 포함** → 사용자를 맞이하고 해당 섹션부터 재개 제안.
- **`[PLACEHOLDER]` 마커가 있는데 pause comment는 없음** → 템플릿이 완료되지
  않았음; 새로 시작할지 placeholder부터 재개할지 제안.
- **populate됨 (placeholder도 pause comment도 없음)** → 이미 구성됨; `--redo`가
  아니면 skip.

템플릿 구조는 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`에 있다 — 섹션 골격으로 사용.
완성된 실무 프로파일을 config 경로에 쓰고 필요시 상위 디렉토리 생성.

옛 캐시 경로 `~/.claude/plugins/cache/claude-for-legal-kr/regulatory-legal/*/CLAUDE.md`에
CLAUDE.md가 있는데 config 경로에는 없다면, 앞으로 복사.

## 공유 회사 프로필 확인

`~/.claude/plugins/config/claude-for-legal-kr/company-profile.md` 확인.

- **있으면:** 읽기. 한 줄 확인 보여주기: "당신은 [성명], [실무 환경], [회사], [업종],
  [관할]에서 운영. 맞나요? (또는 '업데이트'를 말해 공유 프로필 변경.)" 확인되면
  회사 질문은 skip — 바로 플러그인 특화 질문으로.
- **없으면:** 당신이 이 사용자가 셋업한 첫 플러그인. 오리엔테이션과 fork 후, 회사
  질문을 묻고 공유 프로필에 쓴다. 그 후 플러그인 특화 질문 계속. 사용자에게: "회사
  프로필 저장 완료 — 다른 법무 플러그인들이 이걸 읽고 이 질문들을 skip합니다."

공유 프로필에 속하는 회사 질문(있으면 재질문 금지): 실무 환경, 회사명, 업종,
무엇을 파는지, 규모, 관할, 위험 성향, 에스컬레이션 담당자. 플러그인 특화 질문
(watchlist, 재료성 임계값, 정책 라이브러리, 소스 구성 등)은 플러그인별로 유지.

## 설치 스코프 점검

오리엔테이션 전, 작업 디렉토리가 프로젝트 내부(사용자 홈이 아닌)임을 알아채면 한 번
플래그:

> **참고 — 이 플러그인이 프로젝트 스코프로 설치된 것 같습니다. 그러면 저는 [현재
> 디렉토리] 안의 파일만 읽을 수 있습니다. 다른 곳(다운로드·문서·드롭박스)의 정책 문서를
> 읽게 하려면 user 스코프로 설치하세요. 프로젝트 스코프로 계속해도 되지만, 파일을 이
> 폴더로 이동해야 합니다.**

진행 전 사용자 확인 요청: 프로젝트 스코프로 계속, 또는 잠시 멈추고 user 스코프로
재설치. 작업 디렉토리가 사용자 홈이면 이 점검을 침묵으로 skip.

## 인터뷰 시작 전

다른 어떤 것을 묻기 전, fork-first 프리앰블 — 3-4줄, 더 길게 하지 않음:

> **`regulatory-legal`은 규제 동향을 추적하고, 정책 갭을 평가하고, 컴플라이언스 의무를
> 관리하는 사람들을 위한 플러그인입니다.** 본인 영역이 아니신가요? 다른 플러그인을
> 확인하세요.
>
> **2분**이면 역할, watch 기관, 주요 관할이 설정되고 다른 모든 것은 sensible 디폴트로
> 채워집니다. **15분**은 재료성 임계값 캘리브레이션, 정책 라이브러리 인덱스, 소스·피드
> 구성, 갭 대응 절차를 추가합니다.
>
> 빠른 셋업 or 풀? (언제든 `/regulatory-legal:cold-start-interview --full`로 업그레이드.)

사용자 선택을 기다린 뒤 다른 어떤 것도 보여주지 않음.

## 사용자가 빠른/풀을 선택한 후

선택되었으면, 첫 인터뷰 질문 전 오리엔테이션:

> "이 플러그인은 실무 프로파일(규제기관 watchlist, 재료성 임계값, 정책 라이브러리, 소스·피드
> 구성), 갭 트래커, 의견제출 캘린더를 유지합니다. 이 셋업 인터뷰는 실제 일하는 방식 — 어느
> 규제기관이 실제로 중요한지, '중요하다'는 것이 여기서 무엇을 뜻하는지, 정책이 어디 있는지
> — 를 학습하고 플러그인이 매번 참조할 평문 파일에 씁니다. 답한 모든 것은 나중에 바꿀 수
> 있습니다. 끝나면 플러그인 커맨드는 일반 템플릿이 아닌 당신이 일하는 방식대로 동작합니다."
>
> 그 뒤: "셋업은 당신 답변에서 fresh한 전문가 프로파일을 만듭니다. 개인 Claude 기록,
> 다른 대화, 홈 디렉토리 CLAUDE.md를 읽지 않습니다. 대화 컨텍스트에서 관련 정보를
> 발견하면 — 예: 앞에서 규제기관을 언급하신 경우 — 사용 전 묻겠습니다. 타이핑하거나
> 승인하지 않은 정보는 실무 구성에 들어가지 않습니다."
>
> 그 뒤: "준비됐나요? 빠른 질문 몇 개 먼저, 그 뒤 더 깊이."

**왜 중요한가.** 이 플러그인의 모든 다이제스트·diff·갭 리포트는 이 인터뷰가 쓴 구성을
읽습니다. 일반 구성은 일반 출력을 줍니다 — 디폴트 watchlist, 디폴트 재료성 임계값, 모든
기관 보도자료를 시정조치처럼 다루는 다이제스트. 플러그인에게 실제 watch 기관, 실제
"material"의 의미, 실제 정책 라이브러리 구조를 말하는 것이 "규제 AI 도구"와 "신호를
보내는 도구"의 차이를 만듭니다.

실무 프로파일은 사용자의 타이핑 답변과 시드 문서에서만 채운다. `~/CLAUDE.md`를 읽거나
ambient 컨텍스트에서 실무 사실을 끌어오지 않는다. 대화에서 이미 보이는 관련 정보가
있으면 사용 전 묻는다.

**빠른 셋업 경로:** Part 0(역할·통합)과 규제기관 watchlist만 묻고, 나머지는
`[DEFAULT]` 마커로 config 작성 — 재료성 임계값은 CLAUDE.md 정본 표(공포 법령 항상
material / 입법예고·행정예고 review-worthy / 계류 법률안 watch 고정 / 규제기관 의결
분야 매칭 / 보도자료 FYI)를 그대로 쓴다. 마무리: "완료. 이제 커맨드를 쓸 수 있습니다.
정책 라이브러리 인덱스, 갭 대응 절차, 소스 구성에 sensible 디폴트를 썼습니다. 스킬
출력이 부자연스러우면 보통 튜닝해야 할 디폴트 — 어느 것인지 알려드릴 겁니다.
`/regulatory-legal:cold-start-interview --full`을 언제든 실행해 전체 인터뷰, 또는
`/regulatory-legal:cold-start-interview --redo <섹션>`으로 한 부분만 재실행."

**풀 셋업 경로:** 아래의 기존 인터뷰 흐름.

## 인터뷰 페이싱

- **답이 어딘가 적혀 있을 것이라고 가정.** 기존 watchlist·정책 인덱스·갭 분석 메모처럼
  어딘가 적혀 있을 가능성이 높은 정보를 묻는 질문은, 메모리에서 타이핑하라 하기 전에
  link 또는 paste를 요청한다. "링크 또는 doc를 paste 하거나 짧은 버전을 주세요"가 한
  문장 이상 답이 필요한 모든 것의 디폴트 ask.
- **batch 크기 — 부분 질문 카운트.** "한 turn에 2-3 질문 이상 묻지 않음"은 답할 수
  있는 prompt 2-3개를 의미하며, 부분 질문도 카운트. 테스트: 사용자가 스크롤 없이 답할
  수 있나? 가능하면 구조화된 tap-through 질문 선호.

**진짜 답에 멈춤.** 어떤 질문은 빠른 tap-through 답. 어떤 질문은 사용자가 타이핑·설명·
문서 업로드 필요(watchlist, 정책 라이브러리 인덱스, 재료성 캘리브레이션, 소스 구성).

- **묻고 기다림.** 명시적으로 말함: "이건 타이핑 답이 필요합니다 — 기다리겠습니다."
- **시드 문서 업로드:** "내용을 paste하거나, 파일 경로/URL 공유, 또는 '나중에 skip' 말씀.
  skip하면 실무 프로파일에 갭 플래그가 붙어 나중에 채울 수 있습니다." 그 뒤 실제로 기다림.
- **실무 프로파일 쓰기 전:** 인터뷰 검토. skip되거나 placeholder로 답한 질문 모두
  목록화(특히 watchlist·정책 라이브러리·소스 구성). 말함: "실무 프로파일을 쓰기 전, 아직
  열린 것: [목록]. 지금 채우시겠습니까, 아니면 placeholder로 두시겠습니까?" 그 뒤 답 기다림.
- **침묵의 갭이 있는 실무 프로파일을 절대 쓰지 않음.** 모든 `[PLACEHOLDER]`는 사용자가
  의도적으로 skip한 결정이지, 지나간 질문이 아님.
- **일시정지·재개.** 사용자에게 미리 말함: "멈춰야 하면 '잠시 멈춤' 말씀해주세요. 진행
  저장합니다. `/regulatory-legal:cold-start-interview`를 나중에 다시 실행하면 마지막
  지점에서 재개합니다." 일시정지되면, 부분 구성을 config 경로에 쓰고 상단에
  `<!-- SETUP PAUSED AT: [섹션 이름] — /regulatory-legal:cold-start-interview를 실행해
  재개 -->` 코멘트와 답 없는 필드에 `[PENDING]` 마커. 재실행되어 일시정지 config를
  발견하면, 사용자 환영: "다시 오셨네요. [섹션]에서 멈추셨습니다. 이전 답이 저장되어
  있습니다. 거기서 재개할까요, 처음부터 다시 할까요?" 이미 답한 질문은 재질문하지 않음.

**셋업 중 사용자가 주장한 법적 사실을 검증.** 사용자가 인터뷰 질문에 특정 규정·법조
번호·기한·임계점·소관 부처로 답할 때 — sanity-check 가능한 것 — 구성에 쓰기 전 점검.
충돌하면 surface: "임계점을 X라 하셨는데, 제 이해는 Y입니다 — 프로파일에 어느 것이
들어갈지 확인해주실 수 있나요? `[전제 플래그 — 검증 필요]`"

**국가법령정보 MCP가 연결되어 있으면 적극 활용해 검증.** 예: 사용자가 "입법예고 의견제출
기한은 항상 30일"이라 답하면, `mcp__korean-law__legal_analysis {"mode":"verify_citations",
"citation":"행정절차법 제43조"}` 호출로 확인 후 "행정절차법 §43은 입법예고 기간을 40일
이상으로 정합니다(공고문에서 단축 가능) `[검증 필요]` — 프로파일에 어떻게 쓸까요?" 식으로
보완. 검증된 답은 검증 로그에 자동 기록.

#### 누가 쓰고 있나?

> 이 플러그인을 일상에서 누가 쓸 건가요? (이건 모든 스킬의 work-product 헤더와 출력
> framing, 그리고 의견제출·규제 대응 문서 발송 전 게이트를 결정.)
>
> 1. **변호사·법률 전문가** — 변호사, 변호사 자격 없는 사내 법무 실무자(변호사 감독 하),
>    외부 로펌 어소시에이트.
> 2. **사내 법무·컴플라이언스(변호사)** — 사내 변호사 자격을 가진 CCO·법무팀원.
> 3. **비변호사 컴플라이언스 담당 (변호사 상담 가능)** — CCO 사무국의 비변호사, 규제
>    대응 매니저, 사내·외부 변호사를 부를 수 있는 창업자.
> 4. **비변호사 (정기적 변호사 상담 어려움)** — 본인이 직접 처리.

답이 3 또는 4이면, 한 번만(매 출력마다 반복하지 않음):

> 모든 기능을 쓸 수 있습니다 — 피드 점검, 정책 diff, 갭 추적, 의견제출 트래킹, 정책
> 레드라인 제안. 두 가지가 제가 일하는 방식에서 달라집니다:
>
> 1. **출력을 결재가 아닌 변호사 검토용 리서치로 framing합니다.** "이 정책 X로
>    개정하세요" 대신 "발견한 갭과 서명 전 물을 질문"을 받습니다.
> 2. **법적 결과가 있는 단계 전에 멈춥니다** — 규제기관에 의견제출서 제출, 정책 변경
>    확정, 규제 대응 문서 발송. 변호사 검토를 거쳤는지 묻고, 그 대화가 빠르게 진행되도록
>    짧은 brief를 만들어드립니다.
>
> 이건 disclaimer가 아닙니다. 플러그인이 잘하는 것(피드 점검·정리·구조)과 면허된 법적
> 판단(도구가 못 줌)을 구분하는 것입니다.

답이 4이면 추가:

> 자격 있는 변호사를 찾아야 하면: 대한변호사협회 변호사 검색(`https://www.koreanbar.or.kr`),
> 서울지방변호사회·각 지방회 변호사 검색이 빠른 출발점. 규제·행정 분야 전문 변호사도
> 검색 가능. 영세사업자는 중소기업중앙회 법률지원센터, 개인은 대한법률구조공단
> (`https://www.klac.or.kr`)이 출발점.

#### 실무 환경

> 다음 중 어느 것이 당신 실무에 가장 가깝나요? (이건 갭 대응 절차와 에스컬레이션에
> 영향 — 사내는 CCO/GC routing을 묻고, 법무법인은 파트너 결재 체인을 묻고, 단독·소규모는
> "에스컬레이션"을 "외부 변호사 상담"으로 매핑. 법무법인·단독 실무는 매터 워크스페이스도
> 활성화됩니다.)
>
> - **단독 / 소규모 사무소 (계층 없음)** — 결재 체인 질문 skip, 동료나 외부 변호사를
>   언제 부르는지 대신 물음. 매터 워크스페이스 활성화 제안.
> - **중·대형 법무법인** — 결재 체인, 시간당 비용 임계점, 누가 위에서 사인하는지 물음.
>   매터 워크스페이스 활성화 제안.
> - **사내 (인하우스)** — 에스컬레이션 매트릭스, CCO/GC가 누구, 무엇이 경영진으로
>   가는지 물음. 매터 워크스페이스는 기본 비활성.
> - **공공 / 규제개혁 지원 / 클리닉** — 감독 구조와 실무 제약 물음.
> - **위 중 어느 것도 아님** — 말씀해주세요. 적응하겠습니다.

답을 실무 프로파일의 `## 누가 이 플러그인을 사용하나`와 `## 매터 워크스페이스`(활성화
✓/✗)에 기록.

#### 무엇이 연결되어 있나?

> 이 플러그인은 다음과 작업할 수 있습니다: **국가법령정보 MCP**(한국 법령·판례·행정규칙·
> 의결례 부분), 문서 저장소(Google Drive, SharePoint), Slack. 어느 커넥터가 구성되어
> 있는지 점검하겠습니다 — 필요한 기능은 동작하고, 없는 기능은 침묵으로 실패하지 않고
> 수동으로 graceful하게 fallback합니다.

**구성된 것이 아닌 실제로 연결된 것을 점검.** `.mcp.json`에 나열된 커넥터는 *가용*.
실제 응답하는 커넥터는 *연결됨*. 이 둘은 다르며 혼동하면 trust가 깨진다.

**국가법령정보 점검 구체:**
```
mcp__korean-law__search_law "개인정보보호법"
```
응답 객체에 `laws` 또는 유사 필드와 1건 이상 결과가 있으면 ✓. 연결 오류·timeout·빈
배열이면 ✗.

연결되지 않은 커넥터에 대해 사용자에게 어떻게 연결하는지 알림. 그 뒤 다음 형식으로
발견 보고:

> - ✓ [통합] — 연결됨 (테스트됨)
> - ⚪ [통합] — 구성됨이나 검증 안 됨. MCP 설정에서 확인.
> - ✗ [통합] — 발견 안 됨. [기능]은 [수동 대안]으로 fallback. [연결 방법]. 나중에
>   셋업하면 `/regulatory-legal:cold-start-interview --check-integrations` 재실행.
>
> 전부 필요하진 않습니다. 핵심 기능은 파일 접근·수동 paste만으로 동작. **단, 국가법령정보
> MCP가 없으면 인용 검증이 동작하지 않으니 의존 전 외부 검증이 필요합니다.**

#### CLAUDE.md에 기록

`## 누가 이 플러그인을 사용하나`와 `## 연결 가능한 통합` 섹션 작성.

### Part 1: 규제기관 watchlist (2-3분)

*(이건 `/regulatory-legal:reg-feed-watcher`와 `reg-change-monitor` 에이전트에 영향 — 피드는
이 목록의 기관에서만 pull한다. 목록에 없는 기관은 사용자가 `/regulatory-legal:policy-diff`로
paste할 때까지 플러그인에 보이지 않는다.)*

> **[회사]는 무엇을 합니까?** 이게 가장 중요한 컨텍스트 — 어느 소관 부처·위원회가
> 적용되는지가 여기서 갈립니다. 타이핑할 필요 없음: 회사 웹사이트나 한 문장 버전(무엇을,
> 누구에게, 어떻게 파는지)이면 됩니다.

> 인터뷰 전: 기존 watchlist, 규제 추적 스프레드시트, 이전 갭 분석 메모가 있으면 paste
> 또는 파일 경로 공유, 또는 '없음' 말씀. 있으면 규제기관·watch 이유를 추출해 재타이핑을
> 줄입니다.

없으면, 아래 목록을 제시하고 어느 것이 적용되는지, 왜, 우선순위(leading — 정책 영향을
가장 직접적으로 주는 곳, vs monitor — 참고용)를 확인:

> - **개인정보보호위원회(PIPC)** — 개인정보보호법·시행령·고시, 과징금·시정조치 의결
> - **금융위원회·금융감독원** — 자본시장법·전자금융거래법·신용정보법 하위규정, 감독규정·시행세칙
> - **공정거래위원회** — 공정거래법·표시광고법·전자상거래법·하도급법·약관규제법, 의결례·고시
> - **식품의약품안전처(MFDS)** — 식품·의약품·의료기기·화장품 고시·행정예고
> - **방송통신위원회·과학기술정보통신부** — 정보통신망법·전기통신사업법 하위규정, 의결·고시
> - **고용노동부** — 근로기준법·산업안전보건법 하위규정, 고시·행정예고
> - **소관 부처 일반** — 개별법의 소관 부처(국토부·환경부·산업부 등) — 우리 사업에 적용되는
>   것을 지정
>
> 이 중 어느 것도 안 맞으면 자유 형식으로 말씀해주세요 — 다른 위원회·부처도 추가 가능.

각 기관에 대해: 왜 watch하는가("핀테크라 금융위는 당연" vs "PIPC 조사 이력 있음")? 안
watch하는데 watch해야 할 것 같은 곳이 있나?

**"우리 분야" 정의(중요).** 규제기관 의결·시정조치는 재료성 표에서 "우리 분야면
review-worthy, 아니면 FYI"로 분기한다(CLAUDE.md `## 중요도(재료성) 임계값`). 물어본다:
"어떤 사업자가 '우리와 유사한 사업자'입니까? 업종·규모·사업모델로 답해주세요 — 이게
다이제스트에서 남의 시정조치를 review-worthy로 올릴지 FYI로 내릴지 가릅니다."

**watchlist를 안 올리면:** 이 섹션 끝에서 제안: "공유·유지할 독립 watchlist 메모로 write
up할까요? 방금 캡처한 같은 내용 — 규제기관, watch 이유, 피드 — 배포하거나 새 팀원에게
줄 수 있는 팀용 문서 포맷."

### Part 2: 중요도(재료성) 임계값 캘리브레이션 + 갭 대응 절차 (3-4분)

*(이건 `/regulatory-legal:reg-feed-watcher`와 `gap-surfacer`에 영향. 재료성 분류의 **골격**은
CLAUDE.md `## 중요도(재료성) 임계값` 표에 이미 고정되어 있다 — 공포 법령 항상 material /
입법예고·행정예고 review-worthy / 계류 법률안 watch 고정 / 규제기관 의결 분야 매칭 /
보도자료 FYI. **이 표의 골격은 바꾸지 않는다** — 계류 법률안을 material로 올리는 것은 이
플러그인의 가장 위험한 오분류다. 여기서 조정하는 건 "우리 분야" 경계와 예외적 즉시-알림
항목이다.)*

몇 가지 예를 walk through. 각각에 대해, 즉시 알림 vs 주간 다이제스트 vs FYI만:

- watch 기관에서 나온 공포 법령 개정 → **항상 즉시**(정본 고정, 확인만)
- watch 기관의 입법예고(의견제출 기간 열림) → **항상 review-worthy**(정본 고정, 확인만)
- 국회 계류 법률안 → **항상 watch 고정, 절대 material 아님**(정본 고정 — 이 규율을
  사용자가 이해하는지 확인: "계류 법안이 통과 가능성이 매우 높아 보여도, 공포되기 전까지는
  컴플라이언스 의무가 생기지 않습니다. 동의하시나요?")
- 우리 분야 사업자에 대한 규제기관 시정조치·과징금 → review-worthy(우리 분야 정의에 따라)
- 우리 분야 밖 사업자에 대한 시정조치이나 우리도 하는 관행에 대한 것 → 사용자에게 확인:
  즉시? 다이제스트? ("우리가 안 하는 관행이면 FYI, 하는 관행이면 review-worthy로
  올리시겠습니까?")
- 회장·위원장 발언(정책 방향성 신호) → 보통 FYI, 특정 조사·시정조치 하의 회사는 예외로
  올리고 싶어할 수 있음 — 확인
- 부처 보도자료(가이드라인 배포 등, 구속력 없음) → 보통 FYI

**예외적 즉시-알림 항목.** "회사명이 언급되면", "특정 기관의 어떤 결정이든" 같은 사용자
고유의 즉시-알림 규칙이 있으면 캡처. 이건 CLAUDE.md 표의 골격 위에 얹는 추가 규칙이지
표를 바꾸는 게 아니다.

**갭 대응 절차:**
- **누가 규제 변경을 트리아지하나?** (`/policy-diff`가 갭을 찾았을 때 처음 보는 사람)
- **누가 정책 업데이트를 owner하나?** (갭을 닫을 책임)
- **갭 추적 방법?** 티켓 시스템 / 스프레드시트 / `gap-tracker.yaml`(플러그인 내장 디폴트)
- **material 갭 에스컬레이션 대상은?** (성명 또는 역할 — CCO, GC, 상사)

**임계값을 안 올리면:** 이 섹션 끝에서 제안: "이 재료성 캘리브레이션을 팀이 참조할 룰
문서로 write up할까요? 즉시/다이제스트/FYI 구분과 당신 예시를 담아서."

### Part 3: 정책 라이브러리 (2-3분)

*(이건 `/regulatory-legal:policy-diff`와 `/regulatory-legal:gaps`에 영향 — 들어오는 모든
규제 변경이 이 라이브러리와 대조되어 어느 정책이 걸리고 누가 owner인지 찾는다.)*

> 인터뷰 전: 기존 정책 라이브러리 인덱스 — 스프레드시트, 목차, 위키 페이지 — 가 있으면
> paste 또는 파일 경로 공유, 또는 '없음' 말씀. 있으면 재구축 대신 import.

없으면:

> 정책 폴더를 알려주세요. 거기 있는 것을 색인해서 규제가 바뀔 때 어느 정책이 걸리는지
> 알려드릴 수 있게.

- 정책은 어디 있나? (Drive, SharePoint, Confluence, Notion, 로컬)
- 명명 규칙이나 인덱스가 있나, 아니면 그냥 파일들인가?
- 어느 정책을 누가 owner하나? (갭을 올바른 사람에게 라우팅하기 위해)

**정책 인덱스를 안 올리면:** 이 섹션 끝에서 제안: "이걸 독립 정책-owner십 인덱스로
write up할까요? 새 CCO나 컴플라이언스 신입이 첫날 전체 그림을 얻을 수 있는 포맷으로."

### Part 4: 소스·피드 구성 (3-4분)

*(이건 `/regulatory-legal:reg-feed-watcher`와 `reg-change-monitor` 에이전트가 어느 소스를
어느 주기로 점검할지 결정. `references/source-catalog.md`가 이 플러그인의 소스 카탈로그 —
법령·입법 / 중앙부처·위원회 / 분야별 규제기관·심판기관 / secondary 4개 카테고리로 정리되어
있다.)*

**Step 1: 공포 법령 피드(항상 활성).**

국가법령정보 MCP(`search_law`→`get_law_text` 현행/시행예정 대조)가 공포 법률·시행령·
시행규칙의 1순위 소스다. 관보(`gwanbo.go.kr`)가 공포 확인 1차 소스로 병행된다. 이건 모든
설치가 갖는 Tier 1 — 별도 구성 불필요.

**Step 2: 입법예고·행정예고 소스.**

국민참여입법센터(`opinion.lawmaking.go.kr`)와 watchlist 기관의 개별 공고 페이지가 소스다.
국가법령정보 커넥터 범위 밖 — 직접 검색(WebFetch)만 가능하다. watchlist의 각 기관에 대해
공고 페이지 URL을 아는지 물어보고, 모르면 `references/source-catalog.md`의 §2 표를 제시.

**Step 3: 국회 계류 법률안 소스.**

국회 의안정보시스템(`likms.assembly.go.kr/bill`)이 유일한 소스 — 커넥터 범위 밖, 직접
검색만. **이건 watch 고정이지 컴플라이언스 신호가 아님을 다시 확인**: "계류 법안은
추적하지만 절대 material로 올리지 않습니다. 동의하시나요?"

**Step 4: 규제기관 직접 피드.**

`references/source-catalog.md` §2·§3에서 watchlist 기관에 맞는 소스를 선택 — 대부분
공개 RSS가 없어 사이트 직접 점검·메일링 구독이 필요함을 설명. 카탈로그 §6의 추천
스타터 팩(개인정보·데이터 / 금융·자본시장 / 소비자·유통·제조)이 watchlist와 맞으면 그걸
출발점으로 제안.

**Step 5: secondary 소스(선택).**

법률신문·로펌 뉴스레터·Lexology 같은 2차 소스를 구독 중인지. 구독 중이면 enrichment로
추가하되 `[검증 필요]`(2차)로 항상 태그됨을 설명.

**Step 6: 수동 입력(항상 활성).**

> 법률신문, 로펌 뉴스레터, 또는 다른 어디서든 시스템에 넣고 싶은 규제 변경을 보면 —
> 그냥 paste하시면 정책과 diff하고 갭을 추적합니다. 구독 없이도 동작합니다.

**Step 7: 점검 주기·다이제스트 출력·의견제출 owner.**

- 점검 주기? 매일 / 주 1회
- 다이제스트 출력 경로? (디폴트:
  `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/digests/`)
- **의견제출 결정 owner는?** *(`/regulatory-legal:comments`에 영향 — 입법예고·행정예고가
  감지될 때 이 owner에게 배정)*

**커버리지 점검.** watchlist에 있는 카테고리(법령·입법 / 중앙부처·위원회 / 분야별) 중
소스가 0개이거나 매우 적은 곳이 있으면 플래그: "watchlist에 [카테고리]가 있으나
`references/source-catalog.md`에 소스가 [N]개만 확인됩니다. 추가할까요?"

## 실무 프로파일 작성

플러그인 디렉토리의 `CLAUDE.md` 템플릿 구조를 사용. 모든 답을 적절한 섹션에 채우기.
`[PLACEHOLDER]` 마커는 사용자가 의도적으로 skip한 항목에만 남김(silent 갭이 아니라
명시적 결정). **`## 중요도(재료성) 임계값` 표의 골격은 절대 다시 쓰지 않는다** — 그
표는 CLAUDE.md 템플릿에 이미 정본으로 고정되어 있다; 인터뷰는 "우리 분야" 정의와
예외 즉시-알림 항목만 그 아래·주변에 추가한다.

자동화 가능한 검증:
- 사용자가 인용한 절차 조문(행정절차법 §41-§47, 국회법 §82-2 등)은
  `mcp__korean-law__legal_analysis {"mode":"verify_citations"}` 호출로 대조
- 사용자가 인용한 법령 조문은 `mcp__korean-law__get_law_text`로 현행 텍스트 대조
- 사용자가 인용한 규제기관 의결례는 *국가법령정보 부분 커버* — 사용자에게 기관 공식
  사이트 URL 부탁(`docs/DATA_GAPS.md` 참조), `[규제 당국 결정 — 직접 검증 필요]` 태그

작성 후, `verification-log.md`에 다음을 추가:

`[YYYY-MM-DD] 콜드스타트 인터뷰 - [성명]이(가) 다음을 확인: [검증한 항목 목록] - 국가법령정보에 대조해 [확인 / 정정]`

## 작성 후

**이 플러그인이 무엇을 도와줄 수 있는지 보여주기.** 닫기 전 제안:

> **제가 무엇을 도와드릴 수 있는지 보시겠습니까?**

yes면 다음 맞춤 목록 보여줌 (일반 템플릿이 아님 — 이 플러그인이 가장 잘하는 구체적인 것들):

> **규제 대응 실무에서 제가 잘하는 것:**
>
> - **규제 피드 점검** — 예: "watchlist를 재료성 임계값으로 필터한 다이제스트, 공포
>   법령·입법예고·계류 법안을 item-type별로 분기." 시도: `/regulatory-legal:reg-feed-watcher`
> - **규제 변경을 정책 라이브러리와 diff** — 예: "새 규정이 정확히 어느 내부 정책을 건드리고
>   무엇을 갱신해야 하는지." 시도: `/regulatory-legal:policy-diff`
> - **갭 트래커 열기** — 예: "포트폴리오 전체에서 무엇이 플래그되고 아직 안 닫혔는지, owner와
>   기한과 함께." 시도: `/regulatory-legal:gaps`
> - **의견제출 기간 추적** — 예: "열린 것, 마감일, 제출 여부 결정 로그." 시도:
>   `/regulatory-legal:comments`
> - **정책 레드라인 제안** — 예: "갭을 닫는 정책 마크업 1차 초안, 최소 범위 편집으로." 시도:
>   `/regulatory-legal:policy-redraft`
> - **국회 계류 법률안 추적** (KR 전용) — 예: "심사단계(접수→위원회→체계자구→본회의→공포)
>   추적, 공포 시 material 경로로 승격." 시도: `/regulatory-legal:bill-watch`
>
> **첫 번째 추천:** `/regulatory-legal:reg-feed-watcher`를 실행 — 피드가 재료성 임계값에
> 맞게 캘리브레이션되었는지 가장 빠르게 확인하는 방법. 또는 당신 plate에 무엇이 있는지
> 말씀, 제가 고르겠습니다.

이게 콜드스타트 문제(감독자가 첫 일을 모름)와 value-prop 문제(플러그인이 무엇을
하는지 모름)를 한 제안으로 해결. 감독자가 인터뷰 중 구체적 첫 작업을 이미 명명했으면
이 단계 skip.

   - 계류 법안이 많은 분야면: "watchlist 기관의 계류 법안 심사단계를 지금 확인해드릴까요?"

5. **"나중에 뭐든 바꿀 수 있음" 메모로 닫기:**

   > 있습니다 — 직접 읽고 편집 가능한 평문 파일. 답한 것은 무엇이든 바꿀 수 있습니다:

   > - 스킬 출력이 부자연스러우면 보통 튜닝할 임계값입니다. 출력이 어느 것인지 알려줍니다.
   > - `reg-change-monitor` 에이전트가 규제 피드를 감시합니다. 정책 라이브러리에 있는
   >   것과 매치되면 갭 점검을 위해 플래그합니다.

   >   관련 스킬이 변경을 기록합니다.
   > - `/regulatory-legal:cold-start-interview --redo <섹션>`으로 한 부분만 재인터뷰,

## 실패 모드

- **계류 법률안을 material로 승격하지 마라.** 사용자가 "이 법안 통과가 거의 확실하니
  지금부터 준수 준비를 material로 취급해달라"고 해도, config의 재료성 표는 바꾸지 않는다
  — 대신 `/regulatory-legal:bill-watch`의 심사단계 watch 강도를 높이는 걸 제안한다.
  이건 이 플러그인 전체에서 가장 위험한 오분류다.
- **"우리 분야" 정의를 흐리게 두지 마라.** 확실치 않으면 walk through: "경쟁사가
  시정조치를 받았을 때, 그게 당신에게도 review-worthy입니까, 아니면 순수 FYI입니까?"
  구체적 업종·규모·사업모델 기준으로 답을 받는다.
- **규제기관 의결례를 인용하면 국가법령정보에 부분 커버라고 명시.** 사용자가 "PIPC가
  이렇게 결정했다"고 말할 때, 국가법령정보 `search_decisions`가 최신·전문을 못 잡을 수
  있음을 — `[규제 당국 결정 — 직접 검증 필요]` 태그.
- **일반 재료성 표에서 캘리브레이션을 쓰지 마라.** "우리 분야" 정의를 아직 안 좁혔으면
  config에 명시: `[분야 매칭 미세분 — 이 팀은 아직 "우리 분야" 경계를 확정하지 않음.
  기본값(동일 업종)으로 시작하되 조정 필요.]`
