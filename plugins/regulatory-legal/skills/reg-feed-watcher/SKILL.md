# /reg-feed-watcher

1. `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md` 로드 → watch 기관,
   재료성 임계값, 소스·피드 구성.
2. 아래 워크플로우 실행.
3. 각 소스를 점검. 재료성으로 필터.
4. 산출: 새로 나온 것을 재료성 tier별로 정리.

---

## 목적

소스를 점검한다. 재료성으로 필터한다. 남는 것을 산출한다. **필터가 가치다** — 걸러지지 않은
피드는 소음이다. 미국 원본의 "Federal Register API 한 방"은 한국에서 존재하지 않는다 —
공포 법령·예고안·계류 의안·기관 의결이 **서로 다른 소스에 흩어져 있고, 발생하는 법적 효과도
다르다.** 이 스킬의 핵심은 그 매트릭스를 item-type별로 올바른 소스에 붙이고, **계류 의안을
공포 법령으로 착각하지 않는 것**이다(아래 재료성 분류).

## 컨텍스트 로드

`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md` → watch 기관, 재료성
임계값, 소스·피드 구성, 다이제스트 출력 경로(설정 시).

`references/source-catalog.md`(이 스킬 디렉토리) → 법령·입법 / 중앙부처·위원회 / 분야별 규제기관 /
secondary 카테고리별 소스 카탈로그. 새 소스를 구성하거나 watch리스트에 커버리지 갭이 있을 때 참조
(Step 0).

> **침묵의 보완 금지(CLAUDE.md `## 공유 가드레일`).** 소스 점검이 빈 결과·부분 결과를 내면 보고
> 후 멈춘다. **웹 검색·모델 지식으로 갭을 조용히 메우지 않는다.** 옵션 제시: (1) 날짜 범위 확장,
> (2) 다른 소스·도구, (3) 웹 검색(`[웹 검색 — 검증 필요]` 태그, 발령 기관 사이트 대조 전 의존
> 금지), (4) 여기서 멈춤. 낮은 신뢰도 소스를 받아들일지는 변호사가 결정한다 — Claude가 대신
> 결정하지 않는다.

## 워크플로우

### Step 0: 커버리지 점검 (점검 전에)

점검을 돌리기 전, CLAUDE.md의 watch 기관 + 소스·피드 구성을 `references/source-catalog.md`의
카테고리와 대조한다:

- 사용자의 watch리스트가 어느 카테고리(법령·입법 / 중앙부처·위원회 / 분야별 규제기관 / secondary)를
  신경 쓰는가?
- 그 카테고리 중 소스가 0개이거나 매우 적은 곳은 어디인가?

명백한 갭이 있으면 — 예: watch리스트에 "금융 규제"가 있는데 소스에 금융위 하나만 있고 금감원·
여신금융협회 공고가 빠진 경우 — 다이제스트 상단에 한 번 surface한다:

> **커버리지 갭 감지:** watch리스트에 [카테고리]가 있으나 소스가 [N]개만 구성되어 있습니다. 소스
> 카탈로그에는 이 카테고리에 [X]개 옵션이 있습니다(예: [상위 2-3개]). 추가를 제안할까요?
> `/regulatory-legal:cold-start-interview --redo`로 갱신하거나
> `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`를 직접 편집하세요.

같은 갭을 반복해 nag하지 않는다 — 사용자가 "지금은 지방 조례는 skip"이라고 했으면 존중하고
CLAUDE.md에 메모해 고정한다.

### Step 1: 점검 (KR 소스 매트릭스)

**미국 원본과의 결정적 차이:** 단일 API가 없다. item-type마다 소스가 다르고, 그 소스가 발생시키는
법적 효과(즉시 컴플라이언스 의무 / 의견제출 기한 / 아직 아무 의무 없음)도 다르다. watch리스트의 각
기관·분야에 대해 아래 매트릭스대로 점검한다.

| item-type | 소스·점검 방법 | 출처 태그 | 커버리지 |
|---|---|---|---|
| **공포 법령**(법률·시행령·시행규칙 개정) | 국가법령정보 MCP `search_law` → `get_law_text`(현행/시행예정 대조) · 관보(gwanbo.go.kr) | `[국가법령정보]` | 커넥터 커버 |
| **입법예고 / 행정예고**(NPRM 대응) | 법제처 국민참여입법센터(`opinion.lawmaking.go.kr`) + 각 부처 공고 페이지 → **직접 WebFetch** | `[법제처]` | **커넥터 범위 밖** — 직접 검색 |
| **국회 계류 법률안**(ANPR/RFI 대응) | 국회 의안정보시스템(`likms.assembly.go.kr/bill`) → **직접 검색** | `[국회 의안시스템 — 직접 검증 필요]` | **커넥터 범위 밖** — 직접 검색, **watch 고정** |
| **행정규칙**(고시·훈령·예규 / agency guidance 대응) | 국가법령정보 행정규칙 데이터(부분) → 미검색·불완전 시 법제처 행정규칙정보시스템·소관 부처 사이트 | `[국가법령정보]` 부분 / 미검색분 `[검증 필요]` | 부분 커버 — flag |
| **규제기관 의결·시정조치**(enforcement 대응) | 국가법령정보 `search_decisions`(해당 도메인, 부분) + 각 기관 보도자료·의결 페이지 | `[규제 당국 결정 — 직접 검증 필요]` | 부분 커버 — flag |
| **부처·위원회 보도자료·행정예고 페이지**(agency RSS 대응) | 각 기관 사이트 보도자료·공고 페이지(RSS 있으면 파싱, 없으면 페이지 점검) | `[검증 필요]`(발령 기관 대조 전) | 소스별 상이 |

**소스 매트릭스 사용 규율:**

- **공포 법령은 국가법령정보로 현행/시행예정을 반드시 대조한다.** 공포일 ≠ 시행일이다 — 부칙
  경과조치를 확인한다. `search_law`로 lawId/MST를 받고 `get_law_text`로 현행 조문을 가져온다:
  ```
  mcp__korean-law__search_law {"query": "개인정보보호법"}
  mcp__korean-law__get_law_text {"query": "개인정보보호법 제28조의8"}
  mcp__korean-law__legal_research {"task": "amendment_track", "query": "개인정보보호법"}
  ```
- **입법예고·행정예고·계류 의안은 커넥터가 커버하지 않는다.** 국민참여입법센터·의안정보시스템을
  직접 검색(WebFetch)하고 위 태그를 붙인다. **지어내지 않는다** — 접근 못 하면 사용자에게 공고문·
  법안 URL을 부탁한다(침묵의 보완 금지).
- **계류 의안은 절대 material로 분류하지 않는다.** 아직 법이 아니므로 컴플라이언스 의무를 만들지
  않는다 — Step 2의 watch 경로로 고정한다.

**Tier 1 — 무료 소스(항상 활성).** 국가법령정보 MCP(공포 법령·행정규칙 부분·의결례 부분) +
국민참여입법센터·의안정보시스템·관보·각 기관 사이트 직접 점검. 모든 설치가 Tier 1을 가진다.

**Tier 2 — 유료·확장 소스(구성된 경우).** 대안 법제 MCP(`ChangooLee/mcp-kr-legislation` — 위원회
결정문 24종 등) 또는 유료 규제 트래커가 구성되어 있으면 watch 기관으로 필터해 갱신 조회. 소스 간
중복 제거 — 같은 문서가 여러 소스에 나타나면 가장 풍부한 소스를 enriched 출력에 쓴다.

**소스 attribution.** 모든 인용·규제 항목에 출처를 태그한다(위 매트릭스의 태그 + `[사용자 제공]`).
`verify`가 붙은 항목은 도구로 가져온 항목보다 fabrication 위험이 높으므로 먼저 검증한다. 태그를
strip·collapse하지 않는다 — 어느 인용을 먼저 검증할지에 대한 사용자의 가장 빠른 신호다.

**secondary 소스.** 카탈로그의 일부(법률신문, 로펌 뉴스레터·규제 트래커, Lexology·JD Supra 유형)는
1차 규제 조치를 **보도**할 뿐 1차 소스가 아니다. 이 소스에서 온 항목은 소스명 태그에 더해
`[검증 필요]`(2차 소스)를 붙인다 — 예: `[법률신문] [검증 필요]`. 다이제스트에서 2차 소스 항목이
규제기관 조치를 서술하면 메모를 단다: "→ 1차 소스 추적: [알면 기관 사이트 링크, 모르면 '의존 전
발령 기관 사이트에서 확인']." **2차 소스 항목을 그 자체 강도로 "항상 material"로 분류하지 않는다** —
1차 소스가 확인될 때까지 tier를 한 단계 내린다.

**Tier 3 — 수동 입력.**

사용자가 스케줄드 점검에서가 아니라 규제 텍스트·요약을 직접 paste한 경우(입법예고 공고문, 관보
게재 개정문, 의결서, 법안): paste한 내용을 단일 항목으로 취급하고, Step 2 분류로 바로 넘어가며,
소스를 `[사용자 제공]`으로 기록한다. 소스 점검 불필요. 이 경로는 구독 상태와 무관하게 작동한다.

점검 후 확인 시각을 기록한다. 다음 스케줄드 실행은 여기서부터 점검한다.

### Step 2: 분류 (재료성)

각 항목은 CLAUDE.md `## 중요도(재료성) 임계값`에 따라 재료성 tier를 받는다. **이 표는 그 정본을
그대로 옮긴 것이다** — CLAUDE.md와 충돌하면 CLAUDE.md가 우선한다.

| item-type | 분류 | 처리 규율 |
|---|---|---|
| **공포된 법률·시행령·시행규칙 개정** | **항상 material** | 관보 공포 확인 → **시행일을 반드시 기록**(공포일 ≠ 시행일, 부칙 경과조치 확인). `get_law_text`로 현행/시행예정 대조. `[국가법령정보]` |
| **입법예고 / 행정예고** | **review-worthy** | 의견제출 마감을 반드시 기록(아래 마감 추출). comment-tracker로 라우팅. `[법제처]` |
| **국회 계류 법률안** | **watch 고정 (절대 material 아님)** | 심사단계(접수→위원회→체계자구→본회의→공포)만 추적. 공포 확인 전 `status_verified: false`, 🔴 Overdue 진입 금지. 공포되면 위 "공포 법령" 경로로 승격. `[국회 의안시스템 — 직접 검증 필요]` |
| **규제기관 의결·시정조치** | **분야 매칭**(우리 분야면 review-worthy, 아니면 FYI) | `search_decisions` 부분 커버 → 최신·전문은 기관 사이트. `[규제 당국 결정 — 직접 검증 필요]` |
| **보도자료·정책해설·간담회** | **FYI** | 방향성 신호로만. 법적 의무·기한 없음. 인용 시 `[검증 필요]` |

**계류 의안 처리 — 명시적.** 계류 법률안은 입법예고·공포 법령과 결정적으로 다르다: 아직 법을
바꾸지 않고, 컴플라이언스 의무를 만들지 않는다. 별개 분기로 취급한다:

- **절대** 계류 의안을 "항상 material"로 분류하지 않는다 — 공포(관보 게재) 전까지 컴플라이언스
  영향은 0이다.
- 심사단계만 추적하고 방향성 신호로만 쓴다. policy-diff로는 **사전 포지셔닝 분석**으로만 라우팅하고,
  갭 종결 diff로는 라우팅하지 않는다.
- **새로 감지된 계류 의안은 `/regulatory-legal:bill-watch`로 핸드오프한다.** 심사단계 추적과
  gap-tracker.yaml의 `watch` 등록(`status_verified: false`, Overdue 금지)은 bill-watch가 소유한다 —
  reg-feed-watcher는 watch 항목을 직접 만들지 않는다.
- comment-tracker에 `item_type: 의안` 행을 쓰는 것은 **국회입법예고 의견제출 기간이 실제로 열려 있을
  때만**이다(그때 `due:` = 그 국회입법예고 마감). 의견제출 기간이 없는 계류 의안은 comment-tracker에
  기록하지 않는다 — bill-watch의 watch 등록으로 충분하다.
- 다이제스트 항목에 명시한다: "계류 법률안 — 아직 법 아님. 심사단계 [단계]. `status_verified: false`.
  공포 확인 전 material 아님."
- **계류 의안은 gap-tracker의 컴플라이언스-갭(🔴 Overdue 대상) 경로에 넣지 않는다** — watch 등록은
  bill-watch 경유다. `gap-surfacer`는 이 유래 항목을 공포 확인 전 🔴 Overdue로 올리지 않는다.

**입법예고·행정예고 의견제출 마감 처리:**

"skip" 위 tier로 분류된 모든 입법예고·행정예고에 대해:

- **공고문에서 실제 의견제출 마감일을 추출한다**(공고문의 날짜가 정본이다).
- **법정 하한(디폴트 참고용, 사안별 단축 가능):**
  - **입법예고 40일 이상** — 행정절차법 §43 `[검증 필요]`
  - **행정예고 20일 이상** — 행정절차법 §46 `[검증 필요]`
  - **국회입법예고(위원회) 10일 이상** — 국회법 §82-2 `[검증 필요]`
  > 위 세 조문·기간은 라이브 검증 전이므로 전부 `[검증 필요]` 상태다. `verify_citations`로 확정하되

## 산출

다이제스트는 디폴트로 채팅에 나간다. **FYI 위 항목이 하나 이상이면 공유 가능한 파일로도 쓴다** —
CLAUDE.md가 명시적으로 `다이제스트 출력 → 채팅만`으로 설정하지 않는 한.

**파일 출력 동작:**

1. CLAUDE.md `## 소스·피드 구성`의 `다이제스트 출력 경로`를 찾는다. 설정되어 있으면 거기 쓴다.
   미설정 디폴트: `~/regulatory-legal-digests/reg-digest-YYYY-MM-DD.md`.
2. 필요 시 부모 디렉토리 생성.
3. 전체 다이제스트를 Markdown으로 쓴다(채팅 출력과 동일 — work-product 헤더, 출처 태그,
   인용 검증 푸터 포함).
4. 오늘 날짜 파일이 이미 있으면 **덮어쓰지 않고** 타임스탬프 서브헤더로 새 섹션을 append한다 —
   같은 날 여러 번 돌 수 있다(아침 다이제스트, 임시 점검).
5. 쓴 후 알린다: "다이제스트를 `<경로>`에 저장했습니다. 그대로 공유하거나 Pandoc으로 .docx 변환:
   `pandoc <경로> -o <경로>.docx`."
6. 쓰기 실패 시(권한, 사용자가 생성 승인 안 한 디렉토리, 디스크) 채팅 전용 출력으로 폴백하고 그
   사실을 말한다 — 파일 요청을 침묵으로 drop하지 않는다.

## 구성 의존 폴백

이 스킬은 watch 기관·재료성 임계값·소스 구성을
`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`에서 읽는다. 필요한 값이
아직 `[PLACEHOLDER]`이거나 비어 있으면 출력에 구체적으로(일반론 아님) 말한다:

- **watch 기관 비어 있음:** 멈추고 말한다 — "구성의 watch 기관이 비어 있습니다. 어느 규제기관·법령
  분야를 볼지 없이는 소스를 점검할 수 없습니다. `/regulatory-legal:cold-start-interview --redo`를
  실행하거나 `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`를 편집해 최소
  하나의 기관·분야를 추가하세요."
- **재료성 임계값 비어 있음:** 디폴트 tier로 폴백하고 덧붙인다 — "이 출력은 구성에 맞춤 임계값이
  없어 디폴트 재료성 tier(CLAUDE.md `## 중요도(재료성) 임계값`)를 썼습니다.
  `/regulatory-legal:cold-start-interview --redo`로 조정하세요."
- **소스 구성 비어 있음:** 국가법령정보 MCP(공포 법령)만 점검하고 덧붙인다 — "이 출력은 구성에
  입법예고·규제기관 직접 소스가 없어 국가법령정보 공포 법령만 점검했습니다. 국민참여입법센터·
  의안정보시스템·규제기관 소스를 추가하세요 — `/regulatory-legal:cold-start-interview --redo`."

관련 값이 채워져 있으면 구성에 대해 아무 말도 하지 않는다.

FYI 위 아무것도 없으면: "조용합니다. FYI [N]건, 대응 필요 없음."

## 핸드오프

- **policy-diff로:** 정책 영향이 예상되는 "항상 material" 항목(공포 법령) → diff 실행 제안. **계류
  의안·입법예고안은 사전 포지셔닝 분석으로만**(갭 종결 diff 아님).
- **bill-watch로:** 새로 감지된 계류 의안 → 심사단계 추적·gap-tracker.yaml `watch` 등록을 bill-watch가
  소유. reg-feed-watcher는 watch 항목을 직접 만들지 않는다.
- **gap-surfacer로:** diff가 갭을 찾으면 → 추적. **계류 의안 유래 항목(bill-watch가 등록한 watch)은
  `status_verified: false`, 공포 확인 전 Overdue 진입 금지.**
- **comment-tracker로:** "skip" 위로 분류된 모든 입법예고·행정예고 → 의견제출 마감 자동 기록
  (tracking 활성 시). 계류 의안은 **국회입법예고 의견제출 기간이 열려 있을 때만** `item_type: 의안`
  행으로 기록. `/regulatory-legal:comments`가 소비.

## 이 스킬이 하지 않는 것

- 모든 항목을 전문으로 읽기. 분류·enrich만 한다 — 깊은 독해는 필터를 통과한 항목의 몫이다.
- 재료성 임계값 변경. 필터가 틀렸으면
  `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`를 편집한다.
- 국가법령정보 없이 공포 법령을 단정. 커넥터 미응답 시 인용은 `[모델 지식 — 검증 필요]`로 떨어지고
  검토자 메모 **소스:** 라인에 기록한다.
- **계류 의안을 material로 승격.** 이 플러그인의 가장 위험한 오분류 — 존재하지 않는 컴플라이언스
  의무를 만든다. 공포 확인 없이는 어떤 법률안도 material이 아니다.
