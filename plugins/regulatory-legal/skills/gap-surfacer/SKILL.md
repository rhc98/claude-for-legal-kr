# Gap Surfacer (갭 추적기)

> Owner 알림: 디폴트 on. owner를 opt-out하려면 `owner_slack`을 비워 둔다.

## 건별 발송 확인 — 예외 없음

Slack 메시지(배정 알림, 지연 리마인더, 일괄 알림, 상태 보고)를 보내기 전, 모두:

1. 사용자에게 무엇을 누구에게 보낼지 정확히 보여준다: "[N]명에게 이걸 보내려 합니다: [미리보기]."
2. 명시적 yes를 기다린다.
3. 메시지에 인용·기한·컴플라이언스 결론이 포함되면 덧붙인다: "⚠️ 이 메시지의 인용은 미검증입니다 — 발송 전 현행 여부를 확인하지 않았습니다. '발송 전 검증 필요' 문구를 추가할까요?"
4. 확인 없이는 절대 보내지 않는다. 주기적으로도, 일괄로도, "어제도 보냈으니까"도 안 된다.

확인 없는 자동 발송은 이 플러그인에서 가장 돌이킬 수 없는 행동이다 — 이 플러그인 자신의 footer가 틀릴 수 있다고 말하는 내용을, 검사할 방법이 없는 사람들에게 보내는 것. 그 조합은 검토를 건너뛸 수 없다.

## 매터 컨텍스트

**매터 컨텍스트.** 실무 수준 CLAUDE.md의 `## 매터 워크스페이스`를 확인한다. `활성화`가 `✗`(사내 사용자 디폴트)이면 이 단락의 나머지는 건너뛴다 — 스킬은 실무 수준 컨텍스트를 쓰고 매터 machinery는 보이지 않는다. 활성화되어 있고 활성 매터가 없으면 묻는다: "이건 어느 매터입니까? `/regulatory-legal:matter-workspace switch <slug>`를 실행하거나 `practice-level`이라고 말해주세요." 활성 매터의 `matter.md`를 매터-특화 사실·override를 위해 로드한다. 출력은 매터 폴더 `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/matters/<매터-slug>/`에 쓴다. `교차 매터 컨텍스트`가 `on`이 아닌 한 다른 매터의 파일을 절대 읽지 않는다.

## 목적

갭은 발견되고 나서 잊힌다. 이 스킬은 갭이 닫힐 때까지 추적하고, 닫을 책임이 있는 사람에게 알린다.

## 트래커

`~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/gap-tracker.yaml`에 있다:

> **comment-tracker.yaml에 관한 노트:** `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/comment-tracker.yaml`은 comments 스킬이 소유하는(정본 스키마 포함) 형제 파일이다. reg-feed-watcher(입법예고·행정예고를 자동 기록)·bill-watch(국회입법예고 의견제출 마감이 열린 계류 의안을 `item_type: 의안`으로 기록)와 comments 스킬(사용자 주도 의견제출 결정을 추적)이 여기에 쓴다. 이 스킬은 그 파일을 읽거나 대조하지 않는다. comment-tracker 스키마를 수정하면 실제 사용자(reg-feed-watcher, bill-watch, comments) 모두 갱신한다.

```yaml
gaps:
  - id: GAP-001
    requirement: "[규정이 요구하는 것]"
    regulation: "[법령명 + 인용]"
    policy_affected: "[정책명 또는 '신규 정책 필요']"
    gap_type: "partial"  # none | partial | full | new-policy | watch | comment-decision
    owner: "[정책 인덱스의 성명]"
    owner_slack: "[Slack 사용자 ID 또는 핸들, 알면]"
    opened: 2026-03-01
    due: 2026-06-01  # 시행일, 내부 기한, 또는 의견제출 마감
    status_verified: true  # false면 상류 policy-diff가 이 규정이 시행 중임을 확인하지 못한 것 — 미검증 항목은 절대 🔴 Overdue에 진입하지 않는다
    status: "open"  # open | in-progress | closed | risk-accepted
    notified: false  # 배정 알림 발송 후 true
    resolution: ""  # 종결 시 기재
    stage: ""  # (선택) 의안 심사단계 — watch 항목 중 국회 계류 의안에만. bill-watch 전용. 예: 접수 | 위원회 심사 | 체계자구 | 본회의 의결 | 정부이송 | 공포
```

**미검증 규정에 대해 갭을 Overdue로 절대 분류하지 않는다.** 🔴 Overdue 분류는 "구속력 있는 기한을 놓쳤다"를 뜻한다. 규정의 상태가 미검증이면(policy-diff가 `status_verified: false`로 설정했거나, 규정이 12개월 이상 지났거나 적용일이 지났는데 현행성 확인이 없으면), 그 기한은 구속력이 없을 수 있다. 🟡 "검토 필요"를 쓰고 덧붙인다: "이 규정이 공표된 대로 시행 중이라면 [N]일 지연입니다. 에스컬레이션 전 규정 상태를 확인하세요." 미검증 규정 항목은 활성 overdue·due-soon 버킷이 아닌 `watch`로 라우팅한다 — `watch`의 재검토 주기가 항목이 컴플라이언스 갭으로 다시 표면화되기 전 규정 상태 확인을 강제한다.

**KR 규칙 — 국회 의안·미공포 예고안 유래 항목.** 갭 또는 watch 항목이 국회 계류 법률안(`[국회 의안시스템 — 직접 검증 필요]`) 또는 아직 공포되지 않은 입법예고·행정예고안(`[법제처]`, 공포 전)에서 유래한 경우, 트래커 진입 시 **항상** `status_verified: false`로 생성한다 — policy-diff의 판정과 무관하게 이 스킬 자체가 강제한다. 국가법령정보 `get_law_text`로 해당 법령의 공포·시행을 확인하기 전에는 절대 🔴 Overdue에 진입시키지 않는다. `due:`는 이런 항목에서 컴플라이언스 기한이 아니라 재검토일(예상 표결일·시행 예정일)이다. 공포가 확인되면 `status_verified: true`로 전환하고 `regulation` 필드의 인용을 국가법령정보 `[국가법령정보]` 태그로 갱신한다 — 이 승격만이 항목을 Overdue 판정 대상으로 만든다.

**`gap_type` 의미:**

| 값 | 의미 | 전형적 리마인더 주기 |
|---|---|---|
| `none` | 정책이 이미 요건을 커버. 감사 추적용으로만 기록. 드물어야 함 — 대부분이 `none`이면 diff가 잘못된 정책을 대상으로 돌고 있을 가능성. | 자동 리마인더 없음. |
| `partial` | 정책이 주제를 다루지만 새 요건을 완전히 커버하지 않음. 개정 필요. | 기한 30일 전. |
| `full` | 정책이 새 요건과 모순되거나 침묵으로 누락. 재작성 또는 신규 조항 필요. | 기한 30일 전. |
| `new-policy` | 이 요건을 커버하는 기존 정책 없음. 정책 신규 작성 필요. | 기한 30일 전. |
| `watch` | 전망성 항목 — 국회 계류 법률안, 입법예고·행정예고 초안 단계로 아직 확정 아님. 오늘 시점 컴플라이언스 의무 없음, 확정 후 정책 작업 대기. `due:`는 재검토일(예상 표결일 또는 1년 지평)이지 컴플라이언스 기한이 아니다. | 자동 리마인더 없음. 공포·시행 시 또는 재검토일에 재평가. |
| `comment-decision` | 입법예고·행정예고 단계 의견제출 결정 대기 — 팀이 의견제출 여부를 결정 중. `due:`는 의견제출 마감. | 기한 21일 전(의견 작성 기간이 짧으므로 컴플라이언스 갭보다 촘촘). |

`watch`나 `comment-decision` 항목은 컴플라이언스 갭이 아니다 — bill-watch·comments 스킬이 산출하는 확정 전 항목의 추적 산출물이다. 상태 보고에서 별도 버킷으로 표면화해서, 아침에 읽는 담당변호사가 "규제기관이 알아채기 전에 고쳐야 할 것"과 "지켜볼 것"을 한눈에 구분할 수 있게 한다.

## Modes

### Mode 1: policy-diff에서 ingest

policy-diff가 갭을 발견하면 gap-tracker.yaml에 추가한다. 중복 제거 — 같은 요건 + 같은 정책이면 같은 갭, 이중 카운트하지 않는다.

**ingest 후 owner에게 알림:**

Slack MCP가 연결되어 있고 `owner_slack`이 설정되어 있으면:

갭 owner에게 Slack DM을 보낸다 — 단, 이 파일 상단의 건별 발송 확인을 거친 후에만. 사용자에게 메시지를 미리보기하고, 명시적 yes를 기다린 뒤 보낸다:

```
📋 신규 컴플라이언스 갭이 배정되었습니다

갭: [GAP-ID] — [요건, 한 문장]
규정: [법령명 + 링크]
영향받는 정책: [정책명 또는 "신규 정책 필요"]
기한: [시행일]

전체 갭 트래커 보기: /regulatory-legal:gaps
```

발송 후 트래커 항목의 `notified: true`로 설정.

Slack MCP가 연결되어 있지 않으면: 상태 보고에 owner 알림이 발송되지 않았음을 기록하고 수동 후속조치로 플래그.

```markdown
**의존 전 인용을 검증하라.** 이 트래커의 규정 인용은 상류(reg-feed-watcher·bill-watch·policy-diff)에서 AI가 생성했고 1차 소스 대조는 아직이다. 갭을 종결하거나 위험수용 처리하기 전 — 또는 인증서·이사회 보고·규제기관 응답에 인용하기 전 — 국가법령정보, 관보, 또는 소관 기관 사이트로 근거 규정을 확인한다. AI 생성 규제 인용은 때때로 조작되거나 오인용되거나 stale하다. 상류에서 이어받은 출처 태그(`[국가법령정보]`, `[국회 의안시스템 — 직접 검증 필요]`, `[모델 지식 — 검증 필요]` 등)는 각 인용이 어디서 왔는지 보여준다 — `검증 필요` 태그가 조작 위험이 가장 높으므로 먼저 확인한다. 갭을 표면화할 때 태그를 절대 제거하지 않는다.
```

## Config-dependent fallback

이 스킬은 갭 대응 owner와 에스컬레이션 경로를 `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`에서 읽는다. 필요한 값이 비어 있거나 아직 `[PLACEHOLDER]`이면:

- **갭 대응 트리아저 미지정:** 배정을 열어 두고 출력에 덧붙인다: "`## 갭 대응 절차`에 트리아저가 설정되어 있지 않습니다. `/regulatory-legal:cold-start-interview --redo`로 지정하거나 `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`를 직접 편집해 신규 갭이 라우팅되게 하세요."
- **신규 ingest된 갭의 owner 미상(정책 라이브러리에 owner 없음):** `owner: [미배정]`으로 갭을 기록하고 덧붙인다: "[N]개 갭이 owner 없이 ingest되었습니다 — 정책 라이브러리가 해당 정책의 owner를 지정하지 않았기 때문입니다. 정책 라이브러리의 owner 컬럼을 채워 라우팅하세요."
- **material 지연 갭의 에스컬레이션 경로 미상:** 그래도 지연으로 보고하고 덧붙인다: "material 지연 갭의 에스컬레이션 경로가 설정되어 있지 않습니다. `/regulatory-legal:cold-start-interview --redo`로 설정하거나 `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`를 직접 편집하세요."

값이 채워져 있으면 config에 관해 아무 말도 하지 않는다.

**기한 리마인더 로직(상태 보고·스케줄드 에이전트 실행 시 동작):**

리마인더 주기는 `gap_type`의 함수다 — 컴플라이언스 갭은 30일 전 알림, comment-decision 항목은 21일 전(작성 기간이 짧으므로), watch 항목은 자동 리마인더 없음(입법예고·행정예고가 확정되면 재평가).

status가 "open" 또는 "in-progress"인 각 갭에 대해:
- `partial`, `full`, `new-policy`, `none`: 기한이 30일 이내이고 최근 7일 내 리마인더 미발송이면, Slack DM(제목 "⏰ 리마인더: 컴플라이언스 갭 [N]일 후 기한")을 PREVIEW하고 건별 확인을 기다린 후 발송.
- `comment-decision`: 의견제출 마감이 21일 이내이고 최근 7일 내 리마인더 미발송이면, Slack DM(제목 "💬 의견제출 결정 마감 [N]일 후")을 PREVIEW하고 건별 확인을 기다린 후 발송.
- `watch`: 자동 리마인더 없음. 트래커 재검토 시 또는 같은 규정에 대한 입법예고·행정예고가 기록될 때 재검토.
- 컴플라이언스 갭 기한이 지났으면: 보고서에 지연으로 플래그하고 Slack DM을 PREVIEW — 건별 확인을 기다린 후 발송.
- `comment-decision` 항목의 의견제출 마감이 지났고 의견이 미제출이면: 지연으로 플래그, Slack DM을 PREVIEW(건별 확인 대기), owner에게 `risk-accepted`(의도적 무의견) 또는 `closed`(의견 제출됨)로 메모와 함께 갱신 요청.
- 리마인더 발송 시각을 트래커에 기록해 반복 재촉을 방지한다.
- 일괄 리마인더도 건별 확인이 필요하다 — "12건 DM을 보내려 합니다" 미리보기 후 yes를 기다리는 것은 인정되나, 조용히 일괄 발사하는 것은 아니다.

### Consequential-action gate(컴플라이언스 인증)

**갭을 해소로 종결하기 전, 또는 규제 요건 준수를 인증하는 산출물(내부 attestation, 이사회 보고, 감사 응답, 규제기관 응답)을 산출하기 전:** `~/.claude/plugins/config/claude-for-legal-kr/regulatory-legal/CLAUDE.md`의 `## 누가 이 플러그인을 사용하나`를 읽는다. 역할이 **비변호사**이면:

> 컴플라이언스를 인증하는 것 — 또는 갭을 해소로 종결하는 것 — 은 법적 결과를 낳습니다. 인증은 나중에 틀린 것으로 밝혀지면 회사에 불리하게 쓰일 수 있고, 조기 종결은 미해소 노출을 방치합니다. 변호사와 이걸 검토했습니까? 예면 진행합니다. 아니면, 그들에게 가져갈 브리프입니다:
>
> - 갭(요건, 출처, 정책 diff가 발견한 것)
> - 제안된 해소가 커버하는 것과 커버하지 않는 것
> - 잔여 갭 또는 모호성
> - 미해결 질문과 아직 결정되지 않은 것
> - 잘못될 수 있는 것(과도한 인증, 미해소 잔여 의무, 기존 입장과의 불일치)
> - 변호사에게 물어볼 것(진정 종결인가, 근거를 문서화해 위험수용해야 하는가, 외부 자문 확인이 필요한가)
>
> 변호사를 찾아야 하면: **대한변호사협회**(www.koreanbar.or.kr) 또는 **소속 지방변호사회**의 변호사 소개 제도가 가장 빠른 출발점입니다.

이 게이트를 명시적 yes 없이 통과해 갭을 종결하거나 컴플라이언스 인증을 산출하지 않는다. 상태 보고·추적 뷰는 이 게이트를 요구하지 않는다.

### Mode 3: 갭 종결

```
/regulatory-legal:gaps --close GAP-001
해소: "정책 v2.3으로 갱신, [날짜] 승인"
```

상태를 closed로 갱신, 해소 내용과 종결일 기록.

### Mode 4: 갭 위험수용

때로는 "이건 고치지 않는다"가 답이다. 유효한 결정이나 — 문서화되어야 한다.

```
/regulatory-legal:gaps --accept GAP-002
근거: "[우리가 해당하지 않는 조건]에만 적용되는 요건. [트리거] 발생 시 재검토."
승인자: [권한 있는 성명]
```

상태 → risk-accepted. 트래커에는 남지만(삭제 아님) 열린-갭 보고에서는 빠진다.

## 통합: reg-change-monitor 에이전트

에이전트의 다이제스트에는 갭 카운트와 가장 오래된 열린 갭의 age가 포함된다. 뭔가 지연되면 다이제스트 상단에 온다. 에이전트는 기한 리마인더 점검도 실행하고 미발송 Slack 알림을 보낸다.

## 다음 단계 의사결정 트리로 마무리

CLAUDE.md `## 산출물`의 다음 단계 의사결정 트리로 종료한다. 이 스킬이 방금 산출한 것에 맞춰 옵션을 맞춤화한다 — 5개 디폴트 branch(X 초안 작성, 에스컬레이션, 사실 추가 확보, 관찰·대기, 다른 것)는 출발점이지 고정이 아니다. 트리가 산출물, 변호사가 고른다.

트래커에서 열린 갭이 ~10건을 넘거나, 사용자가 요청하면: 대시보드를 제안한다(CLAUDE.md `## 산출물 → 데이터 무거운 산출물의 대시보드 옵션` 참조). 이 산출물에 맞게 제안을 다듬는다 — 심각도별 카운트, 기한별 타임라인, owner·상태·최종 접촉일이 있는 정렬 가능한 그리드.

## 이 스킬이 하지 않는 것

- 스스로 갭을 종결하지 않는다. 종결에는 해소 메모와 그 메모가 설명하는 인간의 행동이 필요하다.
- Slack MCP가 구성되지 않았으면 Slack 알림을 보내지 않는다. 상태 보고에 플래그하는 것으로 대체한다.
- 갭당 7일 주기에 리마인더를 한 번 넘게 보내지 않는다. 한 번 재촉하지, 계속 재촉하지 않는다.
