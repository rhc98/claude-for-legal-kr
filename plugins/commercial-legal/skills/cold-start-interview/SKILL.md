# /cold-start-interview

1. `~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md` 확인 — populate되어
   있고 `--redo`가 아니면 덮어쓰기 전 확인.
2. 아래 인터뷰 워크플로우 실행.
3. 시드 문서: 표준 계약서(우리 양식 또는 자주 받는 상대방 양식), 표준 약관, 표준 NDA. 읽기.
4. 추출: 플레이북 입장(책임·면책·개인정보·기간·준거법), 약관 무효 위험 자세, NDA GREEN/YELLOW/RED
   기준, NDA 마무리 동작(closing_action). stated 입장과 시드 문서의 delta 메모.
5. 마이그레이션: populate된 CLAUDE.md(placeholder 없음)가
   `~/.claude/plugins/cache/claude-for-legal-kr/commercial-legal/*/CLAUDE.md`에는 있는데
   config 경로에 없다면, config 경로로 복사하고 무엇이 마이그레이션되었는지 사용자에게 보여줌.
6. `~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md` 작성(필요시 상위
   디렉토리 생성). 요약 보여주기. 첫 작업 제안.

## `--side`

`--side sales`(매도인·공급자측) 또는 `--side purchasing`(매수인·구매자측)으로 호출하면 한 측
플레이북만 구축·갱신한다. 다른 측은 건드리지 않는다. 검토 스킬이 "맞는 측이 `[미구성]`"으로
멈췄을 때, 또는 회사가 새로운 측의 거래를 시작할 때(예: 줄곧 매도인측이었는데 큰 벤더 조달을
시작) 빠르게 그 측만 채우는 용도.

## `--check-integrations`

통합 가용성 점검(CLM·전자서명·문서 저장소·Slack·**국가법령정보 MCP**)을 재실행하고
`~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md`의 `## 연결 가능한 통합`
섹션을 갱신. 재인터뷰는 하지 않음. MCP를 연결·해제한 뒤 플러그인이 인식하길 원하지만 전체 셋업을
다시 돌리고 싶지 않을 때 사용.

점검 시: MCP 도구 호출이 실제로 성공한 경우에만 ✓ 보고. 구성만 되어 있고 미테스트 커넥터는 ⚪로
표시하고 확인 방법 한 줄 첨부. `.mcp.json` 선언만으로 ✓ 보고하지 않음 — 사용자에게 "연결됨"으로
잘못 알려 trust를 깬다.

**국가법령정보(korean-law) MCP는 이 플러그인에서 1순위 통합이다.** 미연결이면:
- 모든 조문·판례 인용이 `[모델 지식 — 검증 필요]`로 떨어짐
- 환각방지 검증이 동작하지 않음 (약관규제법 §6–14, 민법 §398, 부정경쟁방지법 §2 대조 불가)
- 위임조문 체인 조회 불가

사용자에게 안내: "국가법령정보 MCP를 등록하세요: `claude mcp add korean-law "https://mcp.gomdori.app/law?oc=<OC키>" --transport http`.
무료 — open.law.go.kr에서 OC 인증키 발급 필요(1분). 자세한 내용은 `docs/LAW_MCP_INTEGRATION.md` 참조."

```
/commercial-legal:cold-start-interview
/commercial-legal:cold-start-interview --check-integrations
/commercial-legal:cold-start-interview --side sales
```

# 콜드스타트 인터뷰: 상사·계약 실무

## 목적

*이* 계약팀이 어떻게 일하는지 학습 — 어느 측에서 거래하는지(매도인·매수인·양쪽), 책임·면책·해지에서
무엇은 받아들이고 무엇은 거부하는지, 약관 무효 위험을 어디까지 플래그하는지, NDA를 무엇으로 GREEN·
YELLOW·RED로 보는지. `~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md`에
써서 다른 모든 스킬이 같은 이해에서 읽도록.

한국 상사 실무는 측에 따라 완전히 다르다. 우리 SaaS를 파는 매도인측 플레이북은 클라우드 벤더를
조달하는 매수인측 플레이북과 위험 성향·표준 조건·책임 방향이 정반대다. 인터뷰는 어느 측인지를 다른
어떤 것보다 먼저 알아낸다.

## 콜드스타트 점검

`~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md` 읽기:
- **존재하지 않음** → 인터뷰 시작.
- **`<!-- SETUP PAUSED AT: -->` 포함** → 사용자를 맞이하고 해당 섹션부터 재개 제안.
- **`[PLACEHOLDER]` 마커가 있는데 pause comment는 없음** → 템플릿이 완료되지 않았음; 새로 시작할지
  placeholder부터 재개할지 제안.
- **populate됨 (placeholder도 pause comment도 없음)** → 이미 구성됨; `--redo`가 아니면 skip.

템플릿 구조는 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`에 있다 — 섹션 골격으로 사용. 완성된 실무 프로파일을
config 경로에 쓰고 필요시 상위 디렉토리 생성.

옛 캐시 경로 `~/.claude/plugins/cache/claude-for-legal-kr/commercial-legal/*/CLAUDE.md`에 CLAUDE.md가
있는데 config 경로에는 없다면, 앞으로 복사.

## 공유 회사 프로필 확인

`~/.claude/plugins/config/claude-for-legal-kr/company-profile.md` 확인.

- **있으면:** 읽기. 한 줄 확인 보여주기: "당신은 [성명], [실무 환경], [회사], [업종], [관할]에서
  운영. 맞나요? (또는 '업데이트'를 말해 공유 프로필 변경.)" 확인되면 회사 질문은 skip — 바로
  플러그인 특화 질문으로.
- **없으면:** 당신이 이 사용자가 셋업한 첫 플러그인. 오리엔테이션과 fork 후, 회사 질문을 묻고 공유
  프로필에 쓴다(플러그인 루트의 `references/company-profile-template.md` 템플릿 따름). 그 후 플러그인
  특화 질문 계속. 사용자에게: "회사 프로필 저장 완료 — 다른 법무 플러그인들이 이걸 읽고 이 질문들을
  skip합니다."

공유 프로필에 속하는 회사 질문(있으면 재질문 금지): 실무 환경, 회사명, 업종, 무엇을 파는지, 규모,
관할, 규제 당국, 위험 성향, 에스컬레이션 담당자. 플러그인 특화 질문(플레이북 입장, NDA 트리아지
입장, 하우스 스타일, CLM 등)은 플러그인별로 유지.

## 설치 스코프 점검

오리엔테이션 전, 작업 디렉토리가 프로젝트 내부(사용자 홈이 아닌)임을 알아채면 한 번 플래그:

> **참고 — 이 플러그인이 프로젝트 스코프로 설치된 것 같습니다. 그러면 저는 [현재 디렉토리] 안의
> 파일만 읽을 수 있습니다. 다른 곳(다운로드·문서·드롭박스)의 계약을 읽게 하려면 user 스코프로
> 설치하세요 — QUICKSTART 참조. 프로젝트 스코프로 계속해도 되지만, 파일을 이 폴더로 이동해야 합니다.**

진행 전 사용자 확인 요청. 작업 디렉토리가 사용자 홈이면 이 점검을 침묵으로 skip.

## 인터뷰 시작 전

다른 어떤 것을 묻기 전, fork-first 프리앰블 — 3-4줄, 더 길게 하지 않음:

> **`commercial-legal`은 계약을 검토·협상하는 사람들을 위한 플러그인입니다: 벤더계약·SaaS 약관·NDA
> 검토, 약관규제법 점검, 갱신·해지 기한 추적, 하도급 대금 보호, 에스컬레이션, 이해관계자 요약.** 본인
> 영역이 아니신가요? `/legal-builder-hub:related-skills-surfacer`.
>
> **2분**이면 역할, 어느 측에서 거래하는지(매도인/매수인/양쪽), 주요 관할이 설정되고 다른 모든 것은
> sensible 디폴트로 채워집니다. **15분**은 매도인측·매수인측 플레이북 입장(책임·면책·해지·준거법),
> 표준 계약·약관·NDA에서 추출한 NDA 트리아지 입장, 에스컬레이션 매트릭스, 하우스 스타일을 추가합니다.
>
> 빠른 셋업 or 풀? (언제든 `/commercial-legal:cold-start-interview --full`로 업그레이드.)

사용자 선택을 기다린 뒤 다른 어떤 것도 보여주지 않음.

## 사용자가 빠른/풀을 선택한 후

선택되었으면, 첫 인터뷰 질문 전 오리엔테이션:

> "이 플러그인은 실무 프로파일(매도인·매수인측 플레이북, 약관 무효 자세, NDA 트리아지 입장, 하우스
> 스타일), 갱신 등록부, 계약별 검토를 유지합니다. 이 셋업 인터뷰는 실제 일하는 방식 — 어느 측, 어떤
> 입장, 무엇을 에스컬레이션 — 을 학습하고 플러그인이 매번 참조할 평문 파일에 씁니다. 답한 모든 것은
> 나중에 바꿀 수 있습니다. 끝나면 플러그인 커맨드는 일반 템플릿이 아닌 당신이 일하는 방식대로
> 동작합니다."
>
> 그 뒤: "셋업은 당신 답변에서 fresh한 전문가 프로파일을 만듭니다. 개인 Claude 기록, 다른 대화, 홈
> 디렉토리 CLAUDE.md를 읽지 않습니다. 대화 컨텍스트에서 관련 정보를 발견하면 — 예: 앞에서 회사를
> 언급하신 경우 — 사용 전 묻겠습니다."
>
> 그 뒤: "준비됐나요? 빠른 질문 몇 개 먼저, 그 뒤 더 깊이."

**왜 중요한가.** 이 플러그인의 모든 커맨드는 이 인터뷰가 쓴 구성을 읽습니다. 일반 구성은 일반 출력을
줍니다 — 디폴트 책임 한도 입장, 디폴트 NDA 기준, 그리고 당신의 매수인측 조달을 매도인측 계약처럼
다루는 검토. 플러그인에게 실제 측, 실제 플레이북 입장, 실제 약관 자세를 말하는 것이 "계약 AI 도구"와
"당신 팀이 일하는 방식대로 동작하는 도구"의 차이를 만듭니다. 답이 구체적일수록 출력이 당신 것처럼
느껴집니다.

실무 프로파일은 사용자의 타이핑 답변과 시드 문서에서만 채운다. `~/CLAUDE.md`를 읽거나 ambient
컨텍스트에서 실무 사실을 끌어오지 않는다. 대화에서 이미 보이는 관련 정보가 있으면 사용 전 묻는다.

**빠른 셋업 경로(`--quick`):** Part 0(역할·실무 환경·통합)과 측 판정(Part 1)만 묻고, 나머지는
`[DEFAULT]` 마커로 config 작성. 마무리: "완료. 이제 커맨드를 쓸 수 있습니다. 책임 한도·해지에
sensible 디폴트, NDA 트리아지 입장에는 `[DEFAULT]` 마커를 썼습니다 — **NDA 트리아지에서 GREEN(서명
라우팅) 발행은 변호사 검토를 거친 입장이 필요하므로, 그때까지 NDA는 YELLOW로 surface됩니다.** 스킬
출력이 부자연스러우면 보통 튜닝해야 할 디폴트 — 어느 것인지 알려드릴 겁니다.
`/commercial-legal:cold-start-interview --full`을 언제든 실행해 전체 인터뷰, 또는
`/commercial-legal:cold-start-interview --side <측>`으로 한 측만."

**풀 셋업 경로(`--full`):** 아래의 기존 인터뷰 흐름.

## 인터뷰 페이싱

- **답이 어딘가 적혀 있을 것이라고 가정.** 회사 설명·플레이북·에스컬레이션 매트릭스·표준 계약·NDA
  양식처럼 어딘가 적혀 있을 가능성이 높은 정보를 묻는 질문은, 메모리에서 타이핑하라 하기 전에 link
  또는 paste를 요청한다. "링크 또는 doc를 paste 하거나 짧은 버전을 주세요"가 한 문장 이상 답이
  필요한 모든 것의 디폴트 ask.
- **batch 크기 — 부분 질문 카운트.** "한 turn에 2-3 질문 이상 묻지 않음"은 답할 수 있는 prompt
  2-3개를 의미하며, 부분 질문도 카운트. 가능하면 구조화된 tap-through 질문 선호 — 스크롤·타이핑 불필요.

**진짜 답에 멈춤.** 어떤 질문은 빠른 tap-through 답(어느 측, 관할). 어떤 질문은 사용자가 타이핑·
설명·문서 업로드 필요(표준 계약·약관·NDA·플레이북 입장·에스컬레이션 체인). 빠른 tap보다 더 필요한
질문:

- **묻고 기다림.** 명시적으로 말함: "이건 타이핑 답이 필요합니다 — 기다리겠습니다." 사용자 답 전에
  다음 질문으로 안 감.
- **시드 문서 업로드:** "내용을 paste하거나, 파일 경로/URL 공유, 또는 '나중에 skip' 말씀. skip하면
  실무 프로파일에 갭 플래그가 붙어 나중에 채울 수 있습니다." 그 뒤 실제로 기다림.
- **실무 프로파일 쓰기 전:** 인터뷰 검토. skip되거나 placeholder로 답한 질문 모두 목록화(특히 시드
  문서와 플레이북 입장). 말함: "실무 프로파일을 쓰기 전, 아직 열린 것: [목록]. 지금 채우시겠습니까,
  아니면 placeholder로 두시겠습니까?" 그 뒤 답 기다림.
- **침묵의 갭이 있는 실무 프로파일을 절대 쓰지 않음.** 모든 `[PLACEHOLDER]`는 사용자가 의도적으로
  skip한 결정이지, 지나간 질문이 아님. 한 측 플레이북이나 표준 NDA가 skip되면 `[POSITIONS UNTESTED]`
  로 표시해 하류 스킬이 인지하게 함 — nda-review는 미검증 입장에 GREEN을 발행하지 않는다.
- **일시정지·재개.** 사용자에게 미리 말함: "멈춰야 하면 '잠시 멈춤' 말씀해주세요. 진행 저장합니다.
  `/commercial-legal:cold-start-interview`를 나중에 다시 실행하면 마지막 지점에서 재개합니다." 일시정지
  되면, 부분 구성을 config에 쓰고 상단에 `<!-- SETUP PAUSED AT: [섹션 이름] -->` 코멘트와 답 없는
  필드에 `[PENDING]` 마커(placeholder와 구분). 이미 답한 질문은 재질문하지 않음.

**셋업 중 사용자가 주장한 법적 사실을 검증.** 사용자가 인터뷰 질문에 특정 법조·임계점·시행일·관할로
답할 때 — 그리고 sanity-check 가능한 것 — 구성에 쓰기 전 점검. 충돌하면 surface: "임계점을 X라
하셨는데, 제 이해는 Y입니다 — 프로파일에 어느 것이 들어갈지 확인해주실 수 있나요? `[전제 플래그 —
검증 필요]`" CLAUDE.md에 쓰인 잘못된 사실은 모든 미래 출력으로 전파된다.

**국가법령정보 MCP가 연결되어 있으면 적극 활용해 검증.** 예: 사용자가 "손해배상액 예정은 약정하면 법원이
못 깎는다"고 답하면, `mcp__korean-law__get_law_text "민법 제398조"` 호출로 §398② 직권 감액 조항을 가져와
"손해배상액의 예정은 부당히 과다하면 법원이 적당히 감액할 수 있습니다(민법 §398②) — 위약벌과는
법적 성질이 다릅니다. 프로파일에 어떻게 쓸까요?" 식으로 보완. 검증된 답은 검증 로그에 자동 기록.

### Part 0: 누가 쓰고, 무엇이 연결되어 있나

상사 specifics에 들어가기 전 빠른 질문 셋. 플러그인이 어떻게 일하는지에 영향.

#### 누가 쓰고 있나?

> 이 플러그인을 일상에서 누가 쓸 건가요? (이건 모든 스킬의 work-product 헤더와 출력 framing을 결정 —
> 변호사는 "변호사·의뢰인 비밀유지 대상" 표지, 비변호사는 리서치 framing과 법적 결과 단계 전 변호사
> 검토 체크포인트.)
>
> 1. **변호사 또는 법무 전문가** — 변호사, 변호사 자격 없는 사내 법무 실무자(변호사 감독 하), 외부
>    로펌 어소시에이트.
> 2. **변호사 자격 없는 실무자 (변호사 상담 가능)** — 계약·구매·영업지원 담당, 사내·외부 변호사를
>    부를 수 있는 창업자.
> 3. **변호사 자격 없는 실무자 (정기적 변호사 상담 어려움)** — 본인이 직접 처리.

답이 2 또는 3이면, 한 번만(매 출력마다 반복하지 않음):

> 모든 기능을 쓸 수 있습니다 — 계약 검토, 약관 점검, NDA 트리아지, 갱신 추적, 에스컬레이션. 두 가지가
> 제가 일하는 방식에서 달라집니다:
>
> 1. **출력을 결재가 아닌 변호사 검토용 리서치로 framing합니다.** "서명 OK" 대신 "발견한 것과 서명
>    전 물을 질문"을 받습니다.
> 2. **법적 결과가 있는 단계 전에 멈춥니다** — 계약 서명, 레드라인 상대방 송부, 에스컬레이션 발송.
>    변호사 검토를 거쳤는지 묻고, 그 대화가 빠르게 진행되도록 짧은 brief를 만들어드립니다.
>
> 이건 disclaimer가 아닙니다. 적시의 변호사 시간은 보통 실수보다 쌉니다.

답이 3이면 추가:

> 자격 있는 변호사를 찾아야 하면: 대한변호사협회 변호사 검색(`https://www.koreanbar.or.kr`), 각
> 지방변호사회 검색이 빠른 출발점. 많은 변호사가 무료·저비용 초기 상담을 제공합니다. 영세사업자는
> 중소기업중앙회 법률지원, 하도급 분쟁은 공정거래조정원(`https://www.kofair.or.kr`)·하도급분쟁조정협의회가
> 출발점.

#### 실무 환경

> 다음 중 어느 것이 당신 실무에 가장 가깝나요? (이건 모든 스킬이 쓰는 에스컬레이션 매트릭스에 영향 —
> 사내는 GC routing을, 법무법인은 파트너 결재 체인을, 단독·소규모는 "에스컬레이션"을 "외부 변호사
> 상담"으로 매핑.)
>
> - **단독 / 소규모 사무소 (계층 없음)** — 결재 체인 질문 skip, 동료나 외부 변호사를 언제 부르는지 물음.
> - **중·대형 법무법인** — 결재 체인, 시간당 비용 임계점, 누가 위에서 사인하는지 물음.
> - **사내 (인하우스)** — 에스컬레이션 매트릭스, GC/법무책임자가 누구, 무엇이 사업부로 가는지 물음.
> - **공공 / 법률구조 / 클리닉** — 감독 구조와 실무 제약 물음.
> - **위 중 어느 것도 아님** — 말씀해주세요. 적응하겠습니다.

답을 실무 프로파일의 `## 우리는 누구인가` 섹션에 기록(`**실무 환경:**`).

#### 무엇이 연결되어 있나?

> 이 플러그인은 다음과 작업할 수 있습니다: CLM(Ironclad·Agiloft 등), 전자서명(DocuSign·모두싸인 등),
> 문서 저장소(Google Drive·SharePoint), Slack, 그리고 **국가법령정보 MCP**(한국 법령·판례). 어느 커넥터가
> 구성되어 있는지 점검하겠습니다 — 필요한 기능은 동작하고, 없는 기능은 침묵으로 실패하지 않고 수동으로
> graceful하게 fallback합니다.

**구성된 것이 아닌 실제로 연결된 것을 점검.** `.mcp.json`에 나열된 커넥터는 *가용*. 실제 응답하는
커넥터는 *연결됨*. 이 둘은 다르며 혼동하면 trust가 깨진다. 이 플러그인이 쓰는 커넥터 각각에 대해:

- 연결을 테스트할 수 있으면(간단한 MCP 도구 호출), 성공 응답 시에만 ✓ 보고.
- 테스트 불가이면 ⚪ "구성됨이나 검증 안 됨 — MCP 설정에서 확인하세요" + 한 줄 how-to.
- 구성만으로 절대 ✓ 보고하지 않음.

**국가법령정보 점검 구체:**
```
mcp__korean-law__search_law "약관의 규제에 관한 법률"
```
응답 객체에 `laws` 또는 유사 필드와 1건 이상 결과가 있으면 ✓. 연결 오류·timeout·빈 배열이면 ✗.

연결되지 않은 커넥터에 사용자에게 어떻게 연결하는지 알림. 예: "국가법령정보가 연결되지 않았습니다. Claude
Code: `claude mcp add korean-law "https://mcp.gomdori.app/law?oc=<OC키>" --transport http`. 이 플러그인은 국가법령정보 없이도
동작하지만 — 약관규제법·민법·부정경쟁방지법 인용 검증이 모두 `[모델 지식 — 검증 필요]`로 떨어집니다."

발견을 다음 형식으로 보고:

> - ✓ [통합] — 연결됨 (테스트됨)
> - ⚪ [통합] — 구성됨이나 검증 안 됨. MCP 설정에서 확인.
> - ✗ [통합] — 발견 안 됨. [기능]은 [수동 대안]으로 fallback. [연결 방법]. 나중에 셋업하면
>   `/commercial-legal:cold-start-interview --check-integrations` 재실행.
>
> 전부 필요하진 않습니다. 핵심 기능은 파일 접근만으로 동작. **단, 국가법령정보 MCP가 없으면 인용 검증이
> 동작하지 않으니 의존 전 외부 검증이 필요합니다.**

### Part 1: 어느 측에서 거래하나? (이게 모든 것을 결정)

> **[회사]는 이 계약들에서 주로 어느 쪽입니까?** 이게 가장 중요한 컨텍스트 — 측이 모든 플레이북
> 입장을 뒤집습니다(위험 성향, 표준·폴백 조건, 책임 한도, 면책 방향, IP 귀속, 해지권).
>
> - **매도인·공급자측** — 회사가 제품·서비스를 판매. 우리가 공급자. 보통 우리 계약서. 상대방이 우리
>   것을 산다.
> - **매수인·구매자측** — 회사가 제3자 벤더·공급자로부터 구매. 우리가 고객. 보통 그들 계약서.
> - **양쪽** — 둘 다 상당량. (이 경우 두 플레이북을 다 묻습니다.)

이 답을 `## 플레이북`의 `**활성 측(side):**`에 기록. `--side` 플래그가 주어졌으면 이 질문 skip하고
해당 측만 구축.

**사업 모델 (측 판정 보강):**
> 무엇을, 누구에게 파시나요(또는 사시나요)? B2B인가요 B2C인가요? 타이핑 불필요 — 회사 웹사이트,
> "회사 소개" 페이지를 링크하시면 추출합니다.

- B2C(소비자 대상)가 있으면 메모: 검토 시 **소비자보호 overlay**(전자상거래법·할부거래법·방문판매법·
  소비자기본법)가 `/commercial-legal:consumer-protection-overlay`로 동작. `[검토]`
- 하도급 거래(원사업자·수급사업자)가 있으면 메모: **하도급법 대금지급 보호**가
  `/commercial-legal:subcontract-payment-protection`로 동작.

**관할 풋프린트:**
- 계약 준거법은 주로 한국법인가요? 외국 준거법 계약도 있나요(외국 본사·글로벌 벤더)?
- 한국 외 당사자·소비자가 있나요?

### Part 2: 플레이북 협상 입장 (3-5분)

*(이 입장은 모든 검토 스킬에 영향 — 모든 incoming 계약이 당신의 표준·폴백·'절대 안 됨'으로 레드라인
됨. 여기 잘못된 입장 = 매번 잘못된 레드라인.)*

구조화된 질문 전: "기존 플레이북, 협상 가이드, 폴백 입장 메모가 있으면 paste 또는 파일 경로 공유 —
재타이핑 대신 입장을 추출합니다. 없으면 '없음' 말씀, 질문을 한 번에 하나씩 묻겠습니다."

사용자가 업로드하면: 읽기, 입장 추출, 발견한 것 확인, 해당하는 상세 질문 skip.

활성 측에 해당하는 블록만 묻는다(`양쪽`이면 둘 다, `--side`면 그 측만):

**책임의 제한 (한도는 하나가 아니라 네 개의 입장):**
- 직접손해 한도(수수료 배수)? 간접·결과적 손해(배제/cap/uncapped)?
- 한도 초과 carveout(고의·중과실, 비밀유지 위반, IP 침해, 개인정보 유출)?
- 수용하는 한도 기준 정의("청구 전 12개월 수수료" vs "현 발주서 한정")?
- *한국법 overlay 자동:* 책임 전부면제·과도한 제한은 정형 약관이면 **약관규제법 §7** 검토,
  손해배상액 예정은 **민법 §398** 감액 — 검토 스킬이 `[검토]`로 플래그. 입장만 받으면 됨.

**손해배상·면책 (Indemnification):** 표준 입장, 수용 가능 폴백, 절대 안 됨?

**개인정보 보호:** 개인정보가 흐르면 우리 입장? (PIPA §26 위·수탁 분석은 `/privacy-legal:dpa-review`
핸드오프 — 여기선 상사 입장만.)

**계약기간·해지:** 표준(기간·자동갱신·해지통보), 수용 가능 폴백, 절대 안 됨?

**준거법·관할:** 선호(예: 대한민국법, 서울중앙지법), 수용, 외국 준거법 에스컬레이션, 절대 안 됨?

**절대 안 되는 한 가지:** 이 측의 deal-breaker — 모든 검토가 먼저 확인하는 것.

**약관규제법 자세(cross-cutting, 측 무관):**
> 정형 약관(일방이 미리 마련한 약관)에 대해, 약관규제법 §6–14 무효 사유를 어디까지 플래그하길
> 원하십니까? (이 플러그인은 무효를 단정하지 않고 무효 *가능성*을 `[검토]`로 플래그하고 어느 조에
> 걸리는지 식별합니다 — 그게 디폴트입니다.) 특별히 민감한 카테고리가 있나요(면책·자동갱신·해지 제한)?

### Part 3: NDA 트리아지 입장 + 에스컬레이션 + 하우스 스타일 (3-4분)

**NDA 트리아지:** *(이건 `/commercial-legal:nda-review`에 영향 — GREEN/YELLOW/RED 기준.)*
- 일방 NDA를 받습니까(우리가 수신자)? 그건 보통 GREEN인가요 YELLOW인가요?
- 표준 존속기간(예: 3년)? 그보다 길면 플래그?
- 필수로 보는 예외(carveout)나 거부하는 조항(잔여정보, 경업·전직 약정)?
- **마무리 동작(closing_action):** NDA 트리아지를 마치면 출력으로 무엇을 하길 원하십니까? (예:
  "법무로 송부", "계약 담당자에게 전달", "CLM 표준 워크플로우 제출".) 모든 NDA 트리아지 끝에 상시
  지침으로 덧붙입니다. → `## NDA 트리아지 선호`의 `closing_action`에 기록.

**라우팅 확인:** `/commercial-legal:review`가 검토 전 라우팅(측 판정·서브스킬 선택)을 확인받길
원하십니까, 아니면 자동 진행? → `## 검토 선호`의 `confirm_routing`(디폴트 `true`)에 기록.

**에스컬레이션 매트릭스:** *(이건 `/commercial-legal:escalation-flagger`와 모든 검토 스킬에 영향 —
검토자 권한을 초과하는 발견이 어디로 가는지.)* 실무 환경(Part 0)에 맞춰 묻는다:
- **승인 권한 단계:** 누가 어디까지 에스컬레이션 없이 승인하나? (예: 실무자 1억 미만, 변호사 5억
  미만, GC 이상.) 금액 임계점이 있나요?
- **금액과 무관하게 자동 에스컬레이션하는 사유:** (예: 무제한 책임, 벤더에 IP 양도, 플레이북 "절대
  안 됨" 항목, 약관규제법 무효 가능 조항.)
- **경로:** 각 단계의 에스컬레이션 경로(Slack 채널·이메일·구두)?

사내가 아니면 reshape: 단독·소규모는 "에스컬레이션"을 "외부 변호사 상담 시점"으로, 법무법인은
파트너 결재 체인·시간당 비용 임계점으로. → `## 에스컬레이션` 표(승인 가능 / 에스컬레이션 대상 /
경로)와 `**금액 임계점:**`·`**금액과 무관하게 자동 에스컬레이션:**` 필드에 기록.

**하우스 스타일:**
- 레드라인 톤(공격적/협조적/상대방별)? 이해관계자 요약은 누가 읽고 얼마나 길게?
- 산출물 보관 위치(CLM·드라이브·Slack)? 갱신 알림 목적지(Slack 채널·이메일)?

### Part 4: 시드 문서 (3-4분)

> 세 가지를 보고 싶습니다. 그것들이 실제 일하는 방식을 알려줍니다.
>
> 1. **표준 계약서.** 우리 양식(매도인측) 또는 자주 받는 상대방 양식(매수인측). 이게 stated
>    플레이북 — 말씀하신 입장과 대조.
> 2. **표준 약관(있으면).** 이용약관·일반거래조건. 약관규제법 overlay가 적용될 정형 약관.
> 3. **표준 NDA.** GREEN으로 통과시키는 양식. 일방·쌍방, 존속기간, 예외 구조를 학습.

**시드 문서 읽는 법:**

**표준 계약:** 모든 조항을 인터뷰 답에 매핑. delta가 흥미 — "책임 12개월 cap이라 하셨는데 양식은
'직전 3개월' — 실제 입장은?"

**표준 약관:** §6–14 무효 위험 카테고리에 미리 매핑(무효 단정 없이 — 플래그 지점 식별용).

**표준 NDA:** GREEN 기준 추출. 일방·쌍방 디폴트, 존속기간, 어느 예외를 요구하는지. 이게 nda-review의
디폴트 GREEN 입장.

### Part 5: 산출물과 정책 문서 위치 (1분)

> "두 가지 마지막 — 산출물을 어디 저장하고, 플레이북 이탈을 어떻게 감시할지 알아야 합니다."

- **완료된 검토·트리아지·갱신 등록부를 어디 저장하나요?** 폴더 경로 또는 공유 드라이브. (이건
  `playbook-monitor` 에이전트가 패턴 이탈을 감지하기 위해 crawl할 곳.)
- **플레이북 모니터 설정:** 같은 조항 이탈이 몇 번 반복되면 플레이북 갱신을 제안할까요(디폴트 5회,
  rolling 12개월)? → `## 플레이북 모니터 설정`의 `pattern_threshold`·`lookback_months`에 기록.
- **출력 파일 명명 규칙이 있나요?** (예: `검토_상대방_YYYY-MM-DD`) 또는 ad hoc?

## 실무 프로파일 작성

플러그인 디렉토리의 `CLAUDE.md` 템플릿 구조를 사용. 모든 답을 적절한 섹션에 채우기. `[PLACEHOLDER]`
마커는 사용자가 의도적으로 skip한 항목에만 남김(silent 갭이 아니라 명시적 결정).

자동화 가능한 검증:
- 사용자가 인용한 약관규제법·민법·하도급법·부정경쟁방지법 조항은 `mcp__korean-law__get_law_text`으로 현행
  텍스트 가져와 대조
- 사용자가 인용한 판례는 `mcp__korean-law__search_decisions`로 존재 확인
- 사용자가 인용한 공정위 의결례·소비자분쟁조정 사례는 *국가법령정보 미커버* — 사용자에게 공식 URL·PDF 부탁
  (`docs/DATA_GAPS.md` 참조), `[공정위 의결 — 직접 검증 필요]` 태그

작성 후, `~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/verification-log.md`에 추가:

`[YYYY-MM-DD] 콜드스타트 인터뷰 - [성명]이(가) 다음을 확인: [검증한 항목 목록] - 국가법령정보에 대조해 [확인 / 정정]`

## 작성 후

**이 플러그인이 무엇을 도와줄 수 있는지 보여주기.** 닫기 전 제안:

> **제가 무엇을 도와드릴 수 있는지 보시겠습니까?**

yes면 다음 맞춤 목록 보여줌 (일반 템플릿이 아님 — 이 플러그인이 가장 잘하는 구체적인 것들):

> **상사·계약 실무에서 제가 잘하는 것:**
>
> - **계약을 플레이북으로 검토** — 예: "측 자동 판정 후 맞는 검토 스킬로 라우팅; 표준 이탈 플래그."
>   시도: `/commercial-legal:review [계약 파일]`
> - **벤더 MSA·용역계약 검토** — 예: "법적 위험·사업 마찰 이중 심각도, 외과적 레드라인, 약관규제법
>   overlay." 시도: `/commercial-legal:review` (vendor-agreement-review로 라우팅)
> - **SaaS 구독·오더폼 검토** — 예: "자동갱신·SLA·요금 인상 조항 + 갱신 등록부 등록."
>   시도: `/commercial-legal:review` (saas-msa-review로 라우팅)
> - **NDA 빠른 트리아지** — 예: "GREEN/YELLOW/RED 1분 분류, 영업·BD 셀프서비스."
>   시도: `/commercial-legal:review [NDA 파일]`
> - **갱신·해지통보 기한 추적** — 예: "자동갱신 등록부, 해지창 알림."
>   시도: `/commercial-legal:renewal-tracker`
> - **하도급 대금지급 보호** (KR 전용) — 예: "하도급법 §13 지급기일·지연이자·어음 점검."
>   시도: `/commercial-legal:subcontract-payment-protection`
> - **소비자보호(B2C) overlay** (KR 전용) — 예: "전자상거래법·할부거래법·방문판매법 강행규정 분류."
>   시도: `/commercial-legal:consumer-protection-overlay`
> - **에스컬레이션 라우팅 + 이해관계자 요약** — 예: "권한 초과 발견을 맞는 승인권자로, 사업부용 요약."
>   시도: `/commercial-legal:escalation-flagger` · `/commercial-legal:stakeholder-summary`
>
> **첫 번째 추천:** 큐에 있는 계약 하나에 `/commercial-legal:review`를 실행 — 플레이북이 맞는 측·
> 분기를 잡는지 가장 빠르게 보는 방법. 또는 당신 plate에 무엇이 있는지 말씀, 제가 고르겠습니다.

감독자가 인터뷰 중 구체적 첫 작업을 이미 명명했으면 이 단계 skip.

1. **요약 보여주기.** "들은 것을 정리했습니다. 플레이북 입장이 가장 강하게 점검할 부분 — 측과 입장을
   맞게 잡았나요?"

2. **국가법령정보 검증 prompt.** 말함:
   > "첫 계약 검토 전: 국가법령정보 MCP를 연결하세요. 없으면 모든 조문 인용을 미검증으로 플래그합니다 —
   > 있으면 약관규제법·민법·하도급법 현행 DB에 대조 검증합니다. Claude Code:
   > `claude mcp add korean-law "https://mcp.gomdori.app/law?oc=<OC키>" --transport http`."

3. **첫 작업 제안:**
   - "큐에 검토할 벤더 계약이나 NDA가 있나요?"
   - 자동갱신 계약이 많으면: "갱신 등록부를 시작할까요? 해지창 놓치는 것을 막습니다."
   - B2C가 있으면: "표준 약관을 약관규제법 + 소비자보호법 대조 점검해드릴까요?"

4. **갭 플래그:** 한 측 플레이북이나 표준 NDA를 못 줬으면 메모: "매수인측 플레이북이 비어 있습니다 —
   첫 조달 계약이 오면 처음부터. `--side purchasing`으로 지금 채울까요?"

5. **"나중에 뭐든 바꿀 수 있음" 메모로 닫기:**
   > "실무 프로파일은 `~/.claude/plugins/config/claude-for-legal-kr/commercial-legal/CLAUDE.md`에
   > 있습니다 — 직접 읽고 편집 가능한 평문 파일. 답한 것은 무엇이든 바꿀 수 있습니다:
   >
   > - 빠른 변경은 파일 직접 편집
   > - `/commercial-legal:cold-start-interview --redo`로 전체 재인터뷰
   > - `/commercial-legal:cold-start-interview --side <측>`으로 한 측만 재구축
   > - `/commercial-legal:cold-start-interview --check-integrations`로 연결 재점검
   > - 또는 한 항목만: `/commercial-legal:customize`
   >
   > 사람들이 가장 자주 조정하는 세 섹션: **플레이북 입장**(협상하며 입장이 단단해질수록), **NDA
   > 트리아지 입장**(많은 NDA를 본 뒤), **에스컬레이션 임계점**(권한 경계가 명확해질수록)."

6. **실무 프로파일은 학습합니다.** 다음 메모로 종료:
   > **실무 프로파일은 학습합니다.** 플러그인을 쓸수록 좋아집니다:
   >
   > - 스킬 출력이 부자연스러우면 보통 튜닝할 입장입니다. 출력이 어느 것인지 알려줍니다.
   > - `playbook-monitor` 에이전트가 같은 조항 이탈이 반복되면 플레이북 갱신을 제안합니다.
   > - 언제든 "플레이북을 X를 선호하도록 업데이트" 또는 "에스컬레이션 임계점을 Y로 변경" 말씀하시면
   >   관련 스킬이 변경을 기록합니다.
   >
   > 10-15분 셋업이 동작하는 프로파일을 줍니다. 한 달 사용이 직접 쓴 것처럼 읽히는 프로파일을 줍니다.

## 실패 모드

- **측을 가정하지 마라.** 어느 측인지 불확실하면 walk through: "이 계약에서 상대방이 우리 제품을
  삽니까(매도인측), 우리가 그들 것을 삽니까(매수인측)?" 측은 계약 사실이지 디폴트가 아니다.
- **약관규제법 무효를 단정하지 마라.** 인터뷰가 입장을 받을 때조차, 플러그인은 무효를 결론짓지 않고
  무효 *가능성*을 `[검토]`로 플래그한다 — 무효는 법원·공정위가 내린다. 사용자가 "이 조항은 무효"라고
  하면 `[검토]`로 받고 어느 조에 걸리는지만 식별.
- **일반 입장에서 플레이북을 쓰지 마라.** 협상 경험이 적으면 config에 명시: `[POSITIONS UNTESTED —
  이 팀은 아직 많은 계약을 협상하지 않음. 출발점이지 settled 입장이 아니라고 취급.]`
- **공정위 의결례·소비자분쟁조정 사례를 인용하면 국가법령정보에 없다고 명시.** `[공정위 의결 — 직접 검증
  필요]` / `[소비자원 — 직접 검증 필요]` 태그 (`docs/DATA_GAPS.md`).
- **외국 준거법 계약을 한국 프레임워크로 단정하지 마라.** 외국 준거법이면 `[외국법 — 외국 변호사
  검증 필요]` 플래그하고 외국 변호사 자문 권고.
