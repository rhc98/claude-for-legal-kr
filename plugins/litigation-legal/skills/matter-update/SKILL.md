# /matter-update

1. `~/.claude/plugins/config/claude-for-legal-kr/litigation-legal/CLAUDE.md` 로드 →
   사건유형별 플레이북, 에스컬레이션 체인.
2. 아래 워크플로우와 참조표를 따른다.
3. `$ARGUMENTS`에서 slug·이벤트 설명 추출 — slug 없으면 활성 매터 사용(없으면 묻는다).
4. `matters/<slug>/matter.md` + `matters/<slug>/history.md` 존재 확인.
5. 이벤트 유형·날짜·요약·로그 필드 갱신 사항 입력받는다.
6. `matters/<slug>/history.md`에 날짜별 항목 추가 (최신이 위).
7. `matters/_log.yaml` 갱신 — `last_updated`를 오늘로, 필드 변경 적용.
8. 확인.

## 업데이트 워크플로우

### 1단계 — 매터 확인

slug가 `matters/` 존재하는지 확인. 없으면:
> "[slug]를 매터 로그에서 찾을 수 없습니다. `/litigation-legal:matter-intake`를 먼저 실행해
> 사건을 등록하세요."

### 2단계 — 이벤트 유형 확인

아래 이벤트 유형 중 하나 선택. 복합 이벤트는 복수 유형 적용.

**절차** — 기일 진행·결정 이벤트:
- **기일(변론·준비·심문·결정):** 기일 날짜, 진행 내용, 다음 기일 지정 여부.
  - 기일 지정·변경은 `[전자소송/나의사건검색 — 사용자 확인 필요]` — 기일통지서 paste 권장.
  - 기일 후 다음 기일 날짜 갱신 시 `[기간 계산 — 1차 소스 대조 검증 필요]`.
- **결정·명령:** 재판장 결정 내용, 이행 기한.
- **판결 선고:** 주문, 이유 요지, **송달일 확인 요청** → 상소기간 기산.
- **즉시항고 기한:** 고지일 확인 → 1주 불변기간 `[기간 계산 — 1차 소스 대조 검증 필요]`.

**서면·송달** — 제출·수령 이벤트:
- **준비서면 제출(민사소송법 §272~):** 제출일, 주요 주장·증거 요약.
- **소장·답변서 제출·수령:** 날짜, 주요 내용, 소가·청구취지 확인.
- **송달 완료(§174~):** 송달 대상, **송달일 기록** — 상소기간·이의기간 기산점.
  - 송달 현황은 `[전자소송/나의사건검색 — 사용자 확인 필요]` — 송달통지서 paste 권장.
- **지급명령 송달:** 송달일 기록 → 이의 2주 불변기간 기산 `[기간 계산 — 1차 소스 대조 검증 필요]`.
- **항소장·상고장 제출:** 제출일, 항소/상고이유서 제출기한 확인.

**증거·기록:**
- **서증 제출·수령:** 문서번호, 제목, 관련 쟁점.
- **증인 신청·심문:** 증인명, 신문 사항, 결과.
- **감정 신청·결과:** 감정 대상, 결과 요지.
- **압수수색·수사 관련(형사):** 집행일, 대상, 대응 조치.

**전략:**
- **입장 전환:** 변경된 전략 방향, 이유.
- **화해·조정 시도:** 제안 내용, 상대방 반응, 현재 상태.
- **에스컬레이션:** 보고 대상, 결정 사항.
- **보전처분 신청·결정:** 신청일, 결정 내용, 담보 제공 현황.

**기한 변동:**
- **다음 기일 변경:** 새 날짜 기록, 기존 기일 변경 사유.
- **불변기간 임박 경보:** 남은 일수, 필요 조치. **🔴 자동 에스컬레이션 대상.**
- **추후보완 검토:** 불변기간 도과 사유가 있는지 `[검토]` — 민사소송법 §173 `[검증 필요]`.

**외부 대리인:**
- **OC 변경:** 새 담당 변호사, 인수인계 현황.
- **OC 보고 수령:** 보고 요지, 후속 지시 사항.

**행정(형사) 특유:**
- **처분 수령(행정):** 처분일, 처분 내용 → **제소기간 90일 기산** `[기간 계산 — 1차 소스 대조 검증 필요]`.
- **불기소·기소(형사):** 처분 내용, 불복 여부 검토(항고·재정신청).
- **집행정지 신청·결정(행정):** 신청일, 결정 내용.

### 3단계 — 이벤트 기록

이벤트 기록 전 확인:
1. **날짜:** 이벤트 날짜 (기본: 오늘).
2. **이벤트 유형:** 위 분류 중.
3. **요약:** 2-5문장. 무슨 일이 있었는가, 무엇이 바뀌었는가, 다음 단계는 무엇인가.
4. **로그 필드 갱신:**
   - `next_deadline` 변경 여부 → `[기간 계산 — 1차 소스 대조 검증 필요]`
   - `risk` 등급 변경 여부
   - `status` 변경 여부 (active → 판결 선고 후 상소 검토 중 등)
   - `stage` 변경 여부 (예: 1심 → 항소심)

### 4단계 — 기한 변동 자동 점검

이벤트가 아래 중 하나면 기한 자동 점검:

| 이벤트 | 점검 항목 | 기간 | 태그 |
|---|---|---|---|
| 판결 선고·송달 확인 | 항소/상고기간 | 2주 (판결정본 송달일부터) | `[기간 계산 — 1차 소스 대조 검증 필요]` |
| 결정 고지 | 즉시항고 | 1주 (고지일부터) | `[기간 계산 — 1차 소스 대조 검증 필요]` |
| 지급명령 송달 | 이의기간 | 2주 (송달일부터) | `[기간 계산 — 1차 소스 대조 검증 필요]` |
| 처분 수령(행정) | 제소기간 | 90일 (처분 안 날부터) | `[기간 계산 — 1차 소스 대조 검증 필요]` |
| 불기소 처분(형사) | 항고·재정신청 | 항고 7일(§260), 재정신청 10일(§261) `[검증 필요]` | `[기간 계산 — 1차 소스 대조 검증 필요]` |

> **날짜가 없으면 기간을 산정하지 않는다.** 공식과 필요한 정보를 제시하고 사용자에게 확인 요청.
>
> **전자소송·나의사건검색은 공개 API 없음** — 기일통지서·판결정본·송달통지서를 paste해야 정확한 날짜 확인 가능 `[전자소송/나의사건검색 — 사용자 확인 필요]`.

## `history.md` 항목 형식

```markdown
## [YYYY-MM-DD] — [이벤트 유형]: [짧은 제목]

**유형:** [절차 | 서면·송달 | 증거 | 전략 | 기한 변동 | OC | 행정·형사 특유]
**담당:** [기록자 또는 OC]

[이벤트 요약 — 2-5문장. 무슨 일이 있었는가, 무엇이 바뀌었는가, 다음 단계는 무엇인가.]

**기한 변동:** [있으면: 새 next_deadline YYYY-MM-DD — `[기간 계산 — 1차 소스 대조 검증 필요]` | 없음]
**리스크 변동:** [있으면: [이전] → [새 등급], 이유 | 없음]
**로그 갱신:** [last_updated, next_deadline, risk, status, stage 변경 사항]
```

## 에스컬레이션 자동 트리거

이벤트가 아래에 해당하면 즉시 에스컬레이션 노트 생성 제안:

- **불변기간 7일 이내 임박** (상소기간·즉시항고·행정소송 제소기간)
- **패소·일부패소 판결 선고** — 상소 여부 결정 필요
- **형사 구속·압수수색 집행**
- **가압류·가처분 신청 또는 피신청**
- **소가 [실무 프로파일 임계점] 이상 사건의 전략 변경**
- **OC 교체·사임**

에스컬레이션 노트 형식:
> "[에스컬레이션 대상]에게 — [사건명]: [이벤트 요약]. 기한: [날짜]. 필요한 결정: [1-2가지]."

## 이 스킬이 하지 않는 것

- **기간 계산 단정.** 기산점(송달일·고지일·처분일)이 확인되지 않으면 산정하지 않음.
- **전자소송·사건검색 기일 조회.** 공개 API 없음 — 사용자 제공 필수.
- **하급심 판결 미공개 내용 인용.** → `[하급심 판결 — 미공개, 직접 확인 필요]`.
- **판례 유효성 단정.** 인용 시 cite_check → `[선례 유효성 — cite_check 확인]`.
