# /e-litigation-workflow

## 핵심 목적 — 안내만, 자동화 아님

이 스킬은 **전자소송의 절차·형식·기한을 안내**한다. 실제 제출·전자송달 확인·열람·결제는 **사용자가
ecfs.scourt.go.kr(대법원 전자소송)에서 직접 수행**한다.

> **⚠️ 공개 API 없음 — 데이터 갭.** 전자소송·나의 사건검색(사건현황·기일·송달현황)은 공개 API가 없어
> 국가법령정보로 조회·자동화할 수 없다(CLAUDE.md `## 데이터 갭`). 이 스킬은 사건 상태를 조회하거나 서면을
> 대신 제출하지 않는다. 절차를 안내하고, 사용자가 수행한 결과(제출 완료 화면·송달통지·기일통지서)를
> paste하면 그것을 바탕으로 다음 단계를 안내한다. 모든 사건현황·송달정보는 `[전자소송/나의사건검색 — 사용자 확인 필요]`.

## 플레이북 먼저 로드

`~/.claude/plugins/config/claude-for-legal-kr/litigation-legal/CLAUDE.md`를 읽는다. `## 데이터 갭`·
`## 현행성 워치`(전자소송 확대는 규칙·예규로 변동)를 적용한다.

> **침묵의 보완 금지.** 근거법(민사소송 등에서의 전자문서 이용 등에 관한 법률 — 약칭 민소전자문서법,
> 전자소송 규칙·재판예규)을 국가법령정보에서 확인하지 못하면 `[모델 지식 — 검증 필요]`로 태그하고 진행한다.
> **전자소송 규칙·예규는 개정이 잦다** — 조문·서식·용량 제한은 반드시 현행 대조.
>
> **국가법령정보 조회 예시:**
> ```
> mcp__korean-law__search_law "전자문서 이용"
> mcp__korean-law__get_law_text "민사소송 등에서의 전자문서 이용 등에 관한 법률"
> mcp__korean-law__legal_analysis {"mode": "verify_citations", "citation": "민사소송 등에서의 전자문서 이용 등에 관한 법률"}
> ```
>
> **출처 태그:** `[국가법령정보]`(이 세션 가져온 경우만) · `[사용자 제공]`(제출화면·송달통지 paste) ·
> `[모델 지식 — 검증 필요]`(디폴트) · `[전자소송/나의사건검색 — 사용자 확인 필요]`(사건현황·송달·기일).

## 2단계 — 제출 절차 안내 (사용자 수행)

전자소송 제출은 사용자가 ecfs.scourt.go.kr에서 수행한다. 안내 골자:

1. **전자소송 로그인 → 서류 제출** — 사건 선택(신규 소 제기 또는 진행 중 사건).
2. **전자문서 작성·첨부** — 서면 본문 + 첨부서류(증거) 업로드.
3. **수수료·송달료 납부** — 인지·송달료 전자납부. **요율은 하드코딩하지 않는다**(자주 변동,
   `references/litigation-currency-watch.md`) `[인지·송달료 요율 — 직접 검증 필요]`.
4. **제출 완료 → 접수증** — 접수 결과·사건번호 확인.

> **제출 시각·마감.** 전자제출의 효력 발생 시점(제출 시각)·마감 취급은 규칙에 따른다 — 야간·마감일 제출의
> 효력은 규칙 대조 `[검토]` `[모델 지식 — 검증 필요]`. **불변기간 말일에 제출할 때 제출 완료 시각을 반드시
> 확인**한다 → 기한 산정은 `/litigation-legal:deadline-tracker`.

## 3단계 — 전자문서 형식 요건 (안내)

| # | 항목 | 태그 | 비고 |
|---|---|---|---|
| F-1 | **파일 형식** — 전자소송이 허용하는 형식(PDF 등)·전자서명 요건 `[모델 지식 — 검증 필요]` | `[검토]` | 규칙·예규 대조 |
| F-2 | **용량·페이지 제한** — 파일당·건당 용량 제한 `[모델 지식 — 검증 필요]` | `[검토]` | **구체 수치는 하드코딩 금지 — 현행 대조** |
| F-3 | **첨부서류·증거 표시** — 증거 부호·목차 요건 | — | 서면 작성 스킬과 연계 |

> **용량·형식 수치를 지어내지 않는다.** 전자소송 규칙·전자소송 홈페이지 공지의 현행 값을 확인하도록 안내하고,
> 확인 전에는 `[모델 지식 — 검증 필요]`로 남긴다.

## 4단계 — 전자송달·열람 (사용자 확인)

| # | 항목 | 태그 | 비고 |
|---|---|---|---|
| S-1 | **전자송달** — 전자소송 동의자에게는 전자적으로 송달. 송달의 효력 발생 시점(확인한 때 또는 등재통지 후 일정 기간 경과 시 등)은 규칙 대조 `[모델 지식 — 검증 필요]` | `[검토]` `[기간 계산 — 1차 소스 대조 검증 필요]` | **상소·이의 기산점과 직결** |
| S-2 | **송달 확인** — 사용자가 전자소송에서 송달 확인·열람을 수행하고 결과를 paste | — | `[전자소송/나의사건검색 — 사용자 확인 필요]` |
| S-3 | **기록 열람·출력** — 사건기록 열람은 전자소송에서 수행 | — | 자동 조회 불가 |

> **전자송달의 효력 발생 시점이 불변기간 기산점을 좌우한다.** "언제 송달된 것으로 보는가"가 상소·이의
> 기간의 출발점이므로, 사용자에게 전자송달 확인일·등재통지일을 확인하도록 요청하고
> `/litigation-legal:deadline-tracker`로 넘긴다. 효력 발생 시점 규율은 현행 규칙 대조 `[검토]`.

## 다음 단계 의사결정 트리

**제 체크리스트에는 없지만 한 가지 짚어드리고 싶은 것:** 전자송달은 "언제 송달된 것으로 보는가"가 종이
송달과 달라 상소·이의 기산점 계산이 헷갈리기 쉽습니다 — 송달 확인일·등재통지일을 확인하는 즉시
`/litigation-legal:deadline-tracker`에 기산점을 넘겨 불변기간을 관리하십시오.

> **다음 단계? 하나 골라주세요. 살을 붙이겠습니다:**
> 1. **제출 체크리스트 초안** — 특정 서면의 전자소송 제출 단계·필요 첨부·수수료 확인 목록. *(제출을 앞둔 경우.)*
> 2. **기한 산정 연계** — 전자송달 확인일을 `/litigation-legal:deadline-tracker`에 넘겨 상소·이의 기한 산정.
> 3. **형식 요건 확인 요청** — 현행 전자소송 규칙·공지의 파일 형식·용량 제한을 확인하도록 안내.
> 4. **서면 작성 연계** — 제출할 서면 자체의 초안이 필요하면 해당 작성 스킬로 이동.
> 5. **다른 것** — 이 결과로 무엇을 하고 싶은지 알려주세요.

## 이 스킬이 하지 않는 것

- 전자소송에 서면을 대신 제출한다 (사용자 수행 — 공개 API 없음).
- 사건현황·기일·송달현황을 조회한다 (자동 조회 불가 — 사용자 확인).
- 인지·송달료 요율을 하드코딩한다 (자주 변동 — 현행 대조).
- 파일 형식·용량 제한 수치를 지어낸다 (`[모델 지식 — 검증 필요]`, 현행 대조).
- 전자송달 효력 발생 시점을 단정한다 (규칙 대조 + `[검토]`).
